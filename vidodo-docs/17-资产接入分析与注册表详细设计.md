# 资产接入分析与注册表详细设计

日期：2026-04-01
状态：Draft
前置阅读：06-资产接入与分析缓存设计.md，15-功能模块设计方案总览.md

## 1. 文档目的

本文细化 Asset Ingestion、Analysis Pipeline 与 Asset Registry 三个模块的详细设计。

目标是把“素材文件”稳定转化为“可引用、可分析、可校验、可预热的系统资产”。

## 2. 模块边界

### 2.1 Asset Ingestion

负责：

- 候选素材发现
- 内容指纹生成
- 规范化入口调度
- 资产发布

### 2.2 Analysis Pipeline

负责：

- 音频与视觉分析任务调度
- 缓存键生成
- 分析结果发布

### 2.3 Asset Registry

负责：

- 资产索引
- 资产状态查询
- readiness / warmness / stale 状态管理

## 3. 资产生命周期

```text
Discovered
  -> Fingerprinted
  -> Normalized
  -> Analyzing
  -> Published
```

失败分支：

```text
Discovered -> Rejected
Normalized -> Failed
Analyzing -> Partial
Published -> Stale
```

## 4. Asset Ingestion 设计

### 4.1 内部子模块

- Candidate Scanner
- Fingerprint Builder
- Normalization Dispatcher
- Publish Coordinator
- Ingestion Run Recorder

### 4.2 输入

- source path
- declared kind
- profile
- tags

### 4.3 输出

- IngestionRun
- AssetRecord
- normalized locator
- failure diagnostics

### 4.4 IngestionRun 模型

```json
{
  "ingestion_run_id": "ing-20260401-01",
  "source": "imports/session_a",
  "status": "completed",
  "discovered": 12,
  "published": 10,
  "reused": 2,
  "failed": 0
}
```

### 4.5 规范化策略

音频：

- 统一 sample rate
- 统一通道布局
- 生成 runtime-friendly locator

视觉：

- shader / texture / buffer graph 元数据整理
- 基础依赖图抽取

### 4.6 失败模型

- unsupported_format
- probe_failed
- normalize_failed
- publish_conflict

## 5. Analysis Pipeline 设计

### 5.1 分析任务模型

```json
{
  "analysis_job_id": "job-018",
  "asset_id": "asset-loop-kick-a",
  "analyzer": "beat_track",
  "status": "completed",
  "cache_key": "acache:sha256...",
  "result_ref": "analysis-013"
}
```

### 5.2 音频分析任务

- beat_track
- downbeat_detect
- section_segmentation
- key_estimate
- energy_envelope

### 5.3 视觉分析任务

- shader_compile_probe
- dependency_graph_build
- texture_stat_summary
- gpu_cost_estimate

### 5.4 缓存键

缓存键由以下部分组成：

- asset_content_hash
- normalized_spec
- analyzer_name
- analyzer_version
- params_hash
- dependency_fingerprint

### 5.5 分析状态

```text
Queued
  -> Running
  -> Completed
  -> Failed
  -> Stale
```

### 5.6 Partial 分析策略

允许部分分析完成，例如：

- 节拍结果存在，但 section 失败
- shader probe 通过，但 gpu estimate 缺失

Registry 必须显式标注缺失分析，不允许静默当作完整可用资产。

## 6. Asset Registry 设计

### 6.1 核心职责

- 提供按 asset_id、tag、kind、capability 查询
- 提供 analysis refs 查询
- 提供 warmness 与 readiness 判断
- 提供 patch / compiler 所需 compatibility 信息

### 6.2 核心状态字段

- publish_status
- warm_status
- readiness
- analysis_completeness
- stale_flag
- last_validated_at

### 6.3 AssetRecord 模型

```json
{
  "asset_id": "asset-loop-kick-a",
  "asset_kind": "audio_loop",
  "content_hash": "sha256:...",
  "normalized_locator": "norm://audio/asset-loop-kick-a.wav",
  "analysis_refs": ["analysis-013"],
  "publish_status": "published",
  "warm_status": "cold",
  "readiness": "compile_ready"
}
```

### 6.4 Readiness 级别

- metadata_only
- compile_ready
- live_candidate
- warmed
- unavailable

### 6.5 Warmness 状态

- cold
- staging
- warmed
- expired

### 6.6 查询接口

建议支持：

- get_asset(asset_id)
- list_assets(filter)
- list_analysis(asset_id)
- check_asset_ready(asset_id, target)
- mark_warm(asset_id)

## 7. 模块协作关系

### 7.1 Compiler 依赖

Compiler 读取：

- asset existence
- capability flags
- analysis summary
- compatibility tag

### 7.2 Patch Manager 依赖

Patch Manager 读取：

- warm_status
- live readiness
- fallback asset availability

### 7.3 Capability Layer 依赖

Capability Layer 读取：

- ingestion run status
- asset list
- asset diagnostics

## 9. MVP 范围

MVP 阶段应收敛为：

- 固定目录 ingest
- 有限 asset kind
- 两个音频 analyzer
- 一个视觉 probe 链路
- 基础 registry 查询与 warm 状态标记

## 10. 验收标准

该模块链路达标，当且仅当：

1. 文件可被 ingest 成稳定 AssetRecord。
2. 分析结果可缓存且可按版本失效。
3. Compiler 与 Patch Manager 都只消费 registry 中已发布资产。
4. 能明确区分 compile_ready 与 warmed 状态。

## 11. 结论

资产链路的本质是“把不稳定文件系统变成稳定资产事实层”，它是整个系统可编译、可 patch、可回放的前提。
