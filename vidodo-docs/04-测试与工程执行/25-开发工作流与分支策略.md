# 开发工作流与分支策略

日期：2026-04-02
状态：Active
前置阅读：14-MVP技术架构图与Phase0实施清单.md，24-工作任务卡与开发里程碑.md

## 1. 文档目的

本文定义当前仓库的有效工程工作流。

它不描述未来完整生态，而只描述当前已经落地并应被严格执行的 Phase 0 基线：

```text
vidodo-docs/ 作为文档真源
schemas/ 作为工件真源
scripts/ 作为仓库级验证入口
vidodo-src/ 作为当前唯一实现主线
```

## 2. Canonical Roots

### 2.1 根路径约束

| 根路径 | 角色 | 当前状态 |
| --- | --- | --- |
| `vidodo-docs/` | 产品边界、架构、任务卡、测试策略真源 | 生效中 |
| `schemas/` | 正式 JSON Schema 真源 | 生效中 |
| `scripts/` | 仓库级 schema / artifact 校验入口 | 生效中 |
| `tests/` | schema fixture、受控 plan/asset/patch fixture、e2e smoke | 生效中 |
| `vidodo-src/` | Rust 工作区与 Phase 0 闭环实现 | 生效中 |

### 2.2 当前仓库布局

```text
vidodo/
├── .github/
├── schemas/
├── scripts/
├── tests/
│   ├── schema/
│   ├── fixtures/
│   └── e2e/
├── vidodo-docs/
├── vidodo-src/
│   ├── apps/
│   ├── crates/
│   ├── xtask/
│   └── Cargo.toml

```

### 2.3 当前唯一主线

当前只承认一条实现主线：

```text
avctl
  -> validate
  -> compile
  -> patch submit
  -> run start (fake audio / fake visual backend)
  -> trace show / trace events
  -> rollback decision
```

`core-service`、MCP、灯光、多节点分布式仍然保留为边界设计或占位，不进入当前主线收口标准。

## 3. 分支策略

### 3.1 主干模式

继续采用 trunk-based development：

```text
main ──────────────────────────────────────────────►
  │
  ├── feat/wsa-14-schema-fixtures
  ├── feat/wsc-04-phase0-mainline
  └── fix/trace-event-ordering
```

### 3.2 命名规范

| 分支类型 | 命名模式 | 示例 |
| --- | --- | --- |
| 功能分支 | `feat/{任务卡号}-{简短描述}` | `feat/wsc-04-phase0-mainline` |
| 修复分支 | `fix/{简短描述}` | `fix/patch-window-check` |
| 文档分支 | `docs/{简短描述}` | `docs/normalize-workflow-root` |
| 重构分支 | `refactor/{简短描述}` | `refactor/storage-path-discovery` |

### 3.3 保护规则

- `main` 禁止直接 push
- 变更通过 PR 合入
- PR 至少 1 名 reviewer approve
- CI 全绿后才允许合入

## 4. Commit 与 PR 规范

### 4.1 Commit 格式

```text
<type>(<scope>): <subject>
```

### 4.2 推荐 type

- `feat`
- `fix`
- `schema`
- `test`
- `docs`
- `ci`
- `refactor`
- `chore`

### 4.3 当前有效 scope

PR 模板以仓库根下的 `.github/pull_request_template.md` 为准。

## 5. CI 流程

### 5.1 当前生效 Job

```text
Job 1: Schema Validation
  - 安装 jsonschema
  - 运行 ./scripts/init-artifact-store.sh
  - 运行 ./scripts/schema-validate.sh

Job 2: Rust Quality Gate
  - cargo fmt --all --check
  - cargo clippy --workspace --all-targets --all-features -- -D warnings
  - cargo test --workspace --all-targets
  - cargo audit

Job 3: Phase 0 Smoke
  - 运行 ./tests/e2e/phase0_smoke.sh

Job 4: Bench
  - 仅 main 分支或手动触发
  - cargo bench --workspace
```

### 5.2 失败策略

- 任一 Job 失败都阻止 PR 合入
- `Schema Validation` 是最高优先级修复项
- Rust warning 视为 error
- `Phase 0 Smoke` 失败意味着主线已断，不允许继续扩 scope

## 6. Schema 变更流程

当 schema 变更时，必须同步完成以下动作：

1. 修改 `schemas/` 下的 JSON Schema 文件
2. 同步更新 `tests/schema/` 下的正例和反例 fixture
3. 同步更新 `vidodo-src/crates/ir/` 中的对应 Rust 类型
4. 如影响 CLI 输出，更新 `avctl` 响应 envelope 与 smoke 脚本
5. 如影响边界或对象语义，更新 `vidodo-docs/` 中对应设计文档

## 7. 文档同步机制

### 7.1 文档位置

设计文档继续放在 `vidodo-docs/`。

不再使用 `docs/design/` 作为设计文档迁移目标。

### 7.2 触发更新条件

以下变更必须同步更新文档：

- IR 类型字段增删
- 事件模型变更
- Patch 状态机变更
- CLI 命令面变化
- 仓库根路径和 CI 规则变化

### 7.3 状态标记

- `Draft`：仍在设计中
- `Active`：当前实现必须遵守
- `Superseded`：只保留历史价值，不再作为执行依据

## 8. Artifact 管理

### 8.1 根路径

`artifacts/` 位于仓库根目录，由 `.gitignore` 排除。

### 8.2 当前目录结构

```text
artifacts/
├── assets/
├── analysis/
├── revisions/
├── traces/
├── exports/
└── registry.db
```

### 8.3 初始化方式

- 本地和 CI 都通过 `./scripts/init-artifact-store.sh` 初始化
- `run status`、`trace show`、`trace events` 等命令默认读根级 artifact store

## 9. 环境要求

| 工具 | 当前最低要求 |
| --- | --- |
| Rust | stable latest |
| Python | 3.9+（schema 校验脚本） |
| SQLite | 3.x |

补充说明：

- Python 分析工具链尚未进入当前主线，因此当前不要求 `ruff`、`pytest` 或完整 `pyproject.toml`
- MCP adapter 相关 Node.js 依赖仍然属于后续阶段，不是当前前置条件

## 10. 结论

当前工作流的核心约束只有三条：

1. `vidodo-docs/`、`schemas/`、`scripts/`、`tests/`、`vidodo-src/` 的职责边界不能再漂移。
2. 任何实现都必须围绕当前唯一主线闭环，而不是提前放大外围能力面。
3. schema、fixture、实现、CI 和文档必须在同一次改动中保持一致。
