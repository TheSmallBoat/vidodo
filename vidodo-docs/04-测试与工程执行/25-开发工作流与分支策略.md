# 开发工作流与分支策略

状态：Draft

## 1. 文档目的

本文定义代码仓库结构、分支策略、CI 流程、PR 规范与文档更新机制，为多人并行开发提供统一工程协作规范。

## 2. 仓库结构

### 2.1 Monorepo 策略

推荐使用单仓库 monorepo，原因：

- 核心模块之间共享 IR 类型、schema 和 proto 定义
- CI 可跨模块运行一致性检查
- schema 文件变更可触发关联模块测试

### 2.2 推荐目录布局

```text
av-system/
├── apps/
│   ├── avctl/                    # CLI 主程序（Rust）
│   ├── core-service/             # Core Control Service（Rust）
│   ├── visual-runtime/           # Visual Runtime（Rust + wgpu）
│   └── mcp-adapter/              # MCP 适配层
├── crates/
│   ├── ir/                       # IR 类型定义
│   ├── validator/                # 校验器
│   ├── compiler/                 # 编译器 + Revision Manager
│   ├── scheduler/                # 调度器 + Musical Clock
│   ├── patch-manager/            # Patch 生命周期管理
│   ├── trace/                    # Trace Writer + Query
│   └── storage/                  # SQLite 索引 + Artifact Store
├── python/
│   ├── ingestion/                # 素材接入
│   └── analyzers/                # 分析任务
├── schemas/
│   ├── common/                   # 通用对象头、envelope、diagnostic
│   ├── planning/                 # SetPlan、AudioDsl、VisualDsl、ConstraintSet
│   ├── asset/                    # AssetRecord
│   ├── ir/                       # Asset IR、Structure IR、Performance IR、Visual IR、Timeline
│   ├── runtime/                  # Event Header + 六类 Event Payload
│   ├── patch/                    # LivePatchProposal、PatchTicket
│   ├── trace/                    # Trace Manifest
│   └── mcp-tools/                # MCP Tool Registry（已存在）
├── tests/
│   ├── schema/                   # Schema 正例/反例 fixture
│   ├── fixtures/                 # 测试素材、plan bundle、patch 文件
│   ├── unit/                     # 单元测试辅助
│   ├── integration/              # 模块级集成测试
│   └── e2e/                      # 端到端集成测试
├── docs/
│   └── design/                   # 当前设计文档迁移位置
├── scripts/
│   ├── init-artifact-store.sh    # Artifact Store 初始化
│   ├── schema-validate.sh        # Schema 批量校验
│   └── ci/                       # CI 辅助脚本
├── artifacts/                    # 运行时产物（gitignore）
│   ├── assets/
│   ├── analysis/
│   ├── traces/
│   ├── exports/
│   └── registry.db
├── Cargo.toml                    # Rust workspace
├── pyproject.toml                # Python 项目配置
├── .gitignore
└── README.md
```

### 2.3 哪些目录进入版本控制

| 目录 | 进入 git | 说明 |
| --- | --- | --- |
| apps/ | 是 | 应用代码 |
| crates/ | 是 | 库代码 |
| python/ | 是 | 分析工具链 |
| schemas/ | 是 | 工件 Schema，一等公民 |
| tests/ | 是 | 测试代码与 fixture |
| docs/ | 是 | 设计文档 |
| scripts/ | 是 | 构建与辅助脚本 |
| artifacts/ | 否 | 运行时产物，由 .gitignore 排除 |

## 3. 分支策略

### 3.1 主干发展模型

采用 trunk-based development 简化变体：

```text
main ──────────────────────────────────────────────►
  │                                        │
  ├── feat/wsa-01-object-header-schema     │
  ├── feat/wsb-06-beat-track               │
  ├── feat/wsc-04-compiler-main            │
  └── fix/schema-validation-edge-case      │
```

### 3.2 分支命名规范

| 分支类型 | 命名模式 | 示例 |
| --- | --- | --- |
| 功能分支 | `feat/{任务卡号}-{简短描述}` | `feat/wsa-02-setplan-schema` |
| 修复分支 | `fix/{简短描述}` | `fix/asset-hash-collision` |
| 文档分支 | `docs/{简短描述}` | `docs/update-design-doc-14` |
| 重构分支 | `refactor/{简短描述}` | `refactor/ir-type-cleanup` |

### 3.3 分支生命周期

- 功能分支从 main 创建
- 开发完成后通过 PR 合入 main
- PR 合入后删除功能分支
- 不维护长生命周期的 develop 分支

### 3.4 保护规则

- main 分支禁止直接 push
- 合入 main 必须通过 PR
- PR 至少需要 1 名 reviewer approve
- CI 全部通过后才可合入

## 4. Commit 规范

### 4.1 格式

```text
<type>(<scope>): <subject>

[body]

[footer]
```

### 4.2 Type 枚举

| Type | 含义 |
| --- | --- |
| feat | 新功能 |
| fix | 修复 |
| schema | Schema 定义变更 |
| test | 测试代码 |
| refactor | 重构 |
| docs | 文档更新 |
| ci | CI 配置 |
| chore | 构建或辅助工具 |

### 4.3 Scope 枚举

| Scope | 对应模块 |
| --- | --- |
| ir | crates/ir |
| validator | crates/validator |
| compiler | crates/compiler |
| scheduler | crates/scheduler |
| patch | crates/patch-manager |
| trace | crates/trace |
| storage | crates/storage |
| avctl | apps/avctl |
| visual | apps/visual-runtime |
| ingestion | python/ingestion |
| analyzer | python/analyzers |
| schema | schemas/ |
| e2e | tests/e2e |

### 4.4 示例

```text
schema(planning): add SetPlan v0 JSON Schema

- Define required fields: show_id, mode, goal, sections
- Add positive and negative test fixtures
- Ref: WSA-02

feat(compiler): implement Planning DSL parser

- Read SetPlan + AudioDsl + VisualDsl from JSON
- Deserialize into internal IR types
- Ref: WSC-02
```

## 5. PR 规范

### 5.1 PR 标题

格式与 commit 一致：`<type>(<scope>): <subject>`

### 5.2 PR 描述模板

```markdown
## 任务卡

关联：{任务卡号}，例如 WSA-02

## 变更内容

简述本次变更做了什么。

## 交付物

- [ ] 新增/修改的文件列表
- [ ] 通过的测试

## 验收标准

复述任务卡中的验收标准，标注完成情况。

## 注意事项

需要 reviewer 特别关注的决策或风险。
```

### 5.3 Review 检查清单

Reviewer 应至少检查：

- [ ] 变更是否与任务卡验收标准一致
- [ ] Schema 变更是否有配套 fixture 更新
- [ ] 新增类型是否与对应 JSON Schema 对齐
- [ ] 错误码是否使用 23 号文档定义的标准码
- [ ] 测试覆盖是否充分
- [ ] 是否引入不必要的跨模块依赖

## 6. CI 流程

### 6.1 触发条件

| 触发 | 运行内容 |
| --- | --- |
| PR 创建/更新 | 全部 CI job |
| push to main | 全部 CI job + artifact 发布 |

### 6.2 CI Job 定义

```text
Job 1: Schema Validation
  - 运行 scripts/schema-validate.sh
  - 校验 tests/schema/ 下所有正例通过、反例拒绝

Job 2: Rust Build & Test
  - cargo build --workspace
  - cargo test --workspace
  - cargo clippy --workspace -- -D warnings

Job 3: Python Lint & Test
  - ruff check python/
  - pytest python/

Job 4: Integration Tests
  - 运行 tests/integration/ 下跨模块集成测试

Job 5: E2E Tests（仅 main 分支或手动触发）
  - 运行 tests/e2e/ 端到端测试
```

### 6.3 CI 失败策略

- 任何 Job 失败都阻止 PR 合入
- Schema Validation 失败为最高优先级修复项
- Clippy warning 视为 error

## 7. Schema 变更流程

Schema 是系统工件面的核心，变更需要特殊流程：

### 7.1 变更步骤

1. 修改 `schemas/` 下的 JSON Schema 文件
2. 同步更新 `tests/schema/` 下的正例和反例 fixture
3. 同步更新 `crates/ir/` 中对应的 Rust 类型定义
4. 如影响 CLI 输出，同步更新 avctl envelope/diagnostic 格式
5. PR 描述中明确标注 schema 变更范围

### 7.2 版本策略

- Schema 文件名包含版本号：`set-plan.v0.json`
- 非兼容变更必须升版本号
- 旧版 Schema 文件保留，不删除
- Rust 类型与 Schema 版本号保持同步

## 8. 文档与代码同步机制

### 8.1 触发更新条件

以下变更必须同步更新设计文档：

- IR 类型字段增删
- 事件模型变更
- Patch 状态机变更
- CLI 命令增删或输出格式变更
- 新增模块或模块职责调整

### 8.2 更新方式

- 设计文档存放在 `docs/design/`（从当前目录迁入）
- 文档更新作为 PR 的一部分提交，不允许代码合入后再补文档
- 如果文档更新范围较大，允许单独 `docs/` PR 但必须在同一 milestone 内完成

### 8.3 文档状态标记

- `Draft`：初稿
- `Active`：编码中的有效参考
- `Superseded`：已被新文档替代（注明替代文档编号）

## 9. Artifact 管理

### 9.1 本地开发

- `artifacts/` 目录不入 git
- 开发者通过 `scripts/init-artifact-store.sh` 初始化本地目录结构与 SQLite
- 测试使用 `tests/fixtures/` 目录下的受控素材

### 9.2 CI 环境

- CI 每次运行初始化临时 artifact store
- 不依赖持久化存储
- 测试素材从 `tests/fixtures/` 读取

## 10. 发布策略

### 10.1 Phase 0 阶段

- 不做正式版本发布
- main 分支随时可构建、可测试
- 里程碑达成时打 git tag：`m0-schema-fixed`、`m1-asset-ingest`、...

### 10.2 命名规范

- 里程碑 tag：`m{N}-{简短描述}`
- 未来正式版本：`v0.1.0`（遵循 SemVer）

## 11. 环境要求

### 11.1 开发环境最低要求

| 工具 | 版本要求 |
| --- | --- |
| Rust | stable latest |
| Python | 3.11+ |
| SQLite | 3.x |
| ffmpeg | 6.x+（导出相关任务） |
| Node.js | 20+（MCP adapter 相关任务） |

### 11.2 推荐工具

- cargo-watch：开发期自动重编译
- ruff：Python lint
- pytest：Python 测试
- ajv-cli 或等价工具：JSON Schema 校验

## 12. 结论

本文定义的工作流侧重 Phase 0 阶段的小团队快速推进。核心原则是：

1. Schema 变更优先、CI 门禁严格、分支短命。
2. PR 必须关联任务卡、满足验收标准、通过 CI。
3. 文档与代码同步更新，不允许漂移。
