# Vidodo 第三方系统审计报告

日期：2026-04-02  
审计角色：第三方架构与工程审计  
审计范围：产品边界、系统架构、顶层设计、模块职责、协议与 Schema、测试与验证、已完成 Rust 代码、脚本、产物与追踪链路  
仓库快照：以当前工作区为准，覆盖 `vidodo-docs/`、`schemas/`、`tests/`、`scripts/`、`artifacts/`、`vidodo-src/`，并纳入 `.github` 下 Copilot 指令与工程约束。

## 一、审计结论

本项目的总体方向是正确的，且从第一性原理看，核心方法论具有较强可行性：以外部规划为输入，以确定性编译内核为中心，以统一时间模型驱动音频/视觉/补丁/追踪，并以可回放、可验证、可回退的产物链替代不可控脚本执行。这一设计在产品边界、工程抽象和 Phase 0 的最小闭环思路上是成立的。

但当前项目存在一个明显特征：`设计成熟度` 高于 `契约收口度`，而 `契约收口度` 又高于 `运行时闭环完整度`。换句话说，文档体系已经把中长期方向描述得很完整，但真正影响系统可交付性的关键问题，仍集中在几个更基础的点：协议字段是否完全一致、补丁与回退是否真正闭环、测试是否覆盖负路径与边界条件、以及当前代码实现是否与“Phase 0 已完成范围”对齐。

综合评估如下：

| 维度 | 评价 | 结论 |
| --- | --- | --- |
| 产品边界与顶层目标 | 强 | 边界清晰，外部规划与执行内核分离是正确路线 |
| 架构方向 | 强 | 统一 IR、统一时间、确定性产物链设计成立 |
| 协议与 Schema 收口 | 中 | 结构完整，但已有若干契约不一致问题 |
| Phase 0 实现闭环 | 中 | 编译/调度/追踪主链已成型，运行时与回退仍偏弱 |
| 测试与验证 | 中 | Happy path 较好，负路径和覆盖密度不足 |
| 工程实现质量 | 中上 | Rust 代码整体干净，CLI 稳健性仍需加强 |

本次审计的总判断是：**项目可以继续推进，但在进入更大范围扩展之前，必须先完成“契约收口 + 回退闭环 + 测试补全”三项收敛工作。** 若不先解决这三类问题，后续再叠加 MCP、多视图、灯光、分布式运行时，只会扩大返工面。

## 二、从第一性原理审视项目

如果把 Vidodo 抽象成一个最小可证伪系统，其本质不是“做一个很大的多媒体平台”，而是要回答以下五个问题：

1. 外部规划能否被转换为稳定、确定的内部执行表示。
2. 音频、视觉、补丁是否共享同一个时间与语义参照系。
3. 运行时修改是否有明确边界、授权窗口和失败回退路径。
4. 整个系统是否能把输入、编译、执行、补丁、回退写成可复核的证据链。
5. 当规模扩大时，系统是否还能依靠契约而不是依赖隐式行为维持一致性。

从这五个问题出发，本项目有三点值得肯定：

- 文档层面已经明确采用“外部规划，内部确定执行”的架构，不把生成式能力和执行内核混在一起。
- 统一 Musical Time、IR、Timeline、Trace Bundle 的主线逻辑一致，适合作为 Phase 0 的技术主轴。
- 产物、追踪、补丁、回退都采用结构化对象表达，而不是散乱脚本，这为长期可审计性打下了正确基础。

同时，也暴露出三类核心风险：

- 若 Schema 和运行时字段语义不完全一致，所谓“统一协议”只会停留在文档层。
- 若补丁回退只是对象层面存在，而资源恢复、状态恢复、验证恢复没有闭环，系统安全性会被高估。
- 若测试主要覆盖 happy path，而缺少边界错误、冲突、资源退化与回退失败验证，系统会在最需要稳定的时候暴露问题。

## 三、最高优先级审计发现

以下问题建议按优先级立即处理。

### P0：核心协议契约存在不一致，必须先收口

这是本次审计最重要的问题。当前 `schemas/` 中已有多处字段定义不一致，会直接削弱跨模块互操作性。

#### 1. `route_mode` 枚举在 IR 与 Runtime 中不一致

- [schemas/ir/route-set.v0.json](../../schemas/ir/route-set.v0.json) 定义为：`discrete`、`group`、`fanout`、`priority_failover`
- [schemas/runtime/audio-event.v0.json](../../schemas/runtime/audio-event.v0.json) 定义为：`bus`、`discrete`、`group`、`object`

这不是简单的命名问题，而是契约层级问题。若 `route-set` 是拓扑级权威定义，runtime 事件就不应该发出另一个并不兼容的枚举集合；若两者本来就是不同层级语义，则必须显式声明“编译期路由模式”和“运行期输出模式”的映射关系，而不是复用同一个字段名。

#### 2. `scope` 在补丁提案与运行时补丁事件中的类型不一致

- [schemas/patch/live-patch-proposal.v0.json](../../schemas/patch/live-patch-proposal.v0.json) 中，`scope` 是结构化对象，包含 `from_bar`、`to_bar`、`window`
- [schemas/runtime/patch-event.v0.json](../../schemas/runtime/patch-event.v0.json) 中，`scope` 却是字符串

这会导致“提案可验证、运行时不可重建语义”的问题。对审计和回放而言，运行期记录应能保留足够结构化信息，否则补丁决策无法做严格比对。

#### 3. `status` 枚举在能力层与资源层不一致

- [schemas/capability/adapter-plugin-manifest.v0.json](../../schemas/capability/adapter-plugin-manifest.v0.json) 使用 `ready`
- [schemas/asset/resource-hub-descriptor.v0.json](../../schemas/asset/resource-hub-descriptor.v0.json) 及若干拓扑 Schema 使用 `available`

如果 `ready` 与 `available` 有意区分，需要在文档中定义区别；如果没有语义差异，则应统一，避免上层系统出现条件分支膨胀。

#### 4. `mcp-tools` 的 `$id` 命名未遵循全局规则

- [schemas/mcp-tools/av-tool-registry.v0.json](../../schemas/mcp-tools/av-tool-registry.v0.json) 当前 `$id` 为 `av-tool-registry.v0`
- 其他 Schema 基本遵循 `av.{category}.{name}.v0` 规则

这会增加 Schema 注册、工具发现和后续自动校验的复杂度。建议统一为带命名空间的形式。

#### 5. 多个 IR Schema 仍处于“结构壳体存在，但内部对象未收口”状态

`asset-ir`、`structure-ir`、`timeline-entry`、`visual-ir` 中均存在若干裸对象字段。对早期探索来说可以接受，但在 Phase 0 要求“最小闭环可验证”的前提下，这种开放性需要更明确地被标记为：

- 当前故意保留为扩展点；或
- 当前尚未完成定义，不应被视为已稳定协议。

若两者不区分，团队会误判哪些字段已能当作稳定接口使用。

### P1：Phase 0 实现边界与未来态文档边界尚未完全对齐

从代码现状看，当前真正成型的是：验证器、编译器、补丁管理、调度器、存储、追踪，以及 `avctl` 这一 CLI 主入口。这个主线是可运行的，而且方向正确。

但与此同时，以下应用目前仍是占位态：

- [vidodo-src/apps/core-service/src/main.rs](../../vidodo-src/apps/core-service/src/main.rs)
- [vidodo-src/apps/mcp-adapter/src/main.rs](../../vidodo-src/apps/mcp-adapter/src/main.rs)
- [vidodo-src/apps/visual-runtime/src/main.rs](../../vidodo-src/apps/visual-runtime/src/main.rs)

这意味着文档中已经展开描述的“能力层服务化”“MCP 工具面”“真实视觉运行时”在当前代码快照里并未真正闭环。因此建议增加一份显式的“实现状态矩阵”，把内容拆成三类：

- 已实现并可验证
- 已设计但未实现
- 明确延后到下一阶段

否则，外部阅读者会把文档中的未来态误解为当前态。

### P1：补丁回退语义在文档层很强，在实现层仍需补全闭环

项目对 patch/rollback 的理念是对的：窗口化授权、结构化提案、候选 revision、fallback revision、回退记录、追踪证据链。这套思路符合实时系统的正确方法。

问题在于，当前审计证据更多证明了“补丁对象与补丁决策存在”，而不是完全证明了“资源状态恢复、运行时状态恢复、回退后持续一致性”已经闭环。例如：

- 老 revision 的资源句柄在回退后如何释放或保温，文档仍偏抽象。
- 补丁前后多视图/多后端状态如何重新对齐，文档中未完全落细。
- 回退后的检查点、热状态、缓存命中与 trace 关联策略仍未完全收口。

因此，本次审计认为：**补丁能力已经形成主干，但“安全回退”尚未达到可以过度自信宣传的程度。**

### P1：CLI 主入口存在一批用户面稳健性问题

Rust workspace 整体质量不错，但 [vidodo-src/apps/avctl/src/main.rs](../../vidodo-src/apps/avctl/src/main.rs) 仍存在一批 `unwrap()` 使用，尤其集中在：

- revision 相关 JSON 序列化输出
- 补丁历史的最后一个决策提取
- CLI 输出路径上的若干快捷处理

这些问题并不意味着系统架构错误，但会让用户面程序在异常输入或未来代码演进时更容易出现非预期退出。对于 Phase 0，这类问题不一定致命，但应该作为“收口前必须消除的工程债”。

## 四、架构与模块功能审计

### 4.1 总体架构评价

项目当前最强的部分是“编译型执行内核”的设计：

- 外部输入层：Plan / DSL / ConstraintSet
- 内部确定层：Validator / Compiler / IR / Timeline
- 执行驱动层：Scheduler
- 证据层：Storage / Trace / Artifacts
- 变更层：Patch / Rollback

这个架构的优点是，复杂性被压缩到可验证的中间表示，而不是直接把复杂逻辑留给运行时即时决策。这是非常重要的正确选择。

### 4.2 模块职责划分评价

当前 Rust workspace 的 crate 划分是合理的：

- `ir`：数据结构中心
- `validator`：约束校验
- `compiler`：规划到 revision 的转换
- `scheduler`：时序展开与事件驱动
- `patch-manager`：补丁安全检查与应用
- `storage`：持久化与注册表
- `trace`：运行追踪落盘
- `apps/avctl`：CLI 编排入口

这说明工程实现并没有偏离文档的大方向。

但也存在两个值得提前处理的结构问题：

#### 1. `storage` crate 责任偏重

当前 `storage` 同时承载：目录布局、文件落盘、资产注册、分析缓存、查询与部分辅助逻辑。对于 Phase 0 可接受，但如果继续增长，后续会成为最难演化的模块。建议尽早拆分为至少三类边界：

- artifact layout / file persistence
- asset ingest / normalization metadata
- query / registry / analysis cache

#### 2. “文档中的运行时系统”与“代码中的当前闭环”之间缺少映射文档

现在代码更像一个功能完整的 Phase 0 内核，而不是一个完整的全栈系统。建议在 `vidodo-docs` 增加一份单独状态页，明确：

- 当前真正可运行的是哪个闭环
- 哪些模块只是预留占位
- 哪些文档描述的是 Phase 1/Phase 2 目标态

## 五、协议与 Schema 审计

### 5.1 正面评价

Schema 体系总体基础不错：

- 基本统一使用 JSON Schema Draft 2020-12
- 文件命名风格较稳定
- 字段命名基本统一为 snake_case
- `$ref` 关系有效，未发现明显悬空引用或循环引用
- `common/`、`runtime/`、`planning/` 的骨架较清晰

这说明团队在“先定义契约，再做实现”这件事上是有纪律的。

### 5.2 主要问题

除了上文 P0 问题外，还存在几个需要尽快统一的小问题：

- `musical-time` 中 `beat_in_bar` 缺少与其他数值字段同等级的边界约束。
- 部分 DSL/IR 字段虽然有 `type: object`，但缺少最小结构语义，不利于下游工具做更精确验证。
- 一些能力层与资源层对象虽然都有“状态”字段，但没有形成统一的状态模型和生命周期模型。

### 5.3 审计判断

本次审计认为：**Schema 体系已经从“想法”进入“可执行草案”，但还没有进入“可长期锁定的系统契约”阶段。** 当前最应该做的不是继续增加更多 Schema，而是把已经存在的关键 Schema 彻底对齐。

## 六、测试、验证与质量门审计

### 6.1 已有优势

测试体系比很多早期项目更成熟，尤其有以下优点：

- `scripts/schema_validate.py` 结构清晰，能够对合法/非法 fixture 做系统校验。
- `scripts/init-artifact-store.sh` 具备良好的幂等性与初始化价值。
- `tests/e2e/phase0_smoke.sh` 和 `tests/e2e/asset_ingest_smoke.sh` 已能覆盖一条较完整的 happy path。
- `artifacts/` 下的 revision、trace、summary、show-state 等产物整体保持一致，说明“证据链思路”不是停留在口头层。

### 6.2 主要不足

#### 1. 仍有一批 Schema 没有 fixture 覆盖

本次审计中，以下方向的覆盖仍明显不足：

- `ir/` 中部分拓扑与资产相关 Schema
- `mcp-tools/av-tool-registry.v0.json`
- `runtime/semantic-event.v0.json`
- `runtime/transport-event.v0.json`

这说明“Schema 已定义”不等于“Schema 已验证”。

#### 2. E2E 主要覆盖 happy path，负路径明显不足

当前测试更擅长证明“系统能工作”，但还不够擅长证明“系统在失败条件下仍然安全可控”。建议补充：

- 缺失资产引用
- 无效 patch scope
- 冲突 patch
- 资源退化触发 fallback
- trace/manifest 不一致
- 资产 ingest 部分失败

#### 3. 尚未形成回退闭环专用测试集

既然补丁与回退是项目卖点之一，那么回退测试不应只是 `rollback` 命令存在，而应有明确断言：

- 回退后 revision 是否与 fallback revision 一致
- 回退后 show state 是否落回预期窗口
- 回退前后的 trace 是否能建立因果链
- 回退后的资源引用是否没有悬挂状态

### 6.3 审计判断

当前测试体系可支持 Phase 0 的继续推进，但还不能支撑更高强度的对外承诺。若要把系统从“内部验证样机”推进到“可稳定演示的工程原型”，测试重心必须从 happy path 转向契约、边界与失败恢复。

## 七、代码实现审计

### 7.1 正面评价

Rust 代码整体表现良好，尤其体现在：

- 工作区强制 `unsafe_code = "forbid"`
- crate 依赖关系比较干净，没有明显架构反向依赖
- 多数核心逻辑以 `Result` 和结构化诊断对象表达，不靠 panic 驱动流程
- SQL 查询采用参数化方式，未发现明显注入风险
- 代码与文档主线整体一致，没有出现“文档说一套、代码完全另一套”的严重偏航

### 7.2 主要工程问题

#### 1. `avctl` 作为用户面入口，异常处理应比当前更严格

当前 CLI 承载了大量编排职责，因此其健壮性不能按“内部脚本”标准看待。建议把所有 `unwrap()` / 假定存在路径都替换为结构化错误输出，确保 CLI 失败时仍能产出可读、可追踪的诊断信息。

#### 2. 资产 ingest 对大文件和路径策略仍可进一步收紧

存储层当前已具备较好基础，但从审计视角看，仍建议增加：

- 更明确的输入大小边界
- 更严格的路径规范化策略
- 对部分极端文件场景的失败信息细化

这不是因为当前实现已经危险，而是因为 ingest 会天然成为未来高风险入口。

#### 3. 公开模块文档仍偏少

当前代码风格偏“自解释”，这对熟悉项目的人足够，但对协作和审计来说还不够。建议为每个核心 crate 增加简洁的模块级说明，明确：

- 输入对象是什么
- 输出对象是什么
- 该模块不负责什么

## 八、产物、Trace 与可审计性审计

这是项目当前做得最接近目标态的一块。

`artifacts/revisions/`、`artifacts/traces/` 下的产物结构、JSONL 事件、summary、show-state 等内容，已经体现出较强的证据链意识。对于一个以“可解释、可追踪、可回放”为重要目标的系统来说，这一点非常关键。

本次审计确认：

- 产物链不是空壳，已有可读、可落盘、可检查的数据结构。
- trace 中的事件、节拍、状态信息已经具备较强审计价值。
- patch 决策与 revision 产物已能形成基本关联。

但下一步需要把“可记录”进一步推进为“可证明”：

- 明确检查点保留策略
- 明确 rollback 前后状态等价性的校验方式
- 明确资源回收与热状态保留的证据表达

## 九、建议的调整与优化路线

建议按三阶段推进，而不是并行扩展所有方向。

### 第一阶段：先做契约收口

目标：把系统最核心的协议面固定下来。

建议动作：

1. 统一 `route_mode` 的层级语义与枚举集合。
2. 统一 patch `scope` 的结构表达，至少保证运行时记录可重建提案语义。
3. 统一 `status` 状态模型，明确 `ready` 和 `available` 是否同义。
4. 修正 `mcp-tools` Schema 命名空间规则。
5. 给“开放对象”字段打上明确标签：稳定扩展点或暂未定稿。

### 第二阶段：补全运行时与回退闭环

目标：把“系统能 patch”提升为“系统能安全 patch 和安全 rollback”。

建议动作：

1. 为 rollback 建立独立测试集与检查点验证逻辑。
2. 明确资源释放、热状态迁移、fallback 恢复策略。
3. 增加一份 Phase 0 实现状态矩阵，防止未来态文档和当前态实现混淆。
4. 将 `avctl` 中用户面异常处理全部收口为结构化诊断。

### 第三阶段：再扩大验证范围

目标：在闭环稳定后，扩展复杂度而不扩大不确定性。

建议动作：

1. 补齐缺失 Schema fixture。
2. 增加负路径 E2E 和冲突/退化类测试。
3. 对 `storage` 做职责拆分预案。
4. 再进入 MCP、多视图、灯光、分布式等更高复杂度扩展。

## 十、最终判断

Vidodo 不是一个“缺少想法”的项目，相反，它的问题恰恰在于：想法、文档、长期目标已经足够多，下一步最重要的不是继续扩张设计边界，而是把最关键的几个系统性质真正做实。

本次审计给出的最终判断如下：

- **架构方向正确，应继续推进。**
- **Phase 0 主链已具备雏形，但不宜过早宣称完整运行时能力已经闭环。**
- **当前最该做的是收口，而不是扩面。**
- **只要优先解决契约一致性、回退闭环与测试负路径覆盖，这个项目的工程可信度会显著提升。**

如果必须给出一句最简结论，那就是：

> Vidodo 当前最有价值的资产，不是它已经实现了多少功能，而是它已经选择了正确的系统边界和正确的工程方法；下一步成败，取决于团队能否在扩张之前，先把这些边界真正锁死并验证到底。
