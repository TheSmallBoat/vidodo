# 功能模块设计方案总览

日期：2026-04-01
状态：Draft
前置阅读：05-DSL与IR字段定义草案.md，07-追踪回放与评估数据模型.md，10-CLI命令清单与MCP工具Schema.md，14-MVP技术架构图与Phase0实施清单.md

## 1. 文档目的

本文把已经确定的产品边界、技术栈和 MVP 架构，继续收敛为“可实现的功能模块设计”。

目标不是重复技术栈，而是明确：

1. 系统到底由哪些核心功能模块组成。
2. 每个模块的职责边界、输入输出和状态是什么。
3. 模块之间通过什么对象和接口协作。
4. MVP 阶段每个模块做到什么程度算足够。

## 2. 设计原则

### 2.1 模块围绕工件与状态，而不是围绕界面

本系统不以 GUI 页面或某个编辑器为组织中心，而以以下对象为中心：

- Asset
- Plan
- IR
- Timeline
- Show State
- Patch
- Trace
- Evaluation

### 2.2 模块边界优先于实现细节

在当前阶段，最重要的不是每个模块内部具体用哪个 crate 或库，而是：

- 模块是否有清晰职责
- 输入输出是否结构化
- 状态归属是否明确
- 失败时是否可诊断和可回退

### 2.3 实时链路与非实时链路必须分离

以下模块默认不在实时热路径内：

- Asset Ingestion
- Analysis
- Compiler
- Evaluation
- Replay

以下模块位于执行主链路：

- Scheduler
- Audio Runtime Client
- Visual Runtime

## 3. 模块总览

### 3.1 核心模块清单

| 模块 | 主要职责 | 运行属性 |
| --- | --- | --- |
| Capability Layer | 接收 CLI / 文件 / MCP 请求 | 常驻 |
| Asset Ingestion | 导入素材、生成资产记录 | 按需 |
| Analysis Pipeline | 生成音频 / 视觉分析结果 | 按需 |
| Asset Registry | 管理资产索引与分析引用 | 常驻 |
| Validator | 做 schema、资源、约束校验 | 按需 |
| Compiler | 规划工件编译为 IR / Timeline | 按需 |
| Revision Manager | 管理版本、发布、冻结与切换 | 常驻 |
| Scheduler | 统一时钟与事件发布 | 常驻 |
| Audio Runtime Client | 把时间线动作翻译为音频命令 | 常驻 |
| Visual Runtime | 消费视觉 IR 并执行渲染 | 常驻 |
| Patch Manager | patch 检查、授权、生效、回退 | 常驻 |
| Trace Writer | 记录事件、patch、资源与导出证据 | 常驻 |
| Evaluation Engine | 汇总评分与事实报告 | 按需 |
| Replay Engine | 重演 revision / run / patch 过程 | 按需 |

## 4. 模块依赖关系

### 4.1 总体关系

```text
External Request
  -> Capability Layer
  -> Validator
  -> Compiler / Patch Manager / Query Modules
  -> Revision Manager
  -> Scheduler
  -> Audio Runtime Client / Visual Runtime
  -> Trace Writer
  -> Evaluation / Replay

Asset Side Path
  -> Asset Ingestion
  -> Analysis Pipeline
  -> Asset Registry
  -> Validator / Compiler / Patch Manager
```

### 4.2 一条请求的标准流转

以 `plan.submit -> compile.run -> run.start` 为例：

```text
Capability Layer
  -> Validator
  -> Compiler
  -> Revision Manager
  -> Scheduler
  -> Audio Runtime Client / Visual Runtime
  -> Trace Writer
```

## 5. 各模块设计

### 5.1 Capability Layer

职责：

- 接收 CLI、文件式请求、MCP 调用
- 统一请求 envelope
- 路由到对应能力模块
- 生成 request_id、ticket_id、operation_id

输入：

- PlanSubmission
- PatchProposal
- ReplayRequest
- EvaluationRequest
- AssetIngestRequest

输出：

- 结构化响应 envelope
- diagnostics
- artifact refs
- ticket / operation 状态

持有状态：

- 请求上下文
- 调用身份与权限上下文
- 操作流水号

不负责：

- 不做业务语义判断终局决定
- 不直接控制 Runtime 内部状态

MVP 范围：

- 只支持 CLI 和文件接口
- MCP 可先作为薄适配层对接同一 capability surface

### 5.2 Asset Ingestion

职责：

- 发现候选素材
- 生成 content hash 与 probe 信息
- 触发规范化与分析任务
- 发布资产记录

输入：

- 本地目录
- 单文件路径
- 导入 profile

输出：

- IngestionRun
- AssetRecord
- normalized locator
- analysis job

持有状态：

- ingestion run 状态
- 导入错误与警告

失败处理：

- 单资产失败不阻塞整批
- publish 失败不暴露半成品 asset

MVP 范围：

- 支持固定目录导入
- 支持有限音频与视觉资产类型

### 5.3 Analysis Pipeline

职责：

- 运行音频结构分析
- 运行视觉素材预检与统计
- 生成 analysis cache entry

输入：

- normalized asset
- analyzer profile
- analyzer version

输出：

- beat / section / key / energy 等分析结果
- shader compile probe / texture stats
- AnalysisCacheEntry

持有状态：

- analysis job 生命周期
- 缓存命中与失效状态

MVP 范围：

- 音频先支持 beat_track、section_segmentation
- 视觉先支持 shader probe、基本资源统计

### 5.4 Asset Registry

职责：

- 提供资产索引与查询
- 维护 asset 与 analysis refs 的关系
- 为 compiler / patch manager 提供素材事实基础

输入：

- AssetRecord
- Analysis refs
- tag / capability 更新

输出：

- 可查询资产列表
- asset compatibility 结果
- asset warmness / readiness 状态

持有状态：

- SQLite 索引
- asset publish 状态
- warm / missing / stale 标记

### 5.5 Validator

职责：

- schema 校验
- 资源预算校验
- 锁定区间检查
- patch window 检查
- asset availability 检查

输入：

- Plan bundle
- Patch proposal
- Revision diff
- Runtime 状态快照

输出：

- diagnostics
- accept / reject / warning 结果

持有状态：

- 规则配置
- 资源阈值
- policy profile

MVP 范围：

- 先覆盖 schema、资源预算、锁区检查、asset readiness

### 5.6 Compiler

职责：

- 读取 Planning DSL
- 生成 Asset IR、Structure IR、Performance IR、Visual IR
- 产出 Unified Timeline

输入：

- SetPlan
- AudioDsl
- VisualDsl
- ConstraintSet
- Asset Registry 数据

输出：

- candidate revision
- compile artifacts
- timeline entries

持有状态：

- compiler run context
- artifact refs

失败处理：

- 失败时输出结构化 compile diagnostics
- 不污染已发布 revision

MVP 范围：

- 支持有限 section / layer / scene / transition 模型

### 5.7 Revision Manager

职责：

- 分配 revision
- 管理 candidate / published / running / archived 状态
- 维护 base_revision 与 effective_revision 关系

输入：

- compile result
- patch result
- publish request
- rollback result

输出：

- revision record
- revision diff ref
- active revision pointer

持有状态：

- revision 元数据
- published / frozen 状态

### 5.8 Scheduler

职责：

- 维护 musical clock
- 计算 lookahead 窗口
- 分发 timing / audio / visual / semantic 事件
- 管理最小 Show State

输入：

- Unified Timeline
- active revision
- patch activation info

输出：

- runtime event stream
- timing boundary event
- show state update

持有状态：

- current bar / beat / phrase / section
- lookahead queue
- active event cursor

实时要求：

- 不等待慢查询
- 不依赖外部网络
- 不在热路径内做重型对象构造

### 5.9 Audio Runtime Client

职责：

- 把 Performance IR action 翻译为音频后端命令
- 管理 action 到 server command / node 的映射
- 接收 ack 并回传 Trace Writer

输入：

- audio events
- transport / timing events
- patch activation / rollback instruction

输出：

- 外部音频服务命令
- 执行 ack / error / timeout
- resource summary

持有状态：

- active audio command mapping
- node / buffer 映射表
- rollback command set

MVP 范围：

- 先支持有限动作：load buffer、launch asset、set gain、set filter、crossfade、stop layer

### 5.10 Visual Runtime

职责：

- 执行 scene、uniform、camera、render mode 变更
- 消费 shared time 与 semantic state
- 维护视觉执行状态与资源预算

输入：

- visual events
- timing / semantic events
- patch / degrade / rollback command

输出：

- scene ack
- resource sample
- render export frame / stream

持有状态：

- active scene
- uniform state
- camera state
- render budget mode

MVP 范围：

- 先支持 scene enter / update、uniform automation、camera preset

### 5.11 Patch Manager

职责：

- 检查 patch proposal
- 执行授权和窗口判断
- 生成 patch ticket
- 触发生效、监控、回退和降级

输入：

- PatchProposal
- run status
- validator result
- asset readiness

输出：

- patch decision
- effective revision
- rollback instruction

持有状态：

- patch lifecycle state
- patch window state
- fallback revision

MVP 范围：

- 只支持 local_content 类 patch
- 只支持 deferred rollback

### 5.12 Trace Writer

职责：

- 记录 event_record
- 记录 patch decision
- 记录 resource sample
- 生成 trace bundle manifest

输入：

- scheduler event
- runtime ack
- patch state change
- export artifact info

输出：

- JSON / JSONL trace artifacts
- trace bundle manifest

持有状态：

- run_id
- trace_bundle_id
- event sequence

### 5.13 Evaluation Engine

职责：

- 基于 trace 与导出工件生成评价报告
- 产出 score、fact、issue、recommendation

输入：

- trace bundle
- export artifact
- comparison target revision

输出：

- EvaluationReport
- EvaluationFact
- RevisionDiff impact notes

MVP 范围：

- 支持 sync、transition、resource_stability 三个最小维度

### 5.14 Replay Engine

职责：

- 重放某个 revision / run / patch 区间
- 恢复 checkpoint 或结构事件流
- 为对比与调试提供重演结果

输入：

- ReplayRequest
- trace bundle
- checkpoint snapshot

输出：

- replay session
- replay summary
- event diff

MVP 范围：

- 先支持 structural replay 和 deterministic replay 的最小实现

## 6. 模块间共享对象

### 6.1 一等共享对象

| 对象 | 主要生产者 | 主要消费者 |
| --- | --- | --- |
| AssetRecord | Asset Ingestion | Asset Registry / Compiler / Patch Manager |
| AnalysisCacheEntry | Analysis Pipeline | Asset Registry / Compiler |
| SetPlan / AudioDsl / VisualDsl | Capability Layer | Validator / Compiler |
| Asset IR / Structure IR / Performance IR / Visual IR | Compiler | Revision Manager / Scheduler |
| Unified Timeline | Compiler | Scheduler / Patch Manager |
| Show State | Scheduler | Audio Runtime Client / Visual Runtime / Patch Manager |
| PatchProposal / PatchDecision | Capability Layer / Patch Manager | Validator / Trace Writer / Revision Manager |
| EventRecord | Scheduler / Runtime | Trace Writer / Replay / Evaluation |
| EvaluationReport | Evaluation Engine | Capability Layer / Planner |

## 7. 模块级接口面建议

### 7.1 内部接口类型

建议只保留三类内部接口：

- command interface：触发状态变化
- query interface：读取状态或工件
- event interface：异步发布执行结果

### 7.2 不建议的接口风格

- Runtime 直接暴露可随意改写内部状态的对象引用
- patch manager 直接跳过 validator 修改 timeline
- compiler 直接写入运行态 show state

## 8. 失败模型

### 8.1 模块失败分类

| 失败类型 | 典型模块 | 处理原则 |
| --- | --- | --- |
| Input Failure | Capability / Validator | 拒绝并返回 diagnostics |
| Compile Failure | Compiler | 不发布 revision |
| Asset Failure | Ingestion / Registry | 标记 unavailable / stale |
| Runtime Failure | Audio / Visual | 回传 trace，触发 degrade 或 rollback |
| Patch Failure | Patch Manager | 拒绝、延期或回退 |
| Evaluation Failure | Evaluation Engine | 不影响运行主链路 |

### 8.2 通用失败原则

- 所有失败都必须结构化记录
- 失败默认不应破坏已发布 revision
- live 失败优先 degrade，再考虑 stop

## 9. MVP 实施优先级

### 9.1 第一批必须实现

- Capability Layer
- Asset Ingestion
- Asset Registry
- Validator
- Compiler
- Revision Manager
- Scheduler
- Audio Runtime Client
- Visual Runtime
- Patch Manager

### 9.2 第二批可最小占位

- Evaluation Engine

## 10. 最小验收口径

如果以下链路完整成立，可以认为模块设计已经足够支撑 MVP 实现：

```text
asset.ingest
  -> plan.submit
  -> compile.run
  -> run.start
  -> patch.check
  -> patch.submit
  -> patch.rollback
  -> trace.query
  -> eval.run
```

## 12. 结论

功能模块设计阶段的核心任务，不是继续讨论抽象理念，而是把系统拆成一组真正可实现、可验证、可替换的模块。

关键判断如下：

1. 每个模块必须围绕结构化对象和明确状态建模。
2. 实时执行模块与非实时分析模块必须严格分开。
3. patch、trace、revision 不是附属功能，而是跨模块主骨架。

这份文档可以直接作为后续逐模块细化设计的母文档。
