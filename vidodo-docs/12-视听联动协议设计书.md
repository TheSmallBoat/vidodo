# 视听联动协议设计书

日期：2026-04-01
状态：Draft
前置阅读：02-视听系统产品定位与规划方案书.md，04-音频与视觉运行时事件协议与数据结构.md，05-DSL与IR字段定义草案.md，08-实时补丁授权与回退策略.md

## 1. 文档目的

04 文档已经定义了音频 Runtime 与视觉 Runtime 的事件协议和共享状态。

本文进一步回答另一个问题：

```text
当音频与视觉共享时间、状态与事件之后，它们究竟如何联动，
联动到什么粒度，哪些语义必须强绑定，哪些语义只应弱跟随。
```

因此，本文不是重复 Runtime 事件协议，而是定义：

- 视听联动的层级模型
- 音频语义到视觉语义的映射协议
- 联动优先级、容差和冲突解决策略
- live patch 对联动关系的影响边界
- 联动质量如何被 trace 与 evaluation 捕获

## 2. 顶层设计结论

### 2.1 联动不是频谱驱动特效

本系统中的视听联动不应退化成：

- 低频变亮
- 鼓点来时抖一下
- RMS 增大时镜头推近

这些映射可以存在，但只属于最低层联动。

更高价值的联动必须基于：

- section 与 phrase 结构
- transition 阶段
- shared semantic state
- scene intent
- patch window 与 risk mode

### 2.2 联动协议应采用分层模型

推荐将联动拆为五层：

1. Clock Link：共享时间与相位
2. Structural Link：section / phrase / transition 边界联动
3. Semantic Link：energy / tension / density / intent 联动
4. Signal Link：onset / band / transient 等快速响应联动
5. Safety Link：patch、降级、回退、emergency 联动

### 2.3 联动应显式声明耦合强度

不是所有音频变化都必须强驱动视觉。

联动关系应至少声明四种耦合模式：

- locked：强绑定
- guided：结构绑定，局部自由
- reactive：信号响应型
- decoupled：仅共享 show state，不受实时牵引

## 3. 联动对象模型

### 3.1 Link Source

联动的音频侧来源不只有一个。

推荐分为：

| 来源类型 | 典型字段 | 用途 |
| --- | --- | --- |
| Clock Source | beat、bar、phrase、section | 边界对齐 |
| Semantic Source | energy、density、tension、intent | 场景意图映射 |
| Signal Source | low_band、mid_band、onset、crest | 局部快速响应 |
| State Source | patch_allowed、risk_mode、locked_sections | 安全态联动 |

### 3.2 Link Target

视觉侧可被联动的目标建议分为：

- scene selection
- scene phase
- uniform group
- camera state
- geometry density
- post process mode
- render budget mode

### 3.3 Link Binding

单条联动绑定建议采用如下结构：

```json
{
  "link_id": "link-drop-energy-to-corridor-motion",
  "mode": "guided",
  "source": {
    "domain": "semantic",
    "field": "energy"
  },
  "target": {
    "scene_id": "corridor_peak",
    "field": "u_motion_gain"
  },
  "mapping": {
    "curve": "ease_out",
    "input_range": [0.55, 0.95],
    "output_range": [0.42, 0.88],
    "smoothing_ms": 120
  },
  "activation": {
    "section_refs": ["drop_a"],
    "window": "active_only"
  },
  "policy": {
    "priority": 70,
    "suspend_on_degrade": true,
    "fallback_value": 0.58
  }
}
```

## 4. 联动层级协议

### 4.1 Clock Link

Clock Link 是最底层且最强制的联动约束。

它回答：

- 视觉 scene 何时进入
- 动画相位如何跟随 musical phase
- transition window 如何与 phrase / section 对齐

建议字段：

- quantize：beat、bar、phrase、section
- phase_lock：none、soft、hard
- latency_budget_ms
- drift_tolerance_ms

### 4.2 Structural Link

Structural Link 绑定作品结构，而不是即时信号。

例如：

- build_a 必须映射到 compress_then_expand 场景簇
- drop_a 必须映射到 corridor_peak 或其安全变体
- release 区间禁止启用 high_density geometry mode

### 4.3 Semantic Link

Semantic Link 是视听联动的核心层。

建议统一使用以下语义字段：

- energy
- density
- tension
- brightness
- motion
- intent

这些字段应被视为共享 show state 的公共语义面，而不是某一 Runtime 的私有变量。

### 4.4 Signal Link

Signal Link 只应用于视觉细化，而不应用于全局结构调度。

典型用途：

- transient 驱动粒子 burst
- low_band 驱动局部 bloom 或 depth pulse
- onset 驱动 micro flash

约束：

- 不允许 Signal Link 直接切换 scene
- 不允许 Signal Link 改写 section 或 transition state
- 在 degrade 模式下可整体关闭

### 4.5 Safety Link

Safety Link 用于在 patch、降级、回退或异常时统一收紧音画行为。

例如：

- patch staging 期间冻结高风险 visual automation
- GPU 头部空间不足时切换 low_complexity visual mode
- rollback 期间音频与视觉同时回到最近稳定 checkpoint

## 5. 联动模式

### 5.1 locked 模式

适用场景：

- 强 cue 点
- drop 切入
- 已编排死的演出关键断点

要求：

- 视觉进入点必须量化到指定边界
- 允许的相位漂移极小
- patch 默认不可破坏该绑定

### 5.2 guided 模式

适用场景：

- 大段落中的连续视觉演化
- 共享结构但不要求逐拍绑定的表演段落

要求：

- 结构边界一致
- 局部 uniform / camera 可独立细化

### 5.3 reactive 模式

适用场景：

- 细节特效
- 快速微响应

要求：

- 只能作用于局部参数
- 必须受 smoothing、clamp 和 rate limit 约束

### 5.4 decoupled 模式

适用场景：

- 长段环境画面
- fallback 安全场景
- 资源紧张时的稳态降级

要求：

- 仅保持 clock 与 section 可观测一致
- 不消费高频 signal link

## 6. 联动规则优先级

### 6.1 优先级顺序

当多条联动规则冲突时，建议按以下顺序裁决：

1. Safety Link
2. locked Structural Link
3. Clock Link
4. guided Semantic Link
5. reactive Signal Link

### 6.2 冲突示例

如果 reactive link 试图在当前 beat 突然增大某 visual pass，但 Safety Link 正在执行低复杂度降级，则必须服从 Safety Link。

## 7. 容差与同步预算

### 7.1 建议区分三类容差

| 容差类型 | 含义 | 典型阈值 |
| --- | --- | --- |
| Structural Drift | 结构边界偏差 | 小于半拍 |
| Visual Activation Drift | scene 生效偏差 | 小于 80ms 到 150ms |
| Reactive Jitter | 高频参数抖动 | 取决于 smoothing window |

### 7.2 超出容差的处理

- 先记录 trace
- 再决定 warning、degrade 或 rollback
- 不允许 Runtime 静默吞掉关键偏差

## 8. 联动状态对象

### 8.1 Audiovisual Link State

建议在 Show State 外，再维护一个联动专用快照：

```json
{
  "show_id": "show-2026-04-01-a",
  "revision": 12,
  "link_state": {
    "mode": "guided",
    "active_scene": "corridor_peak",
    "structural_anchor": {
      "section": "drop_a",
      "phrase": 33,
      "quantize": "phrase"
    },
    "semantic_bindings": [
      "link-drop-energy-to-corridor-motion"
    ],
    "signal_bindings_enabled": true,
    "safety_mode": "normal"
  }
}
```

### 8.2 Link State 更新触发

- section 切换
- scene 切换
- patch 生效
- degrade / rollback
- operator 强制改模式

## 9. 与 Patch 的关系

### 9.1 Patch 可改什么

live patch 可以修改：

- binding 的参数范围
- 某个 scene 的 semantic mapping
- 某个 signal link 的启停

### 9.2 Patch 不可改什么

live patch 默认不应修改：

- locked 模式下的关键 cue 绑定
- 已锁定 section 的 structural anchor
- 当前 rollback 安全链路中的 fallback binding

### 9.3 Patch 生效后的联动验证

Patch 一旦影响 link binding，必须额外检查：

- 结构边界是否仍一致
- 视觉生效是否超出 drift_tolerance
- 是否引入新的资源峰值

## 10. Trace / Evaluation 采集要求

### 10.1 最少采集项

- link_id
- source sample / semantic state
- target scene / parameter
- expected activation time
- actual activation time
- drift
- mode
- safety override flag

### 10.2 评价维度

建议至少纳入：

- structural_alignment_score
- semantic_alignment_score
- reactive_stability_score
- safety_compliance_score

### 10.3 典型问题编码

- visual_transition_early
- semantic_mismatch_drop
- reactive_jitter_high
- safety_override_missing

## 11. 与 04 文档的分工

两份文档的分工应明确：

- 04 定义 Runtime 之间交换什么对象和事件
- 12 定义这些对象和事件如何形成稳定、可评价的视听联动关系

换句话说：

```text
04 解决“怎么传”。
12 解决“为什么这样联动，以及联动到什么程度”。
```

## 12. 结论

视听联动协议的核心不是增加更多效果映射，而是建立一套可控的跨模态耦合规则。

关键判断如下：

1. 联动必须分层，不能把结构语义和高频信号混成同一种控制。
2. 每条联动都应显式声明模式、映射、优先级、容差和安全策略。
3. live patch、degrade 与 rollback 必须把联动关系视为一等公民，而不是附带效果。

只有这样，系统才能实现真正稳定、可解释、可评估的视听一体化执行。
