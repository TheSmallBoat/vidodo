# Live Patch 授权与回退策略

日期：2026-04-01
状态：Draft
前置阅读：03-外部规划与介入机制顶层设计书.md，04-音频与视觉运行时事件协议与数据结构.md，07-追踪回放与评估数据模型.md

## 1. 文档目的

本文定义 live 模式下 patch 的授权边界、准入规则、回退路径与降级策略。

目标不是让 patch 更自由，而是让 patch 可控、可审计、可回退。

## 2. 顶层原则

### 2.1 Patch 是受限变更，不是临时重写系统

live patch 的基本定位是：

- 修改局部未来时间线
- 修正资源、素材或语义问题
- 在允许窗口内切换场景或资产

而不是：

- 重写整个 set structure
- 改写当前已执行事实
- 越过验证器直接改 Runtime 内部状态

### 2.2 先授权，后编译，再生效

任何 patch 都必须经过：

```text
Propose
  -> Authorize
  -> Validate
  -> Compile
  -> Stage
  -> Activate
  -> Monitor
  -> Commit or Rollback
```

### 2.3 默认降级优先于停机

当 patch 失败、超时或效果不确定时，系统默认优先：

- 延续当前 section
- 切换到已验证 fallback scene
- 回退到上一 revision

而不是直接中断演出。

## 3. Patch 分类模型

### 3.1 推荐分为四类

| Patch 类别 | 典型内容 | 风险级别 | 默认窗口 |
| --- | --- | --- | --- |
| Param Patch | gain、filter、uniform、camera 参数调整 | 低 | beat / bar |
| Local Content Patch | 替换局部 asset、scene、cue | 中 | phrase |
| Structural Patch | 调整后续 section、transition plan | 高 | section |
| Emergency Patch | 快速降级、硬切、停用重载模块 | 特殊 | emergency slot |

### 3.2 各类 patch 的最小授权面

- Param Patch：允许自动授权，但必须限制参数范围与持续时间
- Local Content Patch：要求已热身资产和已定义 fallback
- Structural Patch：要求人类操作员确认或预授信策略
- Emergency Patch：允许越级，但必须强审计并优先回到安全态

## 4. 参与角色与权限

### 4.1 角色模型

| 角色 | 能力边界 |
| --- | --- |
| Human Operator | 可批准高风险 patch、触发 emergency rollback |
| External Agent | 可提交 proposal，不直接越权生效 |
| Auto Recovery Policy | 仅能触发预定义降级和回退动作 |
| Patch Manager | 执行授权、编译、准入、监控与回退 |

### 4.2 授权决策不应只看发起者身份

授权至少应考虑以下五元组：

```text
(actor, patch_class, scope, target_revision, runtime_budget)
```

### 4.3 授权对象

一个授权令牌建议绑定：

- actor_id
- role
- allowed_patch_classes
- allowed_sections
- allowed_time_windows
- max_resource_delta
- expiry
- requires_human_confirmation

## 5. Patch Window 设计

### 5.1 标准窗口类型

- next_beat
- next_bar
- next_phrase_boundary
- next_section_boundary
- emergency_slot

### 5.2 窗口与 patch 类型的对应关系

| Patch 类别 | 推荐窗口 | 原因 |
| --- | --- | --- |
| Param Patch | next_beat / next_bar | 影响局部且可快速平滑 |
| Local Content Patch | next_phrase_boundary | 避免中句截断与场景跳裂 |
| Structural Patch | next_section_boundary | 保证结构稳定 |
| Emergency Patch | emergency_slot | 优先安全而非平滑 |

### 5.3 锁定区间

以下区间默认不可 patch：

- 已进入 lookahead 且无撤销令牌的事件
- 当前已锁定 section
- 明确标记为 published_frozen 的作品区间

## 6. Patch Proposal 数据结构

```json
{
  "type": "live_patch_proposal",
  "patch_id": "patch-018",
  "submitted_by": "external_agent",
  "patch_class": "local_content",
  "base_revision": 12,
  "scope": {
    "from_bar": 129,
    "to_bar": 161,
    "window": "next_phrase_boundary"
  },
  "intent": {
    "goal": "reduce_density_keep_energy",
    "reason": "vocal_stem_conflict"
  },
  "changes": [
    {
      "op": "replace_asset",
      "target": "deck_b",
      "from": "stem/vocal_A",
      "to": "stem/texture_C"
    }
  ],
  "fallback_revision": 12
}
```

## 7. 授权与准入检查

### 7.1 最小检查序列

任何 live patch 生效前至少经过：

1. Schema 校验
2. Actor 权限检查
3. Patch class 与窗口匹配检查
4. 锁定区间检查
5. 资源预算检查
6. 资产可用性与预热检查
7. 编译与时间线冲突检查
8. Fallback 可用性检查

### 7.2 拒绝原因必须结构化

例如：

- patch_window_missed
- locked_section
- asset_not_warmed
- gpu_budget_exceeded
- missing_fallback_revision
- unauthorized_patch_class

## 8. Patch 生命周期状态机

```text
Proposed
  -> Authorized
  -> Validated
  -> Staged
  -> Activated
  -> Monitored
  -> Committed
  -> Archived
```

失败分支：

```text
Proposed -> Rejected
Staged -> Expired
Activated -> RolledBack
Activated -> Degraded
```

### 8.1 各状态含义

| 状态 | 含义 |
| --- | --- |
| Proposed | 已提交，未授权 |
| Authorized | 角色与窗口授权已通过 |
| Validated | 编译与约束检查已通过 |
| Staged | 已排入未来窗口，尚未生效 |
| Activated | 已切入运行链路 |
| Monitored | 正在观测效果与资源影响 |
| Committed | 已确认保留 |
| RolledBack | 已回退至旧版本 |
| Degraded | 已降级到安全态 |

## 9. 回退策略

### 9.1 三种主要回退方式

| 回退方式 | 触发条件 | 行为 |
| --- | --- | --- |
| Deferred Rollback | patch 已生效但问题不致命 | 在下一个 phrase 或 section 边界回退 |
| Immediate Rollback | 资源或稳定性风险过高 | 立即切换回 fallback revision 或 fallback scene |
| Partial Rollback | 仅部分改动异常 | 只撤销问题动作，保留安全动作 |

### 9.2 回退目标优先级

建议按以下顺序选择：

1. patch proposal 中显式声明的 fallback_revision
2. 当前 section 的最近稳定 checkpoint
3. 当前 show 的上一已验证 revision
4. 预定义 emergency safe scene / safe mix

### 9.3 回退触发条件

- xruns 超阈值
- GPU 峰值超预算
- 目标资产加载失败
- scene 编译失败
- 关键同步偏差持续超阈值
- 人类操作员手动撤销

## 10. 降级策略

### 10.1 降级与回退的区别

- 回退：回到旧 revision 或旧场景
- 降级：进入受限但可持续执行的安全模式

### 10.2 典型降级动作

- 禁用高开销视觉 pass
- 关闭非必要音频层
- 固定 camera 路径与 uniform 自动化
- 停止新 patch 接入，维持当前稳定段落

### 10.3 自动恢复条件

只有在下列条件满足时，才允许从降级态回到正常态：

- 资源指标回到阈值内
- fallback 场景稳定一段最小时长
- 有可用 checkpoint 或已验证 revision

## 11. 审计与留痕

### 11.1 Patch 审计最小内容

- patch_id
- actor
- patch_class
- base_revision
- effective_revision
- 授权决定
- 拒绝或回退原因
- 关联 checkpoint
- 资源影响摘要

### 11.2 为什么审计是核心能力

如果 live patch 没有审计链条，后续就无法：

- 判断是规则错误还是操作错误
- 验证外部 Agent 是否稳定可靠
- 对比不同 patch 策略的效果

## 12. 与 CLI / MCP 的接口要求

CLI / MCP 至少应支持：

- 预检 patch proposal
- 查询当前 patch window
- 查询锁定 section
- 提交 patch proposal
- 批准或拒绝高风险 patch
- 查看 patch 审计与回退记录
- 手动触发 rollback 或 degrade

## 13. 结论

live patch 的正确模型不是“谁能改就先改”，而是“谁在什么窗口里能改什么，并且失败后如何安全退回”。

关键判断如下：

1. patch 必须按风险分类，并绑定窗口、授权与 fallback。
2. 授权、编译、准入、生效、监控与回退必须是显式状态机。
3. 默认策略应是降级与回退优先，而不是把 live 系统暴露给不受控变更。

这使系统既保留现场可修补性，又不破坏长时稳定执行的基础。
