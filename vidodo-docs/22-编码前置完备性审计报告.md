### 3.3 各缺口详细说明

#### GAP-1：错误码与 Diagnostics 字典

21 号文档第 10 节列出了必测错误类型，但没有统一的错误码编码规范。
多个模块（Validator、Patch Manager、Capability Layer）都需要输出结构化诊断信息。
如果不在编码前统一，后续会出现各模块自行定义、CLI 与 MCP 输出不一致的问题。

#### GAP-2：JSON Schema 完整清单

当前 schemas/ 目录只有 MCP Tool Registry 一个 schema 文件。
但 05 号文档定义了 SetPlan、AudioDsl、VisualDsl、ConstraintSet、各层 IR 等大量对象。
14 号文档 Workstream A 明确要求"先把工件面固定住"。
因此正式的 JSON Schema 文件是 Phase 0 第一步的前置条件。

#### GAP-3：工作任务卡与开发里程碑

14 号文档给出了 Workstream A-F 的高层划分和推荐顺序，但没有拆到可分配的任务卡粒度。
缺乏明确的里程碑节点、交付物验收时间点和依赖关系图。

#### GAP-4：开发工作流与分支策略

22 份设计文档，但没有定义代码仓库结构、分支策略、CI 流程、PR 规范等工程协作规范。

## 4. 优势确认

以下方面做得特别好，应在后续开发中维持：

1. **边界清晰**：系统不做什么，始终与系统做什么同等重要地定义。
2. **工件优先**：一等公民是结构化工件，而不是过程或界面。
3. **可追踪性**：从 revision、trace、patch audit 到 evaluation 形成完整证据链。
4. **选型诚实**：13 号文档对技术栈的利弊分析极为平衡，没有预设结论。
5. **端到端验证**：11 号文档串联所有文档，暴露了对象模型之间可能存在的间隙。
6. **测试前置**：21 号文档在架构设计阶段就定义了测试分层策略。

## 5. 风险提醒

| 风险 | 可能后果 | 建议控制措施 |
| --- | --- | --- |
| Schema 未固化就开始编码 | 各模块对接时返工 | 在 Workstream A 完成前不进入 C/D |
| 音频后端集成低估 | 闭环演示延迟 | 先用最小 mock backend 跑通流程 |
| Patch 状态机比预期复杂 | 调试成本爆炸 | 先只做 local_content patch |
| 视觉 Runtime 先做太重 | 拖慢整体进度 | 先做纯色 + 单 uniform 场景 |
| 文档与代码漂移 | 后续无法回溯设计意图 | 建立文档随代码更新机制 |
