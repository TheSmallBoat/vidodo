# 从素材接入到 Live Patch 回退端到端示例

日期：2026-04-01
状态：Draft
前置阅读：05-DSL与IR字段定义草案.md，06-资产接入与分析缓存设计.md，07-追踪回放与评估数据模型.md，08-实时补丁授权与回退策略.md，10-CLI命令清单与MCP工具Schema.md

## 1. 文档目的

本文把 05 到 08 的对象模型串成一条完整链路，展示同一个 show 如何经历：

1. 素材接入
2. 规划提交
3. 编译与运行
4. live patch 预检与生效
5. patch 引发风险后自动或人工回退
6. trace 与 evaluation 收口

这不是演示某个具体实现细节，而是验证各文档之间是否能真正闭环。

## 2. 场景设定

### 2.1 Show 背景

假设存在一个 live show：

- show_id：show-2026-04-01-a
- mode：live
- 目标时长：15 分钟
- 风格：industrial / percussive / narrow_palette

### 2.2 待导入素材

音频素材：

- kick loop
- bass stem
- vocal stem
- texture loop

视觉素材：

- corridor 主 shader
- fog texture
- noise texture
- corridor buffer graph

## 7. 第五步：发现问题并提交 live patch

### 7.1 触发原因

运行监控与人耳监听同时发现：

- vocal stem 与 bass stem 在 build 尾段冲突
- 即将进入的 corridor_peak 视觉 scene GPU 峰值偏高

此时外部 Agent 或人类操作员提出一个 bounded patch：

- 音频侧：把 deck_b 从 vocal stem 换成 texture loop
- 视觉侧：降低 corridor_peak 的 motion_gain 并下调 noise_scale

### 7.2 Patch Proposal

```json
{
  "type": "live_patch_proposal",
  "patch_id": "patch-018",
  "submitted_by": "external_agent",
  "patch_class": "local_content",
  "base_revision": 12,
  "scope": {
    "from_bar": 129,
    "to_bar": 161,
    "window": "next_phrase_boundary"
  },
  "intent": {
    "goal": "reduce_density_keep_energy",
    "reason": "vocal_stem_conflict_and_gpu_headroom_low"
  },
  "changes": [
    {
      "op": "replace_asset",
      "target": "deck_b",
      "from": "asset-stem-vocal-a",
      "to": "asset-loop-texture-c"
    },
    {
      "op": "update_visual_scene",
      "target": "corridor_peak",
      "uniforms": {
        "u_motion_gain": 0.72,
        "u_noise_scale": 0.23
      }
    }
  ],
  "fallback_revision": 12
}
```

### 7.3 预检命令

```text
avctl patch check --file patches/patch-018.json --against-run run-03
```

### 7.4 预检结果

```json
{
  "patch_id": "patch-018",
  "allowed": true,
  "requires_human_confirmation": false,
  "window": "next_phrase_boundary",
  "resource_delta": {
    "cpu": 0.01,
    "gpu": -0.06
  },
  "diagnostics": []
}
```

解释：

- patch class 为 local_content，窗口匹配
- texture 资产已热身
- visual scene 只是参数收缩，不突破预算
- fallback_revision 已存在

### 7.5 提交命令

```text
avctl patch submit --file patches/patch-018.json
```

### 7.6 Patch 生命周期推进

```text
Proposed
  -> Authorized
  -> Validated
  -> Staged
  -> Activated
```

在下一 phrase boundary，revision 13 生效。

## 8. 第六步：patch 生效后出现新问题

### 8.1 新问题

尽管 GPU 峰值下降了，但视觉场景切换仍比音频 drop 提前半拍，导致 evaluation 维度中的 sync 出现偏差。

系统在 trace 中记录到：

```json
{
  "record_type": "event_record",
  "event_id": "evt-000621",
  "revision": 13,
  "kind": "visual.scene.update",
  "phase": "executed",
  "musical_time": {
    "bar": 129,
    "beat": 513.0,
    "phrase": 33,
    "section": "drop_a"
  },
  "ack": {
    "status": "ok",
    "runtime": "visual_runtime",
    "latency_ms": 3.8
  }
}
```

同时 evaluation 快速检查给出问题项：

```json
{
  "issue_id": "issue-01",
  "code": "visual_transition_early",
  "severity": "warning",
  "span": {
    "from_bar": 129,
    "to_bar": 130
  },
  "suggested_patch_scope": "next_phrase"
}
```

### 8.2 回退决策

由于问题不致命，但会破坏 drop 的视听一致性，因此 Patch Manager 选择：

- rollback mode：deferred
- target：fallback_revision 12
- timing：下一个 phrase boundary

### 8.3 回退命令

```text
avctl patch rollback --patch-id patch-018 --mode deferred
```

### 8.4 回退记录

```json
{
  "record_type": "patch_decision",
  "patch_id": "patch-018",
  "base_revision": 12,
  "candidate_revision": 13,
  "decision": "rolled_back",
  "window": "next_phrase_boundary",
  "fallback_revision": 12,
  "reasons": [
    {
      "code": "visual_transition_early",
      "severity": "warning"
    }
  ]
}
```

## 9. 第七步：系统回到稳定态

### 9.1 回退后状态

- active revision：12
- patch-018 state：rolled_back
- current section：drop_a
- fallback scene：corridor_peak revision 12 variant

### 9.2 为什么不立即停机

因为 08 文档的总原则是：

- 优先回退到安全 revision
- 保持 show 持续执行
- 把问题留给后续 evaluation 与下一轮 patch

## 11. 这条链路验证了什么

### 11.1 验证 05 文档

Planning DSL、IR、timeline entry、patch proposal、runtime event 的分层是可串联的。

### 11.2 验证 06 文档

live patch 能否成立，前提是目标资产已经完成接入、分析和预热。

### 11.3 验证 07 文档

没有 trace、checkpoint 与 evaluation，就无法解释为什么 patch 要回退。

### 11.4 验证 08 文档

patch 的核心不是“能改”，而是：

- 在哪一类窗口里改
- 改了以后如何监控
- 出问题后回到哪个稳定点

## 12. 同一流程的 MCP 视角

外部 Agent 若不通过 CLI，而通过 MCP，同一流程可以表示为：

1. `asset.ingest`
2. `plan.submit`
3. `compile.run`
4. `run.status`
5. `patch.check`
6. `patch.submit`
7. `patch.rollback`
8. `trace.query`
9. `eval.run`

这说明 CLI 与 MCP 的确可以共享同一条系统边界。

## 13. 结论

这条端到端示例说明 05 到 08 并不是分散文档，而是一条连贯的运行链：

1. 06 负责把素材变成可引用、可分析、可预热的系统资产。
2. 05 负责把规划与 patch 编译成分层对象和可执行时间线。
3. 08 负责限定 live patch 的授权、生效窗口与回退逻辑。
4. 07 负责把整个过程沉淀为可查询、可比较、可评价的证据链。

只有四者同时成立，这个“可外部规划、可 live patch、可安全回退”的系统才真正闭环。
