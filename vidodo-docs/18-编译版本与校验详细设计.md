# 编译版本与校验详细设计

日期：2026-04-01
状态：Draft
前置阅读：05-DSL与IR字段定义草案.md，08-实时补丁授权与回退策略.md，15-功能模块设计方案总览.md

## 1. 文档目的

本文细化 Compiler、Revision Manager 与 Validator 三个模块的详细设计。

目标是明确如何从规划工件稳定产出 revision、IR、timeline，以及在 patch 与 rollback 场景中保持版本边界清晰。

## 2. 三模块关系

```text
Plan / Patch Input
  -> Validator
  -> Compiler
  -> Revision Manager
```

其中：

- Validator 负责“是否允许”
- Compiler 负责“如何生成”
- Revision Manager 负责“生成结果如何进入系统状态”

## 3. Validator 设计

### 3.1 校验层次

- Schema validation
- Semantic validation
- Resource validation
- Lock / policy validation
- Patch window validation

### 3.2 输入对象

- Plan bundle
- PatchProposal
- current revision state
- current show state
- asset registry snapshot

### 3.3 输出对象

```json
{
  "validation_id": "val-018",
  "status": "warning",
  "diagnostics": [
    {
      "code": "gpu_headroom_low",
      "severity": "warning"
    }
  ]
}
```

### 3.4 典型校验规则

- required field 存在
- asset 引用必须可解析
- tempo map 不能非法修改已锁区间
- patch 不得越过 locked section
- 资源变化不得突破预算阈值

### 3.5 Validator 状态

Validator 原则上无重业务状态，只保留：

- policy profile
- schema registry
- threshold config

## 4. Compiler 设计

### 4.1 职责

- 读取规划层对象
- 生成 Normalized IR
- 生成 Executable IR
- 输出 Unified Timeline
- 生成 compile diagnostics 与 artifact refs

### 4.2 编译阶段

```text
Read Planning DSL
  -> Normalize
  -> Resolve Asset References
  -> Build Structure IR
  -> Build Performance / Visual IR
  -> Build Timeline
  -> Emit Artifacts
```

### 4.3 编译输入

- SetPlan
- AudioDsl
- VisualDsl
- ConstraintSet
- Asset Registry Snapshot
- Base Revision

### 4.4 编译输出

- Asset IR
- Structure IR
- Performance IR
- Visual IR
- Unified Timeline
- CompileRecord

### 4.5 CompileRecord

```json
{
  "compiler_run_id": "compile-032",
  "base_revision": 11,
  "candidate_revision": 12,
  "status": "validated",
  "artifacts": {
    "structure_ir": "compile/structure.json",
    "timeline": "compile/timeline.json"
  }
}
```

### 4.6 编译失败原则

- 不发布 partial revision
- diagnostics 必须绑定 artifact or object target
- 失败 compile run 也要保留记录，便于回溯

## 5. Revision Manager 设计

### 5.1 职责

- 分配 revision id
- 维护 candidate / validated / published / running / archived
- 记录 base_revision 与 derived revision
- 负责 effective revision pointer

### 5.2 Revision 状态机

```text
Candidate
  -> Validated
  -> Published
  -> Running
  -> Archived
```

回退相关：

```text
Running -> Superseded
Running -> RolledBack
```

### 5.3 RevisionRecord

```json
{
  "revision": 12,
  "show_id": "show-2026-04-01-a",
  "base_revision": 11,
  "status": "published",
  "artifact_refs": ["compile/timeline.json"],
  "created_by": "compile-032"
}
```

### 5.4 Patch 与 Revision 的关系

- patch 不一定产生全量新 plan，但会产生新的 effective revision
- rollback 会恢复到旧 revision 或 checkpoint
- 每个 patch 都必须显式声明 base_revision 与 fallback_revision

## 6. 三模块协作流程

### 6.1 全量编译流程

```text
Capability Layer
  -> Validator(plan bundle)
  -> Compiler
  -> Revision Manager.publish(candidate)
```

### 6.2 Patch 编译流程

```text
PatchProposal
  -> Validator(patch)
  -> Compiler(partial rebuild)
  -> Revision Manager.activate(candidate patch revision)
```

## 7. Diff 与可追溯性

### 7.1 RevisionDiff

Revision Manager 应支持结构化 diff：

- timeline entries added / removed
- changed sections
- changed scene bindings
- resource impact delta

### 7.2 Provenance

所有产物需可追溯：

- submission_id
- compiler_run_id
- base_revision
- patch_id

## 8. MVP 范围

MVP 阶段建议收敛为：

- 一条全量编译链路
- 一条 patch partial rebuild 链路
- 基础 revision 生命周期
- 结构化 diagnostics
- 最小 revision diff

## 9. 验收标准

该模块链路达标，当且仅当：

1. 同一输入重复编译结果稳定。
2. compile fail 不污染已发布 revision。
3. patch 与 rollback 都能回溯到 base_revision。
4. validator 可输出机器可消费的 diagnostics。

## 10. 结论

编译、版本和校验三者共同决定系统是否“可信”。没有明确 revision 语义，patch、trace、replay 都会失去工程基础。
