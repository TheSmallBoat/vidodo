# Asset Ingestion 与分析缓存设计

日期：2026-04-01
状态：Draft
前置阅读：02-视听系统产品定位与规划方案书.md，05-DSL与IR字段定义草案.md

## 1. 文档目的

本文定义素材接入、规范化处理、分析任务调度、缓存键设计与失效策略。

目标是让以下流程成立：

```text
外部导入素材
  -> 系统生成稳定指纹与规范化副本
  -> 触发分析任务并缓存结果
  -> 发布 Asset IR
  -> 供规划、编译、调度与评价重复使用
```

## 2. 设计原则

### 2.1 素材接入不是简单文件拷贝

接入阶段至少要完成：

- 内容指纹生成
- 技术规格标准化
- 元数据整理
- 分析任务编排
- 结果发布与版本登记

### 2.2 分析缓存必须是可复用的系统资产

节拍、段落、key、能量包络、shader 编译预检等结果，不能每次编译都重新算。

分析缓存应满足：

- 可寻址
- 可失效
- 可追溯到分析器版本
- 可被多个 show 共享

### 2.3 原始素材、规范化素材与分析结果必须分层存储

最少分三层：

- Raw：原始导入件
- Normalized：统一规格后的副本
- Analysis Cache：分析产物与摘要索引

## 3. 素材分类模型

### 3.1 一等素材类型

| 类型 | 示例 | 主要产出 |
| --- | --- | --- |
| Audio Asset | stem、loop、one-shot、MIDI | 音频规格、节拍、段落、特征摘要 |
| Visual Asset | shader、texture、geometry preset、buffer graph | 依赖图、编译预检、标签、资源估计 |
| Rule Asset | pattern preset、mapping 模板、constraint preset | 规范化规则对象 |
| Derived Asset | 转码音频、代理纹理、预烘焙分析片段 | 派生索引与复用指针 |

### 3.2 Audio Asset 子类型

- audio_stem
- audio_loop
- audio_oneshot
- midi_clip
- control_curve

### 3.3 Visual Asset 子类型

- glsl_program
- texture_2d
- cube_map
- geometry_preset
- buffer_graph
- palette_pack

## 4. 接入流水线

### 4.1 总体阶段

```text
Discover
  -> Fingerprint
  -> Normalize
  -> Analyze
  -> Publish
  -> Warm Cache
```

### 4.2 Discover

负责识别导入源与候选文件。

输入来源可以是：

- 本地目录
- 单个文件
- 压缩包解包结果
- 远程同步后落地目录

输出对象：IngestionCandidate。

```json
{
  "candidate_id": "cand-018",
  "path": "imports/session_a/kick_01.wav",
  "declared_kind": "audio_loop",
  "size_bytes": 1842304,
  "modified_at": 1711938820
}
```

### 4.3 Fingerprint

目标是产生内容稳定标识，而不是只依赖文件路径。

建议字段：

- content_hash
- metadata_hash
- source_uri
- source_mtime
- probe_summary

对音频素材，probe_summary 可包含：

- codec
- sample_rate
- channels
- duration_ms
- peak_dbfs

对视觉素材，probe_summary 可包含：

- language
- include_graph_hash
- texture_dim
- est_gpu_bytes

### 4.4 Normalize

标准化阶段应尽量产出可重复使用的运行副本。

音频规范化建议：

- 统一 sample rate，例如 48000
- 统一声道布局
- 预计算波形摘要
- 根据类型生成可选代理版本

视觉规范化建议：

- 解析 include 与依赖图
- 生成统一换行与编码格式
- 对纹理生成代理尺寸与 mip 计划
- 对 shader 做静态语法与依赖预检

### 4.5 Analyze

分析阶段由多个可单独缓存的任务组成。

音频典型分析任务：

- beat_track
- downbeat_detect
- key_estimate
- section_segmentation
- onset_summary
- energy_envelope
- spectral_profile

视觉典型分析任务：

- shader_compile_probe
- dependency_graph_build
- texture_stat_summary
- palette_extract
- gpu_cost_estimate

### 4.6 Publish

Publish 阶段负责把结果组装成 AssetRecord 与 Asset IR。

必须保证：

- 相同内容哈希下的重复导入不会重复发布新资产
- 新分析结果可升级同一资产的 analysis refs
- 旧 revision 仍可追溯到历史分析结果

## 5. 核心数据模型

### 5.1 AssetRecord


## 6. 缓存键设计

### 6.1 推荐缓存键组成

分析缓存键应至少由以下部分拼成：

```text
cache_key = hash(
  asset_content_hash,
  normalized_spec,
  analyzer_name,
  analyzer_version,
  analyzer_params,
  dependency_fingerprint
)
```

### 6.2 为什么不能只用 asset_id

只用 asset_id 会导致：

- 分析器升级后旧结果被误复用
- 参数变化后无法区分结果
- 规范化规格变化后缓存污染

### 6.3 依赖指纹的必要性

以下情况都应进入 dependency_fingerprint：

- shader include 文件变更
- 共享 palette 变更
- 外部模型版本变更
- 参考 tempo map 变更

## 7. 缓存失效策略

### 7.1 强失效触发条件

以下条件出现时必须强制失效：

- 原始内容哈希变化
- 规范化规则版本变化
- 分析器主版本变化
- 依赖图变化

### 7.2 弱失效触发条件

以下条件可以标记为 stale，但不一定立即删除：

- 仅标签或注释变化
- 次要统计字段补充
- 非关键元数据来源变化

### 7.3 回收策略

建议采用两层回收：

- 引用计数或最近使用时间控制热缓存
- 版本保留窗口控制历史缓存

对于 live 常用素材，建议显式预热，不依赖被动命中。

## 8. 与编译链路的接口

### 8.1 Asset Selection 只消费已发布资产

规划与编译阶段不应直接读取未发布原始文件，而应通过已发布 AssetRecord / Asset IR 获取素材。

### 8.2 编译器需要的最小读取面

编译器通常只需要：

- AssetRecord
- 关键 analysis 摘要
- 能力标记
- 资源估计
- 兼容性标签

不应在编译期重复扫描大文件或重跑完整分析。

### 8.3 Runtime 需要的读取面

Runtime 只需要：

- normalized_locator 或 runtime-ready locator
- 必需摘要
- fallback 资源引用
- 资源预算提示

## 9. Live 模式下的特殊要求

### 9.1 Live Patch 只允许引用已热身资产

为了避免 patch 生效时才发现 I/O 或编译开销过高，live patch 原则上只能引用：

- 已发布资产
- 已完成关键分析的资产
- 已完成预热或可在窗口内装载的资产

### 9.2 临时导入策略

如果现场需要临时导入新素材，建议流程为：

```text
ingest --stage
  -> 只做快速规范化与最小分析
  -> 标记为 patch_pending_asset
  -> 在非关键窗口完成预热
  -> 才允许进入 patch 选择集
```

## 10. 失败与降级策略

### 10.1 单个分析任务失败

处理策略：

- 标记失败原因
- 不阻塞其他分析任务
- 对缺失分析设置 capability 降级标记

例如：

- beat_track 失败，则资产不允许进入 quantized launch
- shader_compile_probe 失败，则资产不允许进入 live scene

### 10.2 规范化失败

规范化失败应阻止发布，但保留失败报告，供人工或外部 Agent 再处理。

### 10.3 发布失败

发布阶段必须具备事务语义：

- 未完成 publish 不应暴露半成品资产
- analysis refs 与 normalized locator 应成组提交

## 11. 建议目录结构

```text
assets/
  raw/
  normalized/
  derived/
analysis/
  cache/
  reports/
registry/
  assets.jsonl
  ingestion-runs.jsonl
  analysis-jobs.jsonl
```

## 12. 结论

Asset Ingestion 与分析缓存不是辅助模块，而是统一 IR 的地基。

关键判断如下：

1. 素材接入必须产出稳定指纹、规范化副本和可复用分析结果。
2. 分析缓存键必须绑定内容、规格、分析器版本和依赖图。
3. 编译与 live patch 都只能消费已发布、可追踪、可预热的资产。

这样才能让规划、编译、执行、评价和 patch 共享同一套素材事实基础。
