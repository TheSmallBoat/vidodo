# Trace / Replay / Evaluation 数据模型

日期：2026-04-01
状态：Draft
前置阅读：03-外部规划与介入机制顶层设计书.md，04-音频与视觉运行时事件协议与数据结构.md，05-DSL与IR字段定义草案.md

## 1. 文档目的

本文定义 trace、replay 与 evaluation 的统一数据模型，使系统满足以下要求：

1. 每次规划、编译、执行、patch 与导出都可追踪。
2. 任意一次 live 或离线运行都可以复盘与比较。
3. 人类与外部 Agent 都能基于结构化证据做下一轮判断。

## 2. 顶层结论

### 2.1 Trace 不是日志堆积，而是证据链

本系统中的 Trace Bundle 应回答以下问题：

- 输入了什么工件
- 编译成了什么版本
- 发布了哪些事件
- 哪些事件被执行、拒绝或回退
- 运行中资源情况如何
- 最终导出了什么结果

### 2.2 Replay 不是视频回放，而是受控重演

Replay 至少有三种模式：

- Structural Replay：重放结构事件与 show state
- Deterministic Replay：在相同资产和版本下重演执行
- Comparative Replay：比较两个 revision 的差异表现

### 2.3 Evaluation 应基于事实、分数和证据引用

评价输出不应只有总分，还应包含：

- 指标值
- 问题项
- 证据位置
- 建议 patch 范围

## 3. Trace Bundle 组成

### 3.1 最小组成

```text
trace_bundle/
  manifest.json
  inputs/
  compile/
  runtime/
  metrics/
  patches/
  exports/
  evaluation/
```

### 3.2 Manifest

```json
{
  "trace_bundle_id": "trace-show-a-rev12-run03",
  "show_id": "show-2026-04-01-a",
  "revision": 12,
  "run_id": "run-03",
  "mode": "live",
  "started_at": 1711941100,
  "completed_at": 1711942008,
  "status": "completed",
  "input_refs": ["inputs/plan_submission.json"],
  "event_log_ref": "runtime/events.jsonl",
  "metrics_ref": "metrics/resource.jsonl",
  "evaluation_ref": "evaluation/report.json"
}
```

## 4. 核心实体定义

### 4.1 PlanInputRecord

记录进入系统的外部输入工件。

```json
{
  "record_type": "plan_input",
  "submission_id": "subm-018",
  "submitted_by": "external_client",
  "submitted_at": 1711941100,
  "artifacts": {
    "set_plan": "inputs/set-plan.json",
    "audio_dsl": "inputs/audio.json",
    "visual_dsl": "inputs/visual.json",
    "constraints": "inputs/constraints.json"
  },
  "content_hashes": {
    "set_plan": "sha256:..."
  }
}
```

### 4.2 CompileRecord

```json
{
  "record_type": "compile_record",
  "compiler_run_id": "compile-032",
  "base_submission_id": "subm-018",
  "input_revision": 11,
  "output_revision": 12,
  "status": "validated",
  "warnings": [
    {
      "code": "gpu_headroom_low",
      "severity": "warning"
    }
  ],
  "artifacts": {
    "asset_ir": "compile/assets.json",
    "structure_ir": "compile/structure.json",
    "timeline_ir": "compile/timeline.json"
  }
}
```

### 4.3 EventRecord

EventRecord 是 trace 的核心行级对象，建议使用 JSONL 保存。

```json
{
  "record_type": "event_record",
  "event_id": "evt-000128",
  "show_id": "show-2026-04-01-a",
  "revision": 12,
  "kind": "audio.layer.update",
  "phase": "executed",
  "source": "scheduler",
  "musical_time": {
    "bar": 65,
    "beat": 257.0,
    "phrase": 17,
    "section": "drop_a"
  },
  "scheduler_time_ms": 93452,
  "wallclock_time_ms": 1711941505123,
  "causation_id": "tle-00128",
  "payload_ref": "runtime/payloads/evt-000128.json",
  "ack": {
    "status": "ok",
    "runtime": "audio_runtime",
    "latency_ms": 4.2
  }
}
```

phase 建议限定为：

- published
- received
- executed
- rejected
- rolled_back

### 4.4 PatchDecisionRecord

```json
{
  "record_type": "patch_decision",
  "patch_id": "patch-018",
  "base_revision": 12,
  "candidate_revision": 13,
  "decision": "accepted",
  "window": "next_phrase_boundary",
  "scope": {
    "from_bar": 129,
    "to_bar": 161
  },
  "authorization_ref": "patches/authorization-018.json",
  "fallback_revision": 12,
  "reasons": []
}
```

### 4.5 ResourceSample

```json
{
  "record_type": "resource_sample",
  "sample_time_ms": 1711941505100,
  "show_id": "show-2026-04-01-a",
  "revision": 12,
  "cpu": 0.43,
  "gpu": 0.58,
  "memory_mb": 914,
  "audio_xruns": 0,
  "video_dropped_frames": 0,
  "active_scene": "corridor_peak"

```

### 4.6 ExportArtifactRecord

用于把导出产物纳入证据链。

建议字段：

- artifact_id
- artifact_type
- locator
- content_hash
- derived_from_run_id
- revision
- duration_sec

## 5. Replay 数据模型

### 5.1 ReplayRequest

```json
{
  "type": "replay_request",
  "trace_bundle_id": "trace-show-a-rev12-run03",
  "mode": "deterministic",
  "from_bar": 97,
  "to_bar": 145,
  "with_audio_analysis": true,
  "with_visual_frames": false
}
```

### 5.2 ReplaySession

```json
{
  "replay_session_id": "replay-004",
  "source_trace_bundle_id": "trace-show-a-rev12-run03",
  "mode": "deterministic",
  "status": "completed",
  "inputs_restored": true,
  "revision_restored": 12,
  "artifacts": {
    "event_diff": "replay/event-diff.json",
    "summary": "replay/summary.json"
  }
}
```

### 5.3 CheckpointSnapshot

为了支持长时 live 复盘，建议保留稀疏 checkpoint。

```json
{
  "checkpoint_id": "ckpt-bar-129",
  "show_id": "show-2026-04-01-a",
  "revision": 12,
  "bar": 129,
  "show_state_ref": "runtime/checkpoints/ckpt-bar-129.json",
  "active_assets": ["asset-loop-kick-a", "scene-corridor-peak"],
  "rollback_ready": true
}
```

建议 checkpoint 边界：

- section 边界必存
- patch 生效前后必存
- 导出阶段关键断点可选存

## 6. Evaluation 数据模型

### 6.1 EvaluationRequest

```json
{
  "type": "evaluation_request",
  "target": {
    "trace_bundle_id": "trace-show-a-rev12-run03",
    "revision": 12
  },
  "dimensions": [
    "sync",
    "transition",
    "visual_semantic_alignment",
    "resource_stability"
  ],
  "compare_with_revision": 11
}
```

### 6.2 EvaluationReport

```json
{
  "type": "evaluation_report",
  "report_id": "eval-021",
  "show_id": "show-2026-04-01-a",
  "revision": 12,
  "scores": {
    "sync_score": 0.94,
    "transition_score": 0.81,
    "visual_semantic_score": 0.78,
    "resource_stability_score": 0.91
  },
  "issues": [
    {
      "issue_id": "issue-01",
      "code": "visual_transition_early",
      "severity": "warning",
      "span": {
        "from_bar": 129,
        "to_bar": 130
      },
      "evidence_refs": [
        "runtime/events.jsonl#evt-000621",
        "runtime/checkpoints/ckpt-bar-129.json"
      ],
      "suggested_patch_scope": "next_phrase"
    }
  ],
  "summary": {
    "verdict": "usable_with_patch",
    "recommended_action": "reduce_visual_scene_lead_in"
  }
}
```

### 6.3 EvaluationFact

为了避免评价层直接变成散文描述，建议将事实拆成独立对象。

```json
{
  "fact_id": "fact-201",
  "dimension": "sync",
  "metric": "section_drop_alignment_ms",
  "value": 243,
  "threshold": 120,
  "status": "above_limit",
  "evidence_refs": ["runtime/events.jsonl#evt-000621"]
}
```

## 7. 差异比较模型

### 7.1 RevisionDiff

```json
{
  "type": "revision_diff",
  "from_revision": 11,
  "to_revision": 12,
  "changes": {
    "timeline_entries_added": 12,
    "timeline_entries_removed": 4,
    "patched_sections": ["drop_a"],
    "visual_scene_changes": 2
  },
  "impact": {
    "sync_score_delta": 0.03,
    "gpu_peak_delta": 0.08
  }
}
```

### 7.2 为什么 diff 必须结构化

如果 diff 只有文本描述，就无法被 CLI、MCP 或外部 Agent 稳定消费。

## 8. 存储与保留建议

### 8.1 存储形式

- Manifest、Summary、Report 使用 JSON
- 高频事件与资源采样使用 JSONL
- 大对象如导出帧索引与波形代理可单独存储并以 ref 引用

### 8.2 保留策略

建议至少保留：

- 已发布 revision 的完整 Trace Bundle
- 所有 patch decision 记录
- 关键 checkpoint snapshot
- EvaluationReport 与 RevisionDiff

### 8.3 可删减部分

以下对象可以按策略过期：

- 高频资源采样原始流
- 调试级别的临时 payload 副本
- 非关键帧级视觉快照

## 9. CLI / MCP 读取面要求

为了让人类和外部 Agent 都能稳定消费 trace，建议提供以下读取能力：

- 获取某次 run 的 manifest
- 查询某个 bar 区间的 EventRecord
- 查询 patch 决策与原因
- 拉取 EvaluationReport
- 比较两个 revision 的 diff
- 触发 ReplayRequest

## 10. 结论

Trace / Replay / Evaluation 的统一数据模型是系统可信性的核心，而不是附加功能。

关键判断如下：

1. Trace Bundle 必须覆盖输入、编译、执行、patch、资源和导出全链路。
2. Replay 必须基于 revision、checkpoint 与事件记录，而不是只依赖视频录像。
3. Evaluation 必须同时输出分数、事实、问题项和证据引用。

这样，系统才能真正支持复盘、比较、回退与下一轮规划。
