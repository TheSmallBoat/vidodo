## 1. 文档目的

本文定义系统的测试与验证顶层设计，用于回答编码前置阶段最关键的问题：

1. 现有文档是否已经足够支持编码启动。
2. 若还不够，最优先补哪些文档或测试资产。
3. 单元测试、场景测试、集成测试、回归测试与验收测试如何分层。

## 2. 当前完成度判断

### 2.1 已经足够支持什么

现有文档已经足够支持以下工作启动：

- 核心模块代码骨架搭建
- schema / artifact / revision 主体设计
- CLI / Capability Layer 实现
- 资产接入、编译、调度、patch、trace 的第一版实现
- MVP 最小闭环原型开发

### 2.2 仍然缺什么

如果目标从“能开始写”提升到“能稳定推进、不靠口头约定返工”，还缺以下前置物：

1. 测试与验证顶层设计
2. 样例工件与测试夹具规范
3. 错误码 / diagnostics 约定表
4. Schema fixture 与 golden output 策略
5. 运行时回放与 patch 回退验证规范

### 2.3 结论

因此，当前文档状态可以概括为：

```text
足够启动编码，
但还不足以支持大规模并行实现和稳定回归，
必须补上测试与验证体系文档。
```

## 3. 测试设计原则

### 3.1 以工件为核心验证对象

测试不应只围绕函数返回值，而应围绕系统的一等对象：

- AssetRecord
- AnalysisCacheEntry
- SetPlan / AudioDsl / VisualDsl
- IR / Timeline
- Show State
- PatchDecision
- Trace Bundle
- EvaluationReport

### 3.2 以确定性为第一优先级

本系统最重要的测试目标之一不是“效果更好”，而是：

- 相同输入能否稳定得到相同输出
- patch / rollback / replay 能否复现
- diagnostics 是否一致可消费

### 3.3 测试分层必须与系统分层一致

不要把所有测试都做成端到端大集成。

应至少分为：

- 单元测试
- 模块级集成测试
- 场景测试
- 回归测试
- 验收测试

## 4. 测试分层模型

### 4.1 单元测试

目标：

- 验证单模块纯逻辑
- 快速定位回归

覆盖对象：

- schema validation rules
- cache key generation
- revision diff logic
- patch window matching
- event translation logic

示例：

- 给定相同 asset fingerprint，cache key 必须稳定
- 给定 locked section，patch validator 必须拒绝越界 patch

### 4.2 模块级集成测试

目标：

- 验证两个或三个相邻模块之间的协作

覆盖对象：

- Capability Layer + Validator
- Ingestion + Registry
- Compiler + Revision Manager
- Scheduler + Audio Runtime Client
- Patch Manager + Trace Writer

示例：

- 提交一个合法 plan bundle，应生成 candidate revision 与 artifact refs
- 提交一个 patch proposal，应生成 patch ticket，并写入 patch decision trace

### 4.3 场景测试

目标：

- 验证一个完整业务场景是否成立

推荐最小场景：

1. 资产接入到编译成功
2. run.start 到 trace 输出
3. local_content patch 生效
4. patch rollback 成功
5. MIDI 绑定到 instrument backend 成功

### 4.4 回归测试

目标：

- 确保已有工件、示例和 golden outputs 不漂移

覆盖对象：

- schema fixtures
- compile output golden files
- trace bundle snapshots
- revision diff snapshots

### 4.5 验收测试

目标：

- 验证 MVP 是否满足阶段性交付标准

通常对应：

- Phase 0 演示脚本
- Phase 1 patch / shared show state 演示脚本

## 6. 样例与夹具策略

### 6.1 必须准备的测试资产

- 最小音频素材包
- 最小视觉素材包
- 最小 fallback 资产包
- 最小 MIDI clip 集
- 最小 SoundFont / instrument preset 示例

### 6.2 Fixture 分类

- schema fixtures
- asset fixtures
- plan bundle fixtures
- patch fixtures
- trace fixtures
- golden compile outputs

### 6.3 为什么 fixture 是前置条件

没有稳定测试夹具，后续会出现：

- 每个人都用不同素材做验证
- 编译结果无法稳定对比
- patch / rollback 无法系统复现

## 7. 测试环境分层

### 7.1 Pure Logic 环境

适合：

- schema
- validator
- compiler
- revision diff

特点：

- 不依赖真实音频设备
- 不依赖真实 GPU

### 7.2 Simulated Runtime 环境

适合：

- Scheduler
- Audio Runtime Client command translation
- Visual Runtime command application
- patch / rollback 状态机

特点：

- 后端可用 mock 或 fake backend
- 验证 event / ack / diagnostics 路径

### 7.3 Real Backend 环境

适合：

- SuperCollider server 接入
- 实际 sample backend
- instrument backend / SoundFont 验证

特点：

- 用于少量关键集成测试
- 不适合作为所有测试默认环境

## 8. 核心测试场景

### 8.1 编译链路场景

输入：

- SetPlan
- AudioDsl
- VisualDsl
- ConstraintSet

断言：

- compile 成功
- candidate revision 生成
- timeline output 与 golden 匹配

### 8.2 Patch 成功场景

输入：

- running revision
- local_content patch

断言：

- patch ticket 生成
- patch 在合法窗口生效
- trace 中存在 patch activated 记录

### 8.3 Patch 回退场景

输入：

- activated patch
- 模拟 visual_transition_early 或 backend timeout

断言：

- rollback 指令发出
- effective revision 恢复
- trace 中存在 rollback 记录

### 8.4 外部乐器场景

输入：

- MIDI clip
- instrument binding
- soundfont sampler backend

断言：

- preset 可解析
- MIDI 事件被正确路由
- backend ack 被 trace 捕获

## 9. Golden Output 策略

### 9.1 适合做 golden 的对象

- normalized plan request
- compile output
- timeline entries
- patch decision
- trace manifest
- evaluation report

### 9.2 不适合直接做 byte-for-byte golden 的对象

- 高频 resource samples
- wallclock 时间戳
- 实时后端原始日志

### 9.3 稳定化策略

- 对时间戳做归一化
- 对随机 seed 固定
- 对非关键字段做 compare ignore

## 10. 错误与诊断验证

### 10.1 必测错误类型

- schema_invalid
- asset_missing
- asset_not_warmed
- locked_section
- patch_window_missed
- backend_timeout
- preset_missing

### 10.2 断言要求

每个错误至少验证：

- code
- severity
- retryable
- target
- diagnostics location

## 11. Phase 0 验证清单

Phase 0 至少应具备以下测试集：

1. Schema fixtures 全通过 / 全拒绝测试。
2. 资产接入与 registry 查询测试。
3. compile.run golden output 测试。
4. run.start 到 trace manifest 生成测试。
5. patch.submit 与 rollback 场景测试。
6. 一个最小 real backend 集成测试。

## 12. 建议补充的前置文档

除了本文，若要进一步支持高效编码，建议补以下文档：

1. 错误码与 diagnostics 字典
2. fixture / sample asset 包规范
3. JSON Schema 文件清单与版本策略
4. SuperCollider / Instrument Adapter 最小接口规范

## 13. 结论

当前文档体系已经足够支持编码启动，但还不足以保证实现过程稳定可控。测试与验证体系不是补充品，而是编码前置的一部分。

关键判断如下：

1. 当前最优先补齐的不是更多架构文档，而是测试与验证顶层设计。
2. 单元、集成、场景、回归、验收测试必须按系统分层设计。
3. fixture、golden output、错误码与 real backend 集成测试，是后续避免返工的关键基础设施。
