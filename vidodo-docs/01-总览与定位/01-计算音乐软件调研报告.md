# 面向可外部规划视听系统的计算音乐软件调研报告

日期：2026-04-01
状态：Draft

## 1. 调研目的

本报告用于支撑一套独立的视听系统设计。

这里的目标对象不是传统 DAW，不是内置 LLM 的自治体，也不是自由度极高的端到端歌曲生成系统。我们关心的是一套更窄但更硬的系统能力：

- 以声音为主轴，但从一开始就考虑声音与视觉的统一编排
- 支持离线生成作品，也支持长时实时执行
- 允许人类或外部 Agent 提供规划、规则、patch 和评价意见
- 倾向确定性、可复现、可验证、可回放的输出
- 尽量复用成熟运行时、引擎和功能模块，而不是重复造轮子

因此，本报告虽然以“计算音乐软件”为切入点，但最终服务的是一个更大的对象：

```text
一套可由外部规划者驱动、由音频与视觉 Runtime 持续执行的视听系统。
```

## 2. 执行摘要

### 2.1 结论先行

当前开源计算音乐生态已经足够成熟，问题不在于“有没有工具”，而在于“缺少面向视听系统的统一编排与执行中枢”。

综合调研，可以得出六个判断：

1. 实时音频引擎层已经成熟，完全没有必要从零实现完整声音运行时。
2. live coding 与 pattern 语言已经证明，音乐时间语义必须上升为系统级概念。
3. 符号音乐与结构分析层已经成熟，适合承担高层规划、评价和约束表达的基础设施角色。
4. 视觉实时系统同样成熟，但通常与音乐系统分离，缺少共享状态机和统一事件协议。
5. 机器学习音乐生成很活跃，但开放式生成不适合作为第一阶段内核，因为可控性、确定性和运行成本都存在明显问题。
6. 现有系统普遍不够“外部规划友好”，缺少结构化交付物、可验证协议、分阶段执行模型与结果追踪机制。

### 2.2 最有参考价值的系统类型

如果从“构建可由人或外部 Agent 驱动的视听系统”这个目标出发，最有参考价值的并不是某一个单点项目，而是四类能力组合：

1. Sonic Pi / TidalCycles / SuperCollider 代表的时间语义与实时调度能力
2. music21 代表的符号音乐结构表示与分析能力
3. GLSL / OpenGL / WebGL / TouchDesigner 类系统代表的视觉实时渲染能力
4. 外部特征分析、素材处理、机器学习模块代表的可插拔扩展能力

### 2.3 对目标系统的直接启示

新系统的核心不应该是“再做一个音乐语言”或“再内置一个 Agent”，而应该是：

- 用统一 IR 和统一时间模型承接音乐与视觉编排
- 用确定性 Runtime 执行长时间的离线或实时流程
- 用清晰的 CLI / 文件接口 / API / MCP 边界接受外部规划
- 把作品交付物而不是推理过程作为一等公民

换句话说，目标系统更接近：

```text
视听状态机 + 统一时间模型 + 可编译交付物 + 外部规划入口。
```

## 3. 相关生态的结构性观察

### 3.1 计算音乐生态已成熟，但分层割裂

现有开源项目大致可分为五类：

| 类别 | 代表项目 | 优势 | 主要局限 |
| --- | --- | --- | --- |
| 实时音频引擎 | SuperCollider, CPAL 生态 | 低延迟、调度稳定、运行时成熟 | 缺少面向统一视听编排的产品边界 |
| live coding / pattern 系统 | Sonic Pi, TidalCycles, FoxDot, Strudel | 时间表达强、反馈快、适合表演 | 更偏人类持续交互，不偏可提交的系统规划 |
| 符号音乐 / 分析工具 | music21 | 结构表达强，适合高层规划与评价 | 不负责稳定实时执行 |
| 视觉实时引擎 | GLSL / OpenGL / WebGL / TouchDesigner 类工作流 | 画面表达强、GPU 能力成熟 | 往往缺少与音乐系统共享的结构语义 |
| 机器学习生成模块 | Magenta, DDSP, 各类外部模型 | 生成纹理和音色能力强 | 不适合作为第一阶段的确定性内核 |

系统设计上的真正空白，是缺少一种面向视听联合控制的中间层：

- 统一资产与分析结果
- 统一结构表示
- 统一时间线与事件协议
- 统一的外部规划与 patch 入口

### 3.2 对目标问题的再定义

因此，这个方向要解决的核心问题不是：

```text
如何让大模型持续不断地产生声音和画面。
```

而是：

```text
如何让人类或外部 Agent 在受约束的素材与规则空间中提交可执行的视听计划，
并由 Runtime 稳定执行很长一段时间。
```

## 4. Sonic Pi 调研要点

### 4.1 为什么 Sonic Pi 依然重要

Sonic Pi 并不是目标系统的直接产品模板，但它证明了一件非常关键的事：

```text
时间必须成为语言级和系统级概念，而不能只靠 sleep、回调或线程拼装。
```

它最值得学习的，不是 Ruby 风格 DSL，而是这三层分离：

- 用户表达层
- 调度与时间层
- 声音执行层

### 4.2 对本项目最有迁移价值的部分

Sonic Pi 的可迁移经验主要有四点：

1. 逻辑时间与物理时间必须显式分离。
2. 编排层和声音执行层必须分离。
3. MIDI、OSC、Link 之类外部协议要纳入统一调度语义。
4. 对用户友好的高层表达，并不意味着底层架构可以简单。

### 4.3 不应直接照搬的部分

Sonic Pi 的问题不在于它不优秀，而在于它的中心假设与本项目不同：

- 它更偏 human-in-the-loop live coding
- 它以持续交互创作过程为中心
- 它没有把“系统可被外部规划者驱动并长期稳定执行”作为主要产品目标

本项目则更强调：

- 规划与执行解耦
- Runtime 持续执行
- 离线与实时共用同一份时间语义与计划

## 5. 其他代表系统的启示

### 5.1 SuperCollider

SuperCollider 的核心价值是 client/server 分层与成熟实时音频执行。

对目标系统的启示：

- 声音执行层应尽量视为独立运行时
- 高层规划与底层渲染不应绑死在同一个解释器里
- 不应在第一阶段重造一套完整 DSP 图系统

### 5.2 TidalCycles

TidalCycles 的核心贡献是 pattern algebra，也就是把时间模式本身提升为可组合对象。

对目标系统的启示：

- 结构化 pattern IR 很重要
- 外部规划者不应直接产出不可验证的自由文本长脚本
- 现场 patch 更适合基于 pattern / phrase / section 边界完成

### 5.3 music21

music21 的价值在于高层乐理结构、语义分析和格式互通。

对目标系统的启示：

- 高层视听规划不能只靠事件流
- 必须有能表达 section、phrase、harmony、meter、tempo map 等对象的结构层
- 评价系统需要引用结构证据，而不仅是波形或频谱统计

### 5.4 FoxDot / Strudel / Extempore

这些系统的价值更多体现在不同的人机交互风格：

- FoxDot 说明显式 Player / Voice 模型很适合组织编排
- Strudel 说明低门槛交互与可视反馈很有价值
- Extempore 说明运行时与语言一体化虽然强大，但复杂度极高

对目标系统来说，应吸收其抽象，不必复制其整体形态。

### 5.5 Magenta / DDSP 与外部生成模块

这些项目最适合扮演外挂模块，而不是第一阶段的主内核。

更合理的定位是：

- 作为素材生成器
- 作为特征分析器
- 作为风格建议器
- 作为音色或视觉参数候选生成器

而不是系统必须内置和持续依赖的能力。

## 6. 从调研中提炼出的统一分层模型

### 6.1 面向视听系统的六层模型

综合音乐与视觉两侧，可提炼出一个更适合本项目的六层模型：

1. 资产层：声音、视觉、规则、分析缓存、版本信息
2. 表示层：Asset IR、Structure IR、Performance IR、Visual IR
3. 变换层：编排、约束、pattern 变换、场景映射、patch 生成
4. 调度层：musical time、scheduler time、wall clock 的映射
5. 执行层：Audio Runtime、Visual Runtime、导出器、桥接器
6. 接口层：CLI、文件接口、API、MCP、监控汇总与回调

### 6.2 真正决定系统质量的机制

高质量视听系统的关键不是某个 DSL 看起来多“聪明”，而是以下机制是否成立：

| 机制 | 为什么关键 |
| --- | --- |
| 统一时间模型 | 决定音频与视觉能否真正联动而非偶然同步 |
| 稳定 IR | 决定是否能支持多前端、多规划者和多执行器 |
| 调度与执行分离 | 决定是否能统一离线与实时 |
| 结构化接口 | 决定系统是否能被人或外部 Agent 稳定调用 |
| 结果可追踪 | 决定是否能回放、解释、比较和复现 |
| 约束和门禁 | 决定 Runtime 是否能在现场稳定运行 |

## 7. 面向外部规划的设计要求

### 7.1 规划友好不等于“内置 Agent”

对外部规划友好的系统，关键不在于自己内置一个 LLM，而在于系统是否提供：

- 结构化输入
- 结构化输出
- 可解释错误
- 阶段化命令
- 可验证结果
- 可边界化 patch 机制

### 7.2 更适合的交付物形态

比起自由文本长脚本，系统更适合接收这些对象：

- Set Plan
- Scene Graph
- Audio DSL
- Visual DSL
- Timeline IR
- Constraint Set
- Evaluation Request
- Timeline Patch

### 7.3 外部规划最适合进入哪些阶段

调研结论与本项目当前定义一致：外部规划最适合进入以下环节：

- 初始规划
- 素材组织与筛选
- 离线评价
- 局部 patch
- 规则变更后的重编译

不适合进入以下环节：

- 音频 callback
- 视觉帧循环
- lookahead 调度内核
- 长时间连续低层参数插值

## 8. 对目标产品的架构建议

### 8.1 不建议的路线

第一阶段不建议做这些事情：

- 自研完整 DSP 图或合成器体系
- 把大模型推理挂在实时播放链路上
- 把视觉联动简化成频谱驱动小效果
- 把系统边界建立在某个特定 Agent provider 上

### 8.2 建议的系统骨架

更合理的产品骨架是：

1. 素材接入与预分析
2. 统一 Asset / Structure / Performance / Visual IR
3. 统一调度器与 show state
4. 独立 Audio Runtime
5. 独立 Visual Runtime
6. CLI / 文件接口 / API / MCP 入口
7. 离线导出、现场执行、trace 与 replay 体系

### 8.3 工程策略

技术实现上应优先复用成熟模块：

- 音频回放、DSP、时间拉伸、移调
- MIDI / OSC / Link / 外部控制协议
- 节拍、段落、key、特征分析
- GLSL 编译、OpenGL/WebGL 渲染、离屏渲染
- 视频编码、音视频合成与导出

自研重点应放在：

- 统一 IR
- 统一时间模型
- 音视频事件协议
- 规划与介入接口
- Trace、replay、diff、validation

## 9. 与一般调研结论的差异收敛

如果只从“计算音乐软件”角度出发，很容易得出一个偏窄的目标：做一个脚本化音乐编排系统。

但结合当前产品定义，真正的方向已经发生了三处变化：

1. 从单音乐系统扩展为视听一体化系统。
2. 从持续交互工具转向“外部规划 + 长时执行”的系统。
3. 从语言中心转向交付物与协议中心。

因此，本调研报告的最终结论不是“应该做一个内置 Agent 的智能音乐产品”，而是：

```text
应该构建一套以统一时间与统一状态为中枢、
可由人类或外部 Agent 提供规划、
并由音频与视觉 Runtime 独立持续执行的视听系统。
```
