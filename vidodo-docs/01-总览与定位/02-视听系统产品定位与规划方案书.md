# 可外部规划的视听系统产品定位与规划方案书

日期：2026-04-01
状态：Draft
前置阅读：01-计算音乐软件调研报告.md

## 1. 文档目的

本方案书从第一性原理出发，定义一套可由外部规划者驱动的视听系统产品。

这里的“外部规划者”可以是：

- 人类创作者
- 人类演出者
- Github Copilot 之类的外部 Agent
- 通过 CLI、文件接口、API 或 MCP 接入的其他自动化系统

因此，本产品既不是传统 DJ 软件，也不是开放式 AI 作曲平台，更不是内置 LLM 的自治体，而是一套：

```text
可由外部提交规划与 patch、由音频与视觉 Runtime 持续执行的确定性视听系统。
```

## 2. 从第一性原理重新定义产品

### 2.1 我们到底在控制什么

从第一性原理出发，系统控制的不是“创意本身”，而是可以被建模、调度和执行的状态转移：

- 音乐素材：track、stem、loop、one-shot、MIDI、规则生成片段
- 视觉素材：shader scene、geometry preset、texture、buffer graph、control curve
- 音乐时间：beat、bar、phrase、section、transition window
- 视觉时间：scene segment、cue、easing phase、frame phase
- 共享特征：tempo、energy、density、tension、brightness、motion、semantic tag
- 系统动作：选择、叠加、切换、混音、过滤、映射、渲染、导出、修补

### 2.2 产品必须保证的性质

这个产品若要成立，必须优先满足以下性质：

1. 相同素材、规则、seed 和版本输入，应得到相同输出。
2. 离线与实时必须共享同一份时间语义和可执行计划。
3. 规划接口不得成为音频 callback、渲染循环和调度器的运行前提。
4. 每次规划、编译、执行和 patch 必须留痕，可追踪、可解释、可回放。
5. 音频与视觉必须共享结构语义，而不是只靠频谱跟随。

### 2.3 产品不是什么

为了避免方向漂移，第一阶段明确不做：

- 自由文本到完整歌曲的开放式创作系统
- 内置 LLM provider 管理与持续在线推理
- 完整 DAW 替代品
- 从零开始的底层音频 DSP 或视觉渲染引擎

## 3. 产品定位

### 3.1 一句话定位

```text
这是一套面向创作者、艺术工程师与现场装置的视听编排、编译与执行系统，
它接受外部规划输入，并稳定驱动音频与视觉 Runtime。
```

### 3.2 产品本质

它同时具有三种属性：

- 编排系统：组织声音与画面的结构、层次与过渡
- 编译系统：把高层意图编译成可执行 DSL、IR 和时间线
- 运行系统：在离线或实时上下文中稳定执行既定计划

### 3.3 目标用户

| 用户角色 | 核心需求 | 本产品的价值 |
| --- | --- | --- |
| 创作者 / 策展人 | 需要稳定产出视听作品集 | 结构化生成、离线导出、版本化交付 |
| Live 演出者 | 需要长时稳定执行和局部 patch 能力 | 实时 Runtime、cue、patch 窗口 |
| 艺术工程师 | 需要可验证、可扩展、可调试的系统 | 统一协议、trace、replay、门禁机制 |
| 外部规划者开发者 | 需要可组合、可解释的调用边界 | CLI / 文件接口 / API / MCP / 评价回路 |

### 3.4 与其他产品形态的区别

| 维度 | 传统 DJ 软件 | live coding 系统 | 开放式 AI 生成系统 | 本产品 |
| --- | --- | --- | --- | --- |
| 核心范式 | 人类操作播放器 | 人类持续编码 | 模型持续生成 | 外部规划 + Runtime 执行 |
| 一等公民 | 演出过程 | 演出过程 | 内容生成结果 | 可执行作品与系统交付物 |
| 确定性 | 中 | 中 | 低 | 高 |
| 离线 / 实时统一 | 弱 | 中 | 弱 | 强 |
| 视听联合建模 | 弱 | 弱 | 弱 | 强 |
| 可验证性 | 弱 | 弱 | 低 | 高 |

## 4. 核心原则

### 4.1 作品优先于过程

系统的一等公民不是 prompt、聊天记录或推理过程，而是这些结构化交付物：

- Set Plan
- Audio DSL
- Visual DSL
- Unified Timeline
- Render Bundle
- Trace Bundle
- Evaluation Report
- Timeline Patch

### 4.2 一个产品，两个 Runtime

从产品层看，它是一个统一的视听系统。

从工程层看，它应拆成两个长期运行的执行单元：

- Audio Runtime
- Visual Runtime

这样做的原因是：

- 音频需要保护实时线程与低延迟稳定性
- 视觉需要独立 GPU 生命周期与渲染容错
- 两侧崩溃隔离更好
- 便于单独替换、扩展、导出与调试

### 4.3 规划者位于系统外部

规划者负责：

- 制定作品结构或演出计划
- 组织素材与约束
- 提交 patch、评价请求或重编译请求

系统内核负责：

- 校验、编译、调度、播放、渲染、导出

系统只提供接口，不绑定特定 Agent 或 provider。

### 4.4 高层语义优先于低层跟随

视觉系统不应只根据频谱和幅度跳动，而应共享音乐结构语义，例如：

- 当前处于哪个 phrase 或 section
- 是否处在 build、drop、release、cooldown
- 当前 tension 是上升还是释放
- 视觉应扩张、压缩、聚焦还是衰减

### 4.5 优先复用成熟模块

应优先复用：

- 音频回放与 DSP 模块
- MIDI / OSC / Link / 外部桥接
- 节拍、key、段落、特征分析模块
- GLSL / OpenGL / WebGL 渲染模块
- 离线编码与音视频导出模块

真正值得自研的是：

- 统一 IR
- 统一时间模型
- 对外规划与介入协议
- 运行时事件协议
- trace / replay / validation 体系

### 4.6 核心应保持精简，设备与资源应外置

系统内核应尽量保持小而稳定。

具体来说：

- 离散式视觉 / 声场 / 灯光装置应通过统一协议约束下的 adapter plugin 接入
- 音频素材、GLSL 代码、3D 场景模型等应作为外部资源，由 HUB 组织和发布
- Compiler、Scheduler、Runtime Core 只依赖稳定引用、能力描述和 topology 事实，不依赖某个资源包或某种硬件实现必须内置

这条原则的意义在于：在保持系统统一抽象的前提下，允许外部生态持续扩展，而不让核心被具体装置和具体资源形式反向绑死。

## 5. 总体架构

### 5.1 系统结构

```text
External Planning Surface
  -> CLI / File Input / API / MCP / Operator Console

Composition Kernel
  -> Asset IR / Structure IR / Performance IR / Visual IR / Trace IR
  -> Validators / Compiler / Policy Gates

Scheduler And Show State
  -> musical clock / event bus / cue bus / policy gates

Execution Layer
  -> Audio Runtime
  -> Visual Runtime

Adapter Plugin Layer
  -> Audio Output / Instrument Adapter Plugins
  -> Visual Output Adapter Plugins
  -> Lighting Output Adapter Plugins

External Resource Hubs
  -> Audio Asset Hub
  -> GLSL Scene Hub
  -> Scene Model Hub
  -> Lighting Preset Hub

Delivery And Observability
  -> Offline Export / Live Execution / Metrics Summary / Trace Bundle / Replay
```

### 5.2 核心组件及职责

| 组件 | 职责 | 是否持续运行 |
| --- | --- | --- |
| Asset Ingestion | 素材接入、分析、版本化 | 否 |
| Plan Submission | 接收规划与 patch | 否 |
| Compiler | 编译 DSL / IR / Timeline | 否 |
| Validators | Schema、约束、资源与冲突检查 | 否 |
| Scheduler | 统一时间语义、事件分发 | 是 |
| Audio Runtime | 音频执行与导出 | 是 |
| Visual Runtime | 视觉执行与导出 | 是 |
| Adapter Plugin Layer | 统一设备协议到具体装置的映射 | 视插件而定 |
| Metrics Summarizer | 结果汇总与诊断素材输出 | 否 |
| Patch Manager | 局部修订、版本切换与回退 | 否 |

补充说明：

- Adapter Plugin Layer 是核心之外、装置之前的边界层。
- Audio、Visual、Lighting 的具体硬件形态都应在这层实现差异化接入。
- 外部资源 HUB 不属于核心 Runtime，而属于资源供给面。

## 6. 关键对象模型

系统不围绕“某个播放器界面”或“某个 shader 文件”组织，而围绕这些对象组织：

- Asset：可分析、可寻址、可版本化的声音或视觉素材
- Scene：一段稳定的视听状态
- Transition：从一个 Scene 到另一个 Scene 的迁移
- Layer：可单独控制的音乐层或视觉层
- Cue：可触发时间点
- Plan：由外部提交的规划工件
- Constraint：不可违背的规则边界
- Trace：规划、执行、修订的证据链

## 7. 统一 IR 设计方向

至少需要四层 IR：

| IR | 作用 | 典型内容 |
| --- | --- | --- |
| Asset IR | 描述可用素材与分析结果 | bpm、key、section、shader preset、texture tag |
| Structure IR | 描述作品宏观结构 | intro、build、drop、release、ending |
| Performance IR | 描述可执行时间线 | cue、crossfade、filter sweep、stem launch |
| Visual IR | 描述视觉场景与参数轨迹 | scene、uniform automation、camera state |

真正直接进入运行时的是 Performance IR 与 Visual IR；
真正由外部规划者关注的是 Asset IR、Structure IR 和 Patch 工件。

## 8. 视听联动模型

### 8.1 联动不等于频谱跟随

真正稳定的联动应采用双通道：

1. 信号通道：onset、band energy、transient、envelope、centroid
2. 语义通道：beat、bar、phrase、section、drop、release、style shift

### 8.2 共享 Show State

音频与视觉之间共享的状态至少应包含：

- 当前 beat / bar / phrase / section
- tempo、phase、swing
- energy、density、brightness、tension
- 当前场景意图
- 当前 transition 阶段
- patch 是否允许、下一个 patch window 在哪里

### 8.3 软件边界判断

结论很明确：

- 对外是一个整体产品
- 对内是两个独立 Runtime
- 两者通过共享时钟、共享 show state 和统一事件协议耦合

## 9. 工作流模型

### 9.1 离线创作工作流

```text
素材接入
  -> 预分析与标注
  -> 外部提交结构规划
  -> 编译 Audio / Visual DSL
  -> Runtime 离线渲染
  -> 汇总结果与评价数据
  -> 如需则重新提交局部 patch 或新规划
  -> 输出作品集
```

### 9.2 Live 工作流

```text
预生成长段 DSL / Timeline
  -> Audio Runtime 与 Visual Runtime 联合执行
  -> Scheduler 按 lookahead 下发动作
  -> 现场出现新需求或异常
  -> 外部规划者在许可窗口内提交小范围 patch
  -> Runtime 接受 patch 并继续执行
```

### 9.3 离线与实时必须共享的东西

- 相同 IR
- 相同时间语义
- 相同验证规则
- 相同 trace 结构

## 10. 交付物体系

### 10.1 一级交付物

| 交付物 | 说明 |
| --- | --- |
| Set Plan | 一场作品或演出的宏观规划 |
| Audio DSL | 音乐执行脚本或规则描述 |
| Visual DSL | 视觉场景脚本或参数描述 |
| Unified Timeline | 统一时间线与 cue 列表 |
| Render Bundle | 离线导出的音视频或分轨结果 |
| Trace Bundle | 计划、执行、评价、patch 的证据包 |

### 10.2 二级交付物

- 素材索引与分析缓存
- 质量评分报告
- 现场 patch 记录
- 版本差异报告
- 监控汇总快照

## 11. 分阶段规划

### 11.1 Phase 0：证明最小闭环

目标：证明“外部提交可执行作品，Runtime 脱离规划者稳定执行”这件事成立。

范围：

- 有限素材池接入
- 基础节拍 / 段落分析
- Structure IR 与 Performance IR 原型
- Audio Runtime 原型
- GLSL Visual Runtime 原型
- CLI 或文件式计划提交
- 离线导出与 trace 记录

### 11.2 Phase 1：统一视听调度

目标：建立共享时钟、事件总线、show state 与 patch 机制。

范围：

- 统一 cue / event 协议
- Audio / Visual 共享 show state
- 现场 patch API
- 结果汇总与评价输入输出

### 11.3 Phase 2：演出系统化

目标：从离线作品生成，推进到可长期稳定 live 的运行时。

范围：

- 长时运行稳定性
- 故障隔离与降级策略
- 外部控制器 / MIDI / OSC / Link 对接
- 演出模板与场景包管理
- MCP 接入口

### 11.4 Phase 3：开放生态

目标：让人类、外部 Agent 和第三方工具都能稳定调用这套系统。

范围：

- 标准化 CLI / API / MCP 能力
- 第三方素材分析器
- 第三方视觉执行器
- 标准化导出与互操作

## 12. 成功判据

第一阶段成立的判据是：

1. 人类可以在不依赖内置 Agent 的情况下提交规划并成功执行。
2. 外部 Agent 也可以通过同一套接口提交等价规划并成功执行。
3. Runtime 在离线和实时模式下都能脱离规划端长时间稳定运行。
4. 现场 patch 能被边界化、验证并安全回退。
5. 音频与视觉能基于统一时间和共享语义实现真实联动。
