# 外部规划与介入机制顶层设计书

日期：2026-04-01
状态：Draft
前置阅读：01-计算音乐软件调研报告.md，02-视听系统产品定位与规划方案书.md

## 1. 文档目的

本设计书定义外部规划者如何与视听系统协作，包括：

- 规划者的准确定义
- 系统边界与不可逾越约束
- 规划输入、patch 介入、监控汇总与回退机制
- CLI / 文件接口 / API / MCP 等交互形态
- 从 MVP 到开放生态的演进方向

本文要明确避免的误区是：

```text
把 Agent、LLM provider 或持续在线推理误设计成视听产品本体的一部分。
```

本系统不采用这种模式。

## 2. 核心定义

### 2.1 规划者的准确定义

在本系统中，规划者位于系统外部，可以是：

- 人类操作员
- 外部 Agent
- 其他自动化编排程序

规划者负责：

- 提交结构规划
- 提交 DSL 或 IR 工件
- 请求离线导出或 live patch
- 消费监控汇总、trace 与评价结果

规划者不负责：

- 持续播放音频
- 持续驱动视觉帧循环
- 替代调度器做每拍实时决策
- 直接进入音频 callback、render loop 或 GPU 提交链路

### 2.2 系统与规划者的关系

```text
规划者位于系统外部。
系统内核负责校验、编译、调度与执行。
```

两者的基本关系是：

- 规划者提交结构化工件
- 系统校验并编译工件
- Runtime 消费工件并稳定执行
- 观测层汇总结果与诊断数据
- 必要时规划者再次介入，提交 patch 或新计划

### 2.3 顶层设计结论

本系统的正确形式不是 always-on Agent，而是“可外部规划的执行系统”。

即：

- 规划时接收外部输入
- 评价时输出结构化结果
- patch 时接受受限介入
- 异常与变更时允许重新提交
- 其余时间不依赖任何在线规划者持续存活

进一步说，系统内部还应保持以下边界：

```text
稳定的核心编排与执行语义
  + 统一协议约束下的硬件适配插件
  + 可替换的外部资源 HUB
```

这样，规划者、资源提供者和硬件装置实现者都位于系统边界之外，但都通过统一对象和协议接入。

## 3. 核心原则

### 3.1 原则一：外部规划不得成为运行前提

当网络不可用、LLM 服务不可用或外部控制端中断时，已编译完成的作品与演出计划仍必须运行。

### 3.2 原则二：系统只接受结构化产物

系统不消费散文式意图，而应接收机器可验证工件，例如：

- SetPlan
- AudioDsl
- VisualDsl
- TimelinePatch
- EvaluationRequest
- ConstraintUpdate

同样，系统也不应直接消费未经登记的私有设备调用和散落资源路径，而应消费：

- 可声明 capability 的 adapter plugin
- 可声明版本、兼容性和元数据的 resource hub record

### 3.3 原则三：所有外部动作必须经过门禁

规划者只能提出 proposal，不能越过约束直接执行。

最终是否生效，应由以下模块共同决定：

- Schema 校验器
- 约束检查器
- 编译器
- 调度器
- Runtime 准入门禁

### 3.4 原则四：高价值节点介入，低价值环节下沉引擎

节拍同步、混音参数插值、shader 帧更新、lookahead 队列调度等低层工作必须交给 Runtime。

外部规划只处理高价值、低频率、可解释的决策节点。

### 3.5 原则五：局部可改，整体受限

在 live 模式中，外部规划者可以做小范围修补，但不应在未授权情况下重写整个结构。

## 4. 交互形态

### 4.1 MVP 阶段

第一阶段不要求系统内置任何 Agent。

MVP 应优先支持这些入口：

- CLI 命令提交计划
- 文件目录提交 DSL / IR
- 导出 trace、metrics、evaluation input

这意味着：

- 人类可以手写规划文件并调用 CLI
- 外部 Agent 也可以生成同样的规划文件并调用同一套 CLI

### 4.2 正式版入口

随着系统成熟，可演进为以下交互面：

| 入口 | 作用 | 是否为 MVP 必需 |
| --- | --- | --- |
| CLI | 提交计划、编译、运行、导出、patch | 是 |
| 文件接口 | 批量输入输出工件 | 是 |
| API | 服务化接入 | 否 |
| MCP | 外部 Agent 能力发现与调用 | 否 |
| Operator Console | 人类值守与现场监看 | 否 |

### 4.3 不推荐的架构

以下两种架构都不推荐：

1. 在产品内置一个万能 Agent，同时控制全局结构、音频细节、视觉细节和实时执行。
2. 把某个特定 LLM provider 写进系统必选依赖，导致产品边界被外部服务锁定。

推荐形态是：

```text
稳定的视听执行内核 + 中立的对外接口 + 可替换的外部规划者。
```

## 5. 生命周期与触发模型

### 5.1 四类工作阶段

| 阶段 | 触发 | 外部输入 / 系统输出 |
| --- | --- | --- |
| Planning | 新作品、新演出、新素材池 | 输入 Set Plan / DSL / Timeline |
| Evaluation | 离线导出完成、演出回放完成 | 输出 Evaluation Input / Metrics Summary / Trace |
| Patching | Live 现场变更、局部失败、临时插入素材 | 输入 Timeline Patch |
| Recovery | 运行异常、素材缺失、资源超限 | 输入 Fallback Plan 或系统自动降级 |

### 5.2 推荐触发方式

规划者应由事件触发，而不是由系统内部时钟强行定时触发。典型事件包括：

- 新建项目
- 更换素材集
- 修改规则或约束
- 一轮离线渲染完成
- 评分低于阈值
- 现场操作请求 patch
- 某个 Runtime 报告资源或素材异常

### 5.3 Live 模式中的介入窗口

为避免破坏当前时间线，patch 只能发生在明确窗口内，例如：

- 下一 phrase 边界
- 下一 section 边界
- 预留的 emergency slot
- 用户手动授权的 hard cut 点

## 6. 边界与约束

### 6.1 外部规划者可以做什么

- 生成宏观结构
- 选择或替换素材
- 调整后续若干 phrase 的计划
- 调整视觉场景切换与参数轨迹
- 基于汇总结果重编译一段作品
- 生成 fallback plan

### 6.2 外部规划者不可以做什么

- 直接操作音频 callback
- 绕过验证器直接热改 Runtime 内部状态
- 在没有授权的情况下改写已经锁定的 section
- 在关键窗口外重写全局 tempo map
- 在没有资源检查时引入更重的视觉图或音频链路

### 6.3 三类不可破坏边界

#### A. 结构边界

- 已锁定 section 不可整体重排
- 已发布作品不可隐式漂移
- 已确认 cue 不可被无痕覆盖

#### B. 运行边界

- 调度器时钟不可被外部规划者直接篡改
- lookahead 队列内已提交事件不可无检查撤销
- runtime buffer 与 shader frame loop 不接受阻塞型调用

#### C. 资源边界

- CPU / GPU 预算不可突破
- 内存占用不可失控
- 网络不可作为已执行作品的必要依赖

## 7. 交付物协议

### 7.1 总原则

规划者与系统之间不共享自由文本，而共享结构化工件。

### 7.2 关键工件

| 工件 | 作用 |
| --- | --- |
| SetPlan | 宏观结构与段落目标 |
| AssetSelection | 当前可用素材与选择结果 |
| AudioDsl | 音乐控制脚本或规则图 |
| VisualDsl | 视觉控制脚本或 shader 场景图 |
| TimelineIR | 统一时间线 |
| TimelinePatch | 有边界的局部修改 |
| ConstraintSet | 约束规则集合 |
| EvaluationRequest | 请求系统导出评价素材 |
| MetricsSummary | 结构化运行汇总 |
| TraceBundle | 规划、执行、反馈的持久化记录 |

补充边界：

- 规划者提交的是结构化工件，不是具体设备驱动指令
- 插件实现者提交的是 backend capability 和 adapter 实现，不是对核心语义的重定义
- 资源提供者提交的是 hub 中的资源记录，不是强制内置到系统仓库中的静态资产

### 7.3 Timeline Patch 示例

```json
{
  "type": "timeline_patch",
  "version": "0.1",
  "target_show": "show-2026-04-01-a",
  "scope": {
    "from_bar": 129,
    "to_bar": 161,
    "patch_window": "next_phrase_boundary"
  },
  "intent": {
    "goal": "reduce_density_keep_energy",
    "reason": "vocal_stem_conflict"
  },
  "changes": [
    {
      "op": "replace_asset",
      "track": "deck_b",
      "from": "stem/vocal_A",
      "to": "stem/texture_C"
    },
    {
      "op": "update_visual_scene",
      "scene": "drop_corridor",
      "uniforms": {
        "motion_gain": 0.72,
        "noise_scale": 0.31
      }
    }
  ]
}
```

### 7.4 进入 Runtime 前的最小检查

任何外部输入都至少经过以下检查：

1. Schema 校验
2. 约束校验
3. 时间线冲突检查
4. 资源预算检查
5. 素材可用性检查
6. 接受、延后、降级或拒绝决策

## 8. 统一状态机

### 8.1 顶层状态

```text
Draft
  -> Planned
  -> Compiled
  -> Validated
  -> Running
  -> Evaluated
  -> Patched
  -> Recompiled
  -> Running
  -> Archived
```

### 8.2 状态含义

| 状态 | 含义 |
| --- | --- |
| Draft | 概念与素材尚未成形 |
| Planned | 外部已提交结构化方案 |
| Compiled | 方案已编译为可执行 IR |
| Validated | 通过约束和资源检查 |
| Running | Runtime 正在执行 |
| Evaluated | 已完成结果汇总或评分素材输出 |
| Patched | 在有界范围内完成局部修改 |
| Archived | 已冻结为作品或演出记录 |

### 8.3 状态机原则

- Running 期间可以进入 Patched，但必须保留旧版本快照
- Patched 不等于全局重规划
- 任何 RejectedPatch 都必须留下诊断理由

## 9. 监控汇总与评价机制

### 9.1 为什么需要监控汇总层

系统不负责替你做审美判断，但必须负责提供足够好的结构化观察结果，让人类或外部 Agent 可以做下一轮判断。

### 9.2 汇总维度

- 结构完整性
- 音乐兼容性
- 过门平滑度
- 视觉语义一致性
- 视听同步程度
- 资源预算达标情况
- 确定性与可回放性

### 9.3 汇总输出应指向事实

例如：

- 第 97 到 105 小节能量跳变过大
- 第 129 小节视觉 scene 切换早于音频 drop 半拍
- 当前 patch 使 GPU 峰值超预算 18%

这些结果可以被人类阅读，也可以被外部 Agent 再次消费。

## 10. 安全性与鲁棒性

### 10.1 失败默认降级，不默认停机

当外部规划端无法及时完成 patch 时，系统应优先执行降级策略：

- 延用当前 section
- 使用预定义 fallback scene
- 切换到低复杂度视觉模式
- 回退到上一已验证版本

### 10.2 对外部依赖的处理原则

- 外部规划端可以断开，系统不能因此崩溃
- LLM provider 可以更换，系统接口不应变化
- 同一份规划工件应可由人类或 Agent 产生并被等价消费

## 11. 演进路线

### 11.1 MVP

MVP 应证明三件事：

1. 人类能通过 CLI / 文件接口提交规划并运行。
2. 外部 Agent 也能通过同一套接口提交规划并运行。
3. 系统能输出足够结构化的 trace 与 metrics，支持下一轮评价和 patch。

### 11.2 正式版

正式版再逐步增加：

- API 与服务化接入
- MCP 能力发现与调用
- 更细粒度的 patch 授权模型
- 更完备的监控汇总与对比分析

## 12. 结论

本系统不是 Agent 产品，而是 Agent-ready 的视听执行系统。

更准确地说：

```text
它提供稳定的视听内核、清晰的对外调用边界和可追踪的执行结果；
至于规划来自人还是外部 Agent，属于系统之外的选择。
```
