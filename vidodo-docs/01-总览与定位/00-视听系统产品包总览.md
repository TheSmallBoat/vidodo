# 视听系统产品包总览

日期：2026-04-01
状态：Draft

## 1. 目录定位

本目录是一套独立的视听系统产品包文档，不依赖工作区中的其他目录作为前置背景。

它定义的对象不是“内置 Agent 或内置 LLM Provider 的 AI 产品”，而是：

```text
一套可由人类或外部 Agent 提供规划、并由音频 Runtime 与 GLSL 视觉 Runtime 持续执行的视听系统。
```

这里的外部规划者可以是：

- 人类创作者或演出者
- Github Copilot 之类的外部 Agent
- 其他通过 CLI、文件接口、API 或 MCP 接入的自动化系统

系统本体只负责：

- 接收规划与 patch
- 校验约束与资源预算
- 编译为统一 IR 与时间线
- 驱动音频与视觉 Runtime 执行
- 导出结果并汇总运行数据

系统本体不负责：

- 内置 LLM provider 管理
- 常驻在线推理
- 持续自治式创作决策

## 4. 这套文档共同回答的问题

这四份文档共同回答以下问题：

1. 这个系统为什么不是“内置 Agent 的 AI 自治体”。
2. 这个系统为什么既不是传统 DJ 软件，也不是开放式 AI 作曲平台。
3. 为什么规划者可以是人，也可以是外部 Agent，但系统本体必须保持中立。
4. 为什么产品上是一体化视听系统，工程上却应拆成音频与视觉两个 Runtime。
5. 外部规划、系统执行、监控汇总与再介入之间如何形成闭环。

## 5. 核心统一结论

本产品包的统一结论如下：

- 产品是一个整体视听系统，而不是两个孤立产品
- 工程上应采用两个 Runtime，而不是一个大单体
- 规划者位于系统外部，而不是系统内部
- 规划来源可以是人，也可以是外部 Agent
- 离线与实时必须共享同一套 IR、时钟和协议
- 一等公民是规划工件、patch、运行结果与 trace，而不是推理过程

## 7. 使用方式

如果当前目标是做产品边界判断，优先读 02 和 03。

如果当前目标是做 Runtime 实现，优先读 ../02-协议与模型/04-音频与视觉运行时事件协议与数据结构.md 和 ../02-协议与模型/12-视听联动协议设计书.md，再回看 02 和 03。

如果当前目标是做接口设计，优先读 09 和 10。

如果当前目标是做实现选型，优先读 13，再对照 02、04、12。

如果当前目标是做 MVP 落地推进，优先读 14，再对照 02、11、13。

如果当前目标是做详细功能设计，优先读 15，并与 05、07、08、10、14 对照。

如果当前目标是做模块级实现，按以下顺序阅读最合适：16 -> 17 -> 18 -> 19。

如果当前目标是做视觉后端抽象、多屏平面显示或多视角空间显示装置设计，优先读 27，并结合 04、12、15、19 一起看。

如果当前目标是做音频后端、MIDI 与外部乐器接入，优先读 20，并结合 13、19 一起看。

如果当前目标是做音频输出后端抽象、空间扬声器矩阵或舞台化离散声场装置设计，优先读 28，并结合 04、14、19、20 一起看。

如果当前目标是做灯光输出后端抽象、离散式分布式灯光装置或统一场景下的空间光场编排设计，优先读 29，并结合 19、27、28 一起看。

如果当前目标是统一理解“核心精简、硬件以适配插件接入、外部资源以 HUB 组织”的系统级架构意图，优先读 30，并结合 02、14、15、19、27、28、29 一起看。

如果当前目标是做编码前置准备与质量体系，优先读 21，并与 14、15、16 到 20 对照。

如果当前目标是做方向论证或对外说明，应从 01 开始完整阅读.

如果当前目标是做编码启动前的完备性审计，优先读 22，它评估了所有文档的完备度并标注了缺口。

如果当前目标是了解错误码规范和 Diagnostics 格式，优先读 23。

如果当前目标是做任务分配与进度管理，优先读 24（任务卡与里程碑），再对照 14、15。

如果当前目标是搭建仓库与 CI 流水线，优先读 25（开发工作流与分支策略）。

如果当前目标是编写或审查 JSON Schema，优先读 26（Schema 完整清单与版本策略），再对照 05、04、07、10。
