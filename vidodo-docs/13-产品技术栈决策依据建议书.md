# 产品技术栈决策依据建议书

日期：2026-04-01
状态：Draft
前置阅读：02-视听系统产品定位与规划方案书.md，04-音频与视觉运行时事件协议与数据结构.md，09-CLI与MCP能力模型与命令语义.md，12-视听联动协议设计书.md

## 1. 文档目的

本文不直接给出“唯一正确”的技术栈答案，而是给出一套面向本产品边界的选型依据与推荐组合。

目标是帮助团队在以下约束下做决策：

- 系统是外部可规划的确定性视听系统
- 工程上是两个 Runtime，而不是一个大单体
- 离线与实时共享 IR、时间语义与 trace
- 第一阶段优先复用成熟能力，而不是重造底层引擎

## 2. 选型准则

### 2.1 必须满足的硬约束

候选技术栈必须同时满足：

1. 能支撑长时稳定运行。
2. 能清晰分离音频实时链路与非实时控制链路。
3. 能把规划、编译、patch、trace 做成结构化工件，而不是散落脚本。
4. 能支持 CLI / 文件接口 / MCP 三种外部接入形态。
5. 能在 macOS 上可靠开发，并具备后续跨平台可能性。

### 2.2 必须避免的错误选型

- 用单一脚本运行时同时承担离线分析、实时调度、GUI、MCP 与音频 callback
- 把在线推理或远程网络调用挂进实时音频线程
- 为了“语言统一”而放弃成熟音频或图形运行时
- 为了“快速 demo”把 trace、revision、patch 审计省掉

## 3. 选型分层

### 3.1 推荐分为六层

| 层级 | 主要职责 | 选型原则 |
| --- | --- | --- |
| Control Plane | CLI、MCP、API、调度指令入口 | 强结构化、易集成 |
| Composition Kernel | DSL 解析、IR 编译、验证、patch 管理 | 类型清晰、可测试、可并发 |
| Analysis Toolchain | 素材分析、结构识别、标签与特征生成 | 生态成熟优先 |
| Audio Runtime | 播放、混音、自动化、导出 | 实时稳定优先 |
| Visual Runtime | shader、scene、camera、render pass | GPU 与跨平台能力优先 |
| Storage And Observability | 资产索引、trace、evaluation、导出 | 可追踪、可回放、易维护 |

## 4. 核心推荐结论

### 4.1 总体推荐

对本产品，最稳妥的总体方向是：

```text
Rust 作为核心控制与编译层
Python 作为离线分析工具链
成熟音频运行时作为音频执行层
wgpu/GLSL 兼容方案作为视觉运行时基础
JSON/JSONL + SQLite 作为第一阶段工件与索引存储
ffmpeg 作为导出链路
```

### 4.2 为什么不是单语言全包

因为本产品的几个关键约束天然冲突：

- 分析生态最强的往往不是实时最稳的语言
- 实时最稳的语言未必最适合快速做 MCP 与数据处理
- 图形后端与音频后端的最佳选型不一定一致

因此更合理的是：

- 核心边界统一
- 语言与运行时按层选型
- 工件和协议统一，而不是语言统一

## 5. Composition Kernel 选型建议

### 5.1 推荐：Rust

推荐 Rust 承担以下模块：

- IR 类型定义
- Validator
- Compiler
- Scheduler
- Patch Manager
- Trace 写入与 Replay 核心
- CLI 主程序

主要理由：

- 强类型适合稳定 schema 和 revision 语义
- 并发与资源控制较好
- 适合做长时运行控制面与本地 IPC 服务
- 单二进制 CLI 交付方便

### 5.2 为什么不是 Python 作为核心

Python 很适合分析工具链，但不适合作为整个系统核心控制层，原因是：

- 长时实时控制的可预测性较弱
- 并发模型与资源控制不够干净
- 容易把“原型便利”演化成“生产耦合”

### 5.3 为什么不是 Node.js 作为核心

Node.js 适合外部接入和协议适配，但不适合作为主调度和主编译内核，主要问题是：

- 实时稳定性和数值型系统建模不占优势
- 与音频线程和本地系统资源边界更难做严谨隔离

## 6. Analysis Toolchain 选型建议

### 6.1 推荐：Python 生态

推荐 Python 承担：

- 音频特征分析
- 节拍 / downbeat / section 识别
- 符号音乐分析
- 标签抽取、规则候选与离线评价辅助

可优先考虑的生态：

- librosa
- essentia
- music21
- numpy / scipy
- PyTorch 仅用于离线模型推断或特征模块

### 6.2 使用原则

- Python 只在离线或非实时链路中运行
- 分析结果通过文件或本地 IPC 发布为结构化工件
- 不让 Python 进入音频 callback、render loop 或 lookahead 热路径

## 7. Audio Runtime 选型建议

### 7.1 第一阶段推荐：复用成熟音频引擎

结合产品边界，第一阶段不建议自研完整 DSP 图。

更合理的方向是复用成熟音频引擎承担：

- stem / loop 回放
- 基础混音与自动化
- 过渡与量化触发
- 离线导出

### 7.2 为什么这里仍然要单独分析 Rust 音频生态

虽然本文前面的结论是“第一阶段优先复用成熟音频引擎”，但这并不意味着 Rust 音频生态不重要。

原因恰恰相反：

- Core Control Service 已经推荐 Rust 实现
- Visual Runtime 已经偏向 Rust + wgpu
- 如果未来要把更多音频执行能力收回本体系，必须清楚 Rust 音频栈的上限和短板

因此，对 Rust 音频生态的正确判断不是“能不能做音频”，而是：

```text
Rust 音频生态在哪些层足够成熟，
哪些层适合 MVP，哪些层只适合中长期替换，
以及哪些层目前仍不应承担本产品的主闭环风险。
```

### 7.3 Rust 音频生态拆解

#### A. 高层音频播放与控制

代表库：

- Rodio
- Kira
- Awedio

适用定位：

- 本地播放器
- 游戏音效
- 轻量交互式音频应用
- 原型阶段的素材试听或工具面

优点：

- 上手成本低
- 跨平台能力较好
- 对常见格式播放和基本控制较友好

对本产品的判断：

- Rodio 很适合做素材试听、验证工具、小型预览器
- Kira 更适合“动态游戏音频”场景，调音台和动态控制思路值得参考
- Awedio 可用于异步音频处理实验，但不应视为本系统主 runtime 的默认核心

主要限制：

- 更偏“播放库”而不是“长时确定性表演引擎”
- 对 sample-accurate 调度、复杂量化触发、显式 revision / rollback 语义支持不足
- 离线渲染、分轨导出、复杂 patch 生命周期并不是其设计中心

结论：

高层 Rust 音频库适合作为：

- 素材预听工具
- 内部开发调试器
- 非核心工具型应用

不适合作为本产品 MVP 阶段的唯一主音频 runtime。

#### B. 底层音频 I/O

代表库：

- CPAL
- 平台后端能力，如 CoreAudio / WASAPI / ALSA / JACK 等由底层系统接口提供

适用定位：

- 与系统音频设备直接打交道
- 建立输入输出流
- 自建 audio callback 与缓冲交换层

优点：

- Rust 生态里最重要的跨平台底层基石之一
- 很适合做“自己掌控音频线程和缓冲边界”的系统
- 与 Rust 核心控制层整合自然

限制：

- 它只解决 I/O，不解决完整 runtime 问题
- 你仍然需要自己处理：
  - 调度
  - 混音
  - 缓冲策略
  - 实时安全内存分配
  - 格式解码接入
  - 导出与回放一致性

对本产品的判断：

- 如果未来要做纯 Rust Audio Runtime，CPAL 几乎是底座级选择
- 但在 MVP 阶段，单靠 CPAL 仍然离“可外部规划、可 patch、可 rollback 的长时音频执行系统”差很远

结论：

CPAL 适合承担底座，不适合单独被描述为完整方案。

#### C. 音频解码与文件处理

代表库：

- Symphonia
- Hound
- ffmpeg-next

适用定位：

- 媒体解复用
- 常见音频格式解码
- WAV 读写
- 更完整的媒体编解码桥接

优点：

- Symphonia 是 Rust 音频媒体栈里非常关键的一环
- Hound 简单直接，适合 WAV 中间工件和测试输出
- ffmpeg-next 可以在需要更广媒体格式支持时作为桥接层

对本产品的判断：

- Symphonia 很适合作为 Rust 路线中的标准解码层
- Hound 适合做最小离线中间产物和测试工件
- 如果需要完整音视频导出或更复杂编解码，优先仍应考虑外部 ffmpeg 工具链或 ffmpeg 绑定，而不是试图让 runtime 自己吞下所有媒体处理复杂度

限制：

- 解码只是素材进入 runtime 的前半段，不解决调度和 DSP
- 如果把复杂媒体导出完全塞进实时进程，会破坏本产品的边界清晰性

结论：

在 Rust 音频生态中，解码层是相对成熟、最值得采用的一段。

#### D. 实时音频处理与 DSP

Rust 这一层没有单一“统治级”音频 DSP 运行时，更像是若干可组合库的集合。

典型能力可来自：

- dasp 类基础数字音频处理组件
- rustfft / realfft 类频域处理能力
- fundsp 类合成与 DSP 组合能力
- rubato 类重采样能力
- 自定义 CPAL + DSP pipeline

对本产品的判断：

- 如果只需要少量基础处理，如 gain、filter、crossfade、resample，Rust 完全可做
- 如果要做复杂 live coding 级别、图式化 DSP、成熟合成器体系，Rust 现有生态的工程整合成本仍然较高

风险点：

- 生态分散，需要自行拼装
- 实时安全需要团队自己严格把控
- 稍不注意就会从“做产品”滑向“造音频引擎”

结论：

Rust DSP 适合承担受控、范围明确的处理层，不适合作为 MVP 阶段的主要研发黑洞。

#### E. 网络音频与特殊能力

代表方向：

- WebRTC 音频传输
- 蓝牙协议栈交互

对本产品的判断：

- 这些方向是后续生态扩展项，不是当前 MVP 或 Phase 0 的关键路径
- 在“本地优先、可回放、可验证”的产品阶段，不应优先投入

### 7.4 针对本产品的分层判断

如果采用 Rust 路线，建议把音频栈再拆成四层：

| 层级 | 推荐 Rust 组件 | 在本产品中的角色 |
| --- | --- | --- |
| Playback / Tooling | Rodio / Kira | 素材预听、内部工具、轻量预演 |
| Audio I/O | CPAL | 设备流接入与底层回调基座 |
| Decode / File | Symphonia / Hound / ffmpeg-next | 素材解码、WAV 工件、媒体桥接 |
| DSP / Processing | 小范围 DSP crate 组合 | 基础处理、crossfade、resample、特征前处理 |

这张表的关键意思是：

- Rust 音频生态不是没有答案
- 但答案是“分层组合”，不是“一个库包打天下”

### 7.5 对 MVP 的推荐音频路线

#### 路线一：成熟引擎优先

推荐程度：最高。

组合思路：

- 主音频执行由成熟引擎承担
- Rust core 只负责编译、调度、patch、trace
- Rust 音频生态先用于素材处理、工具层或未来替换预研

这里需要进一步明确一件事：

```text
如果选择 SuperCollider server，
我们的 Audio Runtime 更准确地说不是“把音频引擎写在自己进程里”，
而是“一个受 Core 调度的音频客户端 / 桥接层”，
由它去控制外部独立音频服务器进程。
```

### 7.5.1 SuperCollider server 的工程定位

在这种集成模型里，至少有三个角色：

- Core Control Service：负责 revision、timeline、patch、trace、授权与调度
- Audio Runtime Client / Bridge：负责把统一时间线动作翻译成音频服务器可执行命令
- SuperCollider server：负责真正的实时音频执行

建议理解方式如下：

| 角色 | 本质 | 主要职责 |
| --- | --- | --- |
| Core Control Service | 系统主控制面 | 编译、验证、show state、patch 生命周期 |
| Audio Runtime Client | 领域客户端 / 适配层 | 把 timeline action 映射为音频命令、收集 ack、上报 trace |
| SuperCollider server | 外部独立执行服务 | 播放、节点图、buffer、合成、混音、实时调度 |

### 7.5.2 是否像消息中间件

可以类比为“调用成熟的外部服务”，这和调用消息中间件、数据库、搜索引擎在工程关系上确实相近：

- 目标进程独立运行
- 主系统通过协议与其通信
- 外部服务维护自己的内部状态
- 调用方不直接进入服务内部线程模型

但它又不能被过度简化成普通中间件，因为：

- 它处理的是实时音频执行，而不是通用数据请求
- 命令很多带有严格时间语义
- 服务端内部状态是 DSP graph、bus、buffer、node tree，而不是通用记录集

所以最准确的说法是：

```text
它在系统集成方式上像“外部服务”，
在领域语义上则是“实时音频执行引擎”。
```

### 7.5.3 推荐进程关系

建议采用以下关系：

```text
CLI / MCP / File Input
  -> Core Control Service
  -> Scheduler
  -> Audio Runtime Client
  -> SuperCollider server
```

如果展开成更明确的进程关系，可表示为：

```text
+-------------------------------------------------------------------+
| External Planning Surface                                         |
|                                                                   |
|  CLI avctl            MCP Adapter            Plan / Patch Files   |
+-----------------------------------+-------------------------------+
                                    |
                                    v
+-------------------------------------------------------------------+
| Core Control Service                                              |
|                                                                   |
|  - Capability Layer                                               |
|  - Validators                                                     |
|  - Compiler                                                       |
|  - Scheduler                                                      |
|  - Patch Manager                                                  |
|  - Trace / Revision / Policy State                                |
+-----------------------------------+-------------------------------+
                                    |
             timeline actions / patch decisions / revision context
                                    |
                                    v
+-------------------------------------------------------------------+
| Audio Runtime Client / Bridge                                     |
|                                                                   |
|  - Translate Performance IR to audio server commands              |
|  - Maintain action <-> command <-> node mapping                   |
|  - Collect ack / timeout / error                                  |
|  - Report execution result back to trace / patch manager          |
+-----------------------------------+-------------------------------+
                                    |
               OSC or equivalent command / response channel
                                    |
                                    v
+-------------------------------------------------------------------+
| SuperCollider Server Process                                      |
|                                                                   |
|  scsynth or supernova                                             |

+-------------------------------------------------------------------+

State ownership:
  Core Control Service      -> revision / patch / policy / timeline / trace
  Audio Runtime Client      -> command mapping / ack state / rollback command set
  SuperCollider Server      -> synth nodes / buses / buffers / DSP execution state
```

在这里：

- Core 不直接向 SuperCollider 裸发业务指令
- Audio Runtime Client 负责做领域翻译、状态对账与错误归因
- SuperCollider server 只负责音频执行，不承担业务级 revision / patch 审计

### 7.5.4 为什么这里默认写 scsynth，而不是 supernova

这两个名字都属于 SuperCollider 的 server 侧实现，但工程含义不同：

- `scsynth`：经典默认服务器，实现成熟、资料更多、集成路径更清晰
- `supernova`：偏多核并行执行的服务器实现，适合更重的并行 DSP 场景

对本产品的 MVP 阶段，默认写 `scsynth`，主要是因为：

1. 它是更常见、更标准的集成入口。
2. 文档、社区经验和外部示例更多。
3. MVP 当前要解决的是“稳定闭环和清晰边界”，不是先把多核音频执行压到极致。

### 7.5.5 scsynth 与 supernova 的选择依据

| 维度 | scsynth | supernova |
| --- | --- | --- |
| 默认程度 | 更像默认主线 | 更像性能导向变体 |
| 集成复杂度 | 较低 | 较高 |
| 资料与案例 | 更多 | 相对少 |
| 适合阶段 | MVP / Phase 0 / Phase 1 | 性能瓶颈明确后的后续阶段 |
| 当前推荐 | 高 | 中低 |

### 7.5.6 对当前产品的建议判断

对你们当前系统，最合理的顺序不是一开始就在 `scsynth` 和 `supernova` 间做性能竞赛，而是：

1. 先把 `scsynth` 当作外部独立音频执行服务接入。
2. 先验证 revision、timeline、patch、rollback、trace 与音频执行之间的边界是否稳定。
3. 只有当音频图复杂度、并行压力或 CPU 负载真的成为实证瓶颈时，再评估是否切换或补充 `supernova` 路线。

换句话说：

```text
MVP 阶段优先解决“系统是否成立”，
而不是先解决“同一个后端的哪种 server 变体跑得更满”。
```

### 7.5.7 协议与调用边界

Audio Runtime Client 与 SuperCollider server 之间建议采用“命令流 + 确认回执”的模型。

最小能力通常包括：

- boot / shutdown server
- load / free buffer
- create / free synth node
- set node parameter
- trigger transport-like cue
- query health / node tree / resource summary

而我们的 Audio Runtime Client 额外负责：

- 把 Performance IR action 映射为具体 server command
- 保证命令与 revision、timeline entry、causation id 关联
- 记录 server ack 或失败原因
- 把错误上卷到 patch manager / trace 系统

### 7.5.8 状态归属必须分清

这一步非常关键。

以下状态属于 Core：

- revision
- patch ticket
- lock / policy state
- trace id
- timeline ownership
- rollback decision

以下状态属于 SuperCollider server：

- active synth node
- audio bus routing
- loaded buffers
- server-side DSP graph
- runtime-level audio resource usage

以下状态属于 Audio Runtime Client：

- revision 到 server node 的映射表
- timeline action 到 server command 的映射
- ack / timeout / retry 状态
- rollback 时需要撤销的命令集合

如果这三层状态不分清，后续最容易出现的问题是：

- rollback 不知道该回滚哪一层
- trace 无法说明是编译问题、桥接问题还是 server 执行问题
- patch 看似成功，但 server 实际状态与 revision 漂移

### 7.5.9 为什么不让 SuperCollider 直接承担业务主控

原因不是 SuperCollider 不够强，而是本产品的一等公民不是 synth 脚本，而是：

- SetPlan
- IR
- revision
- patch proposal
- trace bundle

因此，SuperCollider 更适合作为：

- 外部音频执行后端
- 低延迟声音引擎
- 受控命令消费者

而不适合作为：

- 全系统的业务状态中心
- patch 审计中心
- CLI / MCP 的直接能力面

### 7.5.10 对 MVP 的具体意义

如果采用这个模型，MVP 阶段的工作会更清晰：

- 你们只需要先定义一小组“可被翻译成 SuperCollider 命令”的音频动作
- Audio Runtime Client 只实现这些动作的映射与确认
- Core 保持对 revision、patch、trace 的完整主权

这正是它适合 MVP 的原因：

- 能快速借用成熟音频执行能力
- 不破坏你们自有 IR / timeline / trace 主架构
- 后续如果替换音频后端，也只需要替换 client bridge，而不必推翻上层协议

适合目标：

- 尽快证明最小闭环
- 控制实时风险
- 把研发重心放在 IR、协议、scheduler、patch、trace

### 7.5.11 音频输出能力模型

从系统设计角度看，最终发声不应被绑定为单一路径，而应抽象为三类能力：

| 能力 | 输入对象 | 实际发声者 | 适用场景 |
| --- | --- | --- | --- |
| audio_asset_playback | stem、loop、one-shot、rendered clip | 播放器 / buffer engine | MVP 主路径、稳定回放 |
| audio_synth_render | note event、control event、synth params | 合成器引擎，例如 SuperCollider | 无素材也能发声、参数驱动生成 |
| audio_midi_to_instrument | MIDI clip、MIDI note stream | sampler / soundfont / 外部虚拟乐器 | 乐器化编排、MIDI 驱动音源 |

这三条路径此前在文档中已有零散覆盖，但还没有被显式收敛成统一的“音频输出模型”。

更准确地说：

- `audio_asset_playback`：已在资产接入、timeline、Audio Runtime Client 中隐含存在
- `audio_synth_render`：已通过 SuperCollider server 路线被部分描述
- `audio_midi_to_instrument`：此前只提到 MIDI 资产与外部桥接，没有把音源后端写清楚

因此，本节的作用是把这三类能力正式固定为音频后端抽象面。

### 7.5.12 代码写完以后，系统要真正工作还需要哪些流程

仅有系统代码并不足以让系统稳定发声。要让系统真正工作，至少还需要补齐以下运行流程：

1. 资产准备流程。
导入外部音频素材，或准备最小 fallback 资产；生成分析缓存并发布到 Asset Registry。

2. 音频后端准备流程。
启动音频引擎或音频服务器；建立 Audio Runtime Client 到后端的连接；校验最小命令集可用性。

3. 绑定配置流程。
建立 `layer -> sample backend` 绑定、`layer / note -> synth backend` 绑定，以及 `MIDI source -> instrument backend` 绑定。

4. 运行前校验流程。
检查 revision 是否已发布、资源预算是否满足当前 mode、fallback 是否存在、instrument preset / buffer 是否可加载。

5. 运行期观测流程。
收集 backend ack / timeout / load fail，并写入 trace、diagnostics、resource summary。

换句话说，代码完成只是前提，系统真正可运行还依赖“资产、后端、绑定、校验、观测”五条流程一起打通。

### 7.5.13 外部虚拟乐器如何处理

外部虚拟乐器不应被当作系统核心内建逻辑，而应被视为一种可替换的 instrument backend。

推荐建模如下：

```text
MIDI / note events
  -> Audio Runtime Client
  -> Instrument Adapter
  -> External Instrument Backend
  -> Audio Output
```

这里的 External Instrument Backend 可以是：

- soundfont sampler
- sfz sampler
- 插件宿主中的 virtual instrument
- 外部硬件音源
- 网络音源后端

### 7.5.14 SoundFont 在系统中的位置

SoundFont 是 `audio_midi_to_instrument` 路径下的一种具体实现，不是系统音频模型本身。

也就是说：

- MIDI 是控制数据
- SoundFont 是某种 preset / sample mapping 资源
- 真正发声的是载入该 SoundFont 的 sampler / instrument engine

因此，对系统更合理的抽象不是“直接支持 SF2”，而是：

- `instrument_backend`
- `instrument_preset_ref`
- `midi_binding`

例如：

```text
instrument_backend = soundfont_sampler
instrument_preset_ref = preset://gm/piano-1
midi_binding = channel-1 -> preset://gm/piano-1
```

这能保证后续即使换成 SFZ、插件宿主或硬件音源，也不需要改动上层 IR 与调度模型。

#### 路线二：Rust 混合式音频 runtime

推荐程度：中。

组合思路：

- CPAL 负责 I/O
- Symphonia 负责解码
- Hound 负责 WAV 类中间工件
- 小范围 DSP crate 组合负责基础处理
- 导出仍走 ffmpeg 或离线后处理链路

适合目标：

- 想尽早为长期 Rust 收敛做技术储备
- 但愿意接受音频 runtime 研发复杂度明显上升

限制：

- MVP 范围必须大幅收窄
- 只能先支持有限素材类型、有限效果、有限调度语义

#### 路线三：纯 Rust 自建完整音频引擎

推荐程度：低。

原因：

- 当前产品最需要自研的是协议和闭环，不是整套底层音频引擎
- 风险、工期与不确定性都过高

### 7.6 候选路线

| 路线 | 适用阶段 | 优点 | 风险 |
| --- | --- | --- | --- |
| SuperCollider server | MVP 到 Phase 1 | 成熟、实时能力强、生态经验丰富 | 与主控制层桥接需要额外封装 |
| Rust 混合式栈：CPAL + Symphonia + Hound + 小范围 DSP crate | Phase 0 预研到 Phase 1 | 控制面统一、可逐步掌控 runtime 内部结构 | 需要自行补齐调度、混音、实时安全和导出语义 |
| JUCE/C++ 自定义播放器层 | Phase 1 以后 | 工程控制力强、插件化与设备兼容更好 | 开发成本高 |
| Rust 音频栈自建轻量播放层 | 仅限范围收窄后 | 统一语言、部署简洁 | 容易滑向重造音频引擎 |

### 7.7 建议判断

若团队目标是先证明“外部规划 + 长时稳定执行 + patch 回退”闭环，应优先选择成熟音频引擎而不是自建底层。

更具体地说：

- MVP 阶段：优先成熟引擎路线
- 并行预研：可用 Rust 混合式栈验证 CPAL + Symphonia 的可行性
- Phase 1 以后：如果闭环已经稳定，再评估是否把部分播放、解码或基础 DSP 逐步收回 Rust

### 7.8 最小 Rust 音频预研建议

如果团队希望尽早掌握 Rust 音频方向，建议只做以下最小预研，不要一开始就做完整 runtime：

1. 用 Symphonia 解码固定格式素材并输出统一 PCM。
2. 用 CPAL 搭一个最小输出流，验证缓冲与时钟边界。
3. 加入最简单的 gain、crossfade、resample 处理链。
4. 用 Hound 输出离线 WAV，验证回放与导出一致性。
5. 把这一套只接到单层 deck 原型，不碰复杂 patch 生命周期。

这样既能获得 Rust 音频栈的真实工程反馈，又不会过早把主项目拖进底层实现泥潭。

## 8. Visual Runtime 选型建议

### 8.1 推荐：Rust + wgpu 为执行基底，GLSL 作为创作输入之一

原因是：

- macOS 上 OpenGL 已不再是理想长期选项
- wgpu 可覆盖 Metal、Vulkan、DX12 等后端
- 视觉 Runtime 仍可保留“以 GLSL 场景为素材输入”的产品表达

### 8.2 实现建议

- 创作层接受 GLSL / scene graph / texture / buffer graph
- 编译层把 shader 输入规范化为可执行后端对象
- 运行层基于 wgpu 统一生命周期、资源预算与离屏渲染

### 8.3 不建议直接把 Web 技术当主视觉 Runtime

Web 技术可用于预览或控制台，但不建议直接承担核心 live 视觉 Runtime，除非明确限定平台和复杂度。

主要原因：

- GPU 生命周期与资源控制边界较弱
- 与系统级时钟、离屏导出和设备控制的整合成本更高

## 9. CLI / MCP / API 适配层建议

### 9.1 CLI

推荐直接由 Rust 主程序提供。

原因：

- 与 core schema 同源
- 部署和版本管理简单
- 便于直接访问本地 artifact store 与 runtime control socket

### 9.2 MCP

推荐两种方式择一：

1. MVP 用 TypeScript 适配层封装 core service。
2. 中长期用 Rust 原生 MCP server，减少双语言接口维护。

推荐判断：

- 如果目标是尽快接入外部 Agent 生态，TypeScript 适配层更快。
- 如果目标是减少协议漂移，后续应把业务语义收敛回 core service。

### 9.3 API

API 可以后置，不必在 MVP 阶段优先。

第一阶段优先级应低于：

- CLI
- 文件式工件流
- MCP tool surface

## 10. 存储与工件格式建议

### 10.1 第一阶段推荐组合

- JSON：小型结构化工件
- JSONL：事件流与资源采样
- SQLite：资产索引、revision、job、ticket、patch audit 元数据
- 内容寻址文件目录：大对象和导出产物

### 10.2 为什么不是一开始就上复杂数据库

第一阶段的核心问题不是分布式扩展，而是：

- 工件是否稳定
- revision 是否可靠
- trace 是否可查
- replay 是否可信

SQLite 加文件对象存储对本地优先的创作系统更合适。

## 11. IPC 与进程边界建议

### 11.1 推荐边界

```text
CLI / MCP / Operator Console
  -> Core Control Service
  -> Scheduler / Patch Manager
  -> Audio Runtime Process
  -> Visual Runtime Process
```

### 11.2 IPC 推荐

MVP 阶段推荐：

- 文件工件 + 本地 socket 控制
- JSON / MessagePack 作为控制面负载

原因：

- 易调试
- 与 trace 结构自然一致
- 不强绑定某个重量级 RPC 框架

### 11.3 注意事项

- 实时线程不得等待远程响应
- Runtime 只消费已验证的 Executable IR 或事件 payload
- 所有跨进程变更都必须带 revision / ticket / causation id

## 12. 导出与媒体处理建议

### 12.1 推荐：ffmpeg 作为统一导出链路

适用范围：

- 音视频封装
- 分轨导出
- 转码代理生成
- evaluation 输入素材拼装

原因：

- 成熟稳定
- 自动化友好
- 与本地文件工件流配合简单

## 13. 可观测性建议

### 13.1 推荐第一阶段能力

- JSONL trace
- patch audit records
- checkpoint snapshot
- evaluation report
- 关键资源峰值与 drift 记录

### 13.2 不建议一开始重投入的方向

- 复杂分布式 observability 平台
- 过重的在线指标系统
- 与实时链路耦合过深的日志库

## 14. 分阶段技术栈建议

### 14.1 MVP 建议组合

- Core / CLI：Rust
- Analysis：Python
- Audio Runtime：SuperCollider server 或等价成熟音频引擎
- Visual Runtime：Rust + wgpu
- Artifact Store：JSON / JSONL + SQLite + 文件目录
- Export：ffmpeg
- MCP：TypeScript 适配层或最小 Rust server

### 14.2 Phase 1 到 Phase 2

- 收敛 CLI / MCP 到同一 core capability surface
- 强化 patch manager、checkpoint、replay
- 如果音频控制需求扩大，再评估 JUCE/C++ 路线

### 14.3 长期演进

- 保持工件格式稳定
- 保持 runtime 分进程
- 允许替换分析器、音频引擎或视觉后端，但不破坏上层协议

## 15. 决策表

| 模块 | 推荐 | 主要依据 |
| --- | --- | --- |
| Core Kernel | Rust | 类型清晰、长时稳定、便于 CLI 与调度一致化 |
| Analysis | Python | 音频与结构分析生态成熟 |
| Audio Runtime | 成熟音频引擎优先 | 避免重造实时音频底层 |
| Visual Runtime | Rust + wgpu | macOS 兼容性、跨平台潜力、GPU 生命周期控制 |
| Artifact Store | SQLite + JSON/JSONL + 文件目录 | 本地优先、易回放、易调试 |
| Export | ffmpeg | 成熟、稳定、自动化友好 |
| MCP Adapter | TypeScript 或 Rust | 生态接入速度与长期一致性权衡 |

## 16. 结论

本产品的技术栈决策不应以“哪门语言最流行”为中心，而应以系统边界和运行约束为中心。

关键判断如下：

1. 核心控制层与实时执行层必须分离，且分别选择最合适的技术。
2. 应优先复用成熟音频与媒体能力，把自研精力集中在 IR、协议、调度、patch 和 trace 上。
3. 工件、schema 与 capability surface 的统一，比单语言统一更重要。

如果遵守这三条，技术栈可以演进，但产品架构不会被栈本身反向绑架。
