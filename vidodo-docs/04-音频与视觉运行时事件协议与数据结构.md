# 音频 Runtime 与 GLSL 视觉 Runtime 事件协议与数据结构

日期：2026-04-01
状态：Draft
前置阅读：02-视听系统产品定位与规划方案书.md，03-外部规划与介入机制顶层设计书.md

## 1. 文档目的

本协议定义音频 Runtime 与 GLSL 视觉 Runtime 之间的共享状态、事件类型、数据结构和运行边界。

目标不是规定某种具体语言或具体 IPC 技术，而是定义一个稳定、可实现、可验证的运行时契约，使以下三件事成立：

1. 音频与视觉在同一套时间语义下联动。
2. 离线渲染与实时执行共用同一套事件模型。
3. 人类或外部 Agent 提交的规划、DSL、IR 和 patch 可以被两个 Runtime 一致消费。

## 2. 设计原则

### 2.1 协议服务于运行时，而不是服务于聊天

协议中的所有对象都应可序列化、可校验、可版本化、可回放。

### 2.2 语义优先于信号

视觉 Runtime 接收的不仅是频谱和幅度，还包括结构语义，例如：

- 当前 beat、bar、phrase、section
- 当前 transition 阶段
- 当前能量、密度、张力状态
- 当前场景意图与 patch 权限窗口

### 2.3 调度优先于渲染

本协议描述的是“何时发生什么”，而不是 shader 内部怎么写、音频链路怎么实现。

### 2.4 时间可映射、状态可冻结、事件可追踪

任何事件都应能回答三个问题：

- 它在音乐时间上的位置是什么
- 它在调度时间上的提交点是什么
- 它对应的状态变更是否已被确认与记录

## 3. 总体拓扑

```text
External Control Client
  -> Plan / Patch / Constraint Input
  -> Compiler / Validators
  -> Scheduler / Show State Bus
  -> Audio Runtime
  -> Visual Runtime
  -> Trace / Replay / Metrics Summary / Export
```

其中：

- External Control Client 可以是 CLI、文件驱动流程、API 调用方或 MCP 客户端
- Scheduler 是事件发布中心
- Audio Runtime 与 Visual Runtime 是事件消费者与局部状态执行者
- Trace 系统记录每次事件发布、接收、执行与回退结果

## 4. 共享时间模型

### 4.1 三种时间

协议中显式区分三种时间：

| 时间类型 | 含义 | 用途 |
| --- | --- | --- |
| MusicalTime | 音乐结构时间 | beat、bar、phrase、section 定位 |
| SchedulerTime | 调度器内部时间 | lookahead、提交、排序、冲突解决 |
| WallClockTime | 真实设备时间 | 音频设备、VSync、外部同步 |

### 4.2 MusicalTime 结构

```json
{
  "beat": 257.0,
  "bar": 65,
  "beat_in_bar": 1.0,
  "phrase": 17,
  "section": "build_a",
  "tempo": 128.0,
  "time_signature": [4, 4]
}
```

### 4.3 时间映射原则

- Scheduler 以 MusicalTime 为主语义来源
- Runtime 执行时映射到本地时钟
- 离线模式下，WallClockTime 可以被虚拟化，但 MusicalTime 与 SchedulerTime 仍必须保留

## 5. 共享 Show State

### 5.1 Show State 的角色

Show State 不是单个事件，而是运行中的共享上下文快照。

它用于：

- 让视觉理解当前音乐结构位置
- 让音频和视觉共享统一场景意图
- 为 patch、回退、评价提供稳定引用点

### 5.2 Show State 最小字段

```json
{
  "show_id": "show-2026-04-01-a",
  "revision": 12,
  "mode": "live",
  "time": {
    "beat": 257.0,
    "bar": 65,
    "phrase": 17,
    "section": "build_a",
    "tempo": 128.0,
    "phase": 0.0,
    "swing": 0.08
  },
  "semantic": {
    "energy": 0.74,
    "density": 0.58,
    "tension": 0.81,
    "brightness": 0.46,
    "motion": 0.69,
    "intent": "compress_then_release"
  },
  "transition": {
    "state": "building",
    "from_scene": "corridor_low",
    "to_scene": "corridor_peak",
    "window_open": true
  },
  "patch": {
    "allowed": true,
    "scope": "next_phrase",
    "locked_sections": ["intro", "drop_a"]
  }
}
```

### 5.3 Show State 更新频率

建议分层：

- 结构字段按 beat、bar、phrase 更新
- 信号摘要按固定采样窗口更新，例如 20ms 到 100ms
- patch 权限和 revision 在事件生效时更新

## 6. 事件模型

### 6.1 事件分类

协议中的事件分为六类：

| 事件类 | 作用 |
| --- | --- |
| TransportEvent | 播放、暂停、跳转、停止、定位 |
| TimingEvent | beat、bar、phrase、section 等结构时间信号 |
| AudioEvent | 音频层触发、混音、过渡、自动化 |
| VisualEvent | 场景切换、uniform 更新、相机与图形状态 |
| SemanticEvent | 能量、张力、意图、过门状态等高层语义 |
| PatchEvent | patch 提交、接受、拒绝、回退 |

### 6.2 通用事件头

所有事件共享统一头：

```json
{
  "event_id": "evt-000128",
  "show_id": "show-2026-04-01-a",
  "revision": 12,
  "kind": "audio.transition.begin",
  "source": "scheduler",
  "musical_time": {
    "bar": 65,
    "beat": 257.0,
    "phrase": 17,
    "section": "build_a"
  },
  "scheduler_time_ms": 93452,
  "wallclock_hint_ms": 1711942105123,
  "priority": 80
}
```

字段说明：

- event_id：全局唯一事件标识
- revision：事件所属已验证版本
- kind：事件类型
- source：external_client、compiler、scheduler、audio_runtime、visual_runtime、patch_manager 等
- musical_time：结构时间定位
- scheduler_time_ms：调度器内部提交时间
- wallclock_hint_ms：实时模式下的目标执行提示
- priority：冲突解决时的优先级参考

## 7. 核心事件类型

### 7.1 TimingEvent

用于广播结构时间边界。

```json
{
  "kind": "timing.phrase.enter",
  "payload": {
    "phrase": 18,
    "section": "drop_a",
    "tempo": 128.0,
    "downbeat": true
  }
}
```

典型事件：

- timing.beat
- timing.bar
- timing.phrase.enter
- timing.section.enter
- timing.transition.window.open

### 7.2 AudioEvent

用于描述音频执行动作。

```json
{
  "kind": "audio.layer.update",
  "payload": {
    "layer_id": "deck_b",
    "op": "crossfade_in",
    "gain_db": -9.0,
    "duration_beats": 8,
    "filter": {
      "type": "lowpass",
      "cutoff_hz": 8200
    }
  }
}
```

典型事件：

- audio.asset.launch
- audio.layer.update
- audio.transition.begin
- audio.transition.end
- audio.fx.automation
- audio.analysis.summary

### 7.3 VisualEvent

用于描述视觉场景与 GLSL 参数更新。

```json
{
  "kind": "visual.scene.update",
  "payload": {
    "scene_id": "corridor_peak",
    "shader_program": "glsl/corridor_peak",
    "uniforms": {
      "u_motion_gain": 0.84,
      "u_noise_scale": 0.29,
      "u_depth_mix": 0.63
    },
    "duration_beats": 16,
    "blend": "ease_in_out"
  }
}
```

典型事件：

- visual.scene.enter
- visual.scene.update
- visual.uniform.automation
- visual.camera.update
- visual.render.mode

### 7.4 SemanticEvent

用于表达高层联动语义。

```json
{
  "kind": "semantic.energy.update",
  "payload": {
    "energy": 0.78,
    "density": 0.61,
    "tension": 0.87,
    "intent": "hold_then_release"
  }
}
```

视觉 Runtime 应优先消费这类事件，再结合音频信号特征做本地细化。

### 7.5 PatchEvent

用于描述 patch 生命周期。

```json
{
  "kind": "patch.accepted",
  "payload": {
    "patch_id": "patch-018",
    "scope": {
      "from_bar": 129,
      "to_bar": 161,
      "window": "next_phrase_boundary"
    },
    "effective_revision": 13,
    "fallback_revision": 12
  }
}
```

典型事件：

- patch.proposed
- patch.validated
- patch.accepted
- patch.rejected
- patch.rolled_back

## 8. 音频到视觉的特征摘要

### 8.1 为什么不直接传整段频谱

视觉 Runtime 不应依赖高带宽原始分析流做主逻辑，因为那会导致：

- 协议噪声过大
- 调试困难
- 离线重放成本高
- 语义层被信号层淹没

更合理的是发送摘要。

### 8.2 AudioAnalysisSummary 结构

```json
{
  "kind": "audio.analysis.summary",
  "payload": {
    "window_ms": 50,
    "rms": 0.42,
    "crest": 1.87,
    "spectral_centroid": 2860.0,
    "low_band": 0.71,
    "mid_band": 0.53,
    "high_band": 0.28,
    "transient_density": 0.66,
    "onset": true
  }
}
```

## 9. 对外规划相关数据结构

### 9.1 PlanSubmission

系统接收的规划提交应至少包含：

```json
{
  "type": "plan_submission",
  "version": "0.1",
  "submitted_by": "external_client",
  "show_id": "show-2026-04-01-a",
  "mode": "offline_render",
  "inputs": {
    "set_plan": "plans/set-plan-a.json",
    "audio_dsl": "dsl/audio-a.json",
```

## 10. 实时边界

### 10.1 外部输入进入实时链路前的规则

任何来自外部规划者的提交，都必须先经过：

1. Schema 校验
2. 约束校验
3. 资源预算检查
4. patch window 检查
5. 版本与锁定区间检查

### 10.2 Runtime 不接受的输入方式

Runtime 不接受以下方式：

- 阻塞式远程调用直接进入音频回调
- 自由文本直接驱动 shader uniform
- 未编译 IR 直接落到 lookahead 队列
- 无版本号 patch 覆盖当前 show state

## 11. Trace 与 Replay

### 11.1 Trace 最小内容

每次执行至少应记录：

- 输入工件版本
- 编译结果版本
- 事件发布时间与执行确认
- patch 接受或拒绝原因
- Runtime 资源峰值
- 导出产物索引

### 11.2 Replay 的用途

Replay 不是附属能力，而是系统可信性的核心：

- 用于复盘 live 演出
- 用于对比两个规划版本
- 用于把系统结果交给外部 Agent 再评价
- 用于故障定位与回退验证

## 12. 结论

本协议不是为某个具体 Agent 服务，而是为“任何外部规划者都能稳定驱动同一套视听 Runtime”服务。

更准确地说：

```text
它定义的是一个中立、可验证、可回放的运行时契约，
使人类和外部 Agent 都能以同样方式调用系统。
```
