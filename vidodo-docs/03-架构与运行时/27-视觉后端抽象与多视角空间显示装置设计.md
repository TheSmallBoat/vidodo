# 视觉后端抽象与多视角空间显示装置设计

日期：2026-04-02
状态：Draft
前置阅读：04-音频与视觉运行时事件协议与数据结构.md，12-视听联动协议设计书.md，15-功能模块设计方案总览.md，19-调度运行时与补丁详细设计.md

## 1. 文档目的

现有文档已经定义了 Visual Runtime 的技术基础、事件协议和运行边界，但对“视觉输出到底面向哪些装置形态、后端如何抽象、功能上需要支持什么”仍然描述不足。

本文补齐以下设计问题：

1. 视觉执行层为什么不能等同于单一图形技术栈。
2. 为什么需要一个与 Runtime 松耦合、可替换的输出后端抽象。
3. 单屏 / 多屏平面显示与多视角空间显示装置应如何统一建模。
4. 以通用 3D GLSL 粒子 / 线条算法为核心时，多个屏幕如何共同呈现一个空间场景。

## 2. 顶层设计结论

### 2.1 Visual Runtime 不应绑定单一输出装置

Visual Runtime 的核心职责应是：

- 消费视觉 IR、timing、semantic 与 patch 事件
- 维护 scene、uniform、camera、budget 等内部状态
- 生成可被显示装置消费的统一渲染结果

它不应直接假定最终输出一定是：

- 单一全屏窗口
- 单机单屏 shader 画面
- 固定数量、固定布局的显示器组合

### 2.2 应引入可替换的视觉输出后端抽象

推荐把视觉执行层拆成两部分：

```text
Visual Runtime Core
  -> Visual Output Backend
```

其中：

- `Visual Runtime Core` 负责 scene、shader 参数、camera rig、视图集合与渲染调度
- `Visual Output Backend` 负责把渲染结果映射到具体显示装置

这样可以保证：

- 上层 DSL / IR / Scheduler 不需要理解具体硬件形态
- 输出装置可以替换，而不改动核心联动与调度逻辑
- 单屏、多屏和平面 / 空间装置可以共享同一套 scene 与时间语义

在系统级边界上，这类后端应进一步被理解为统一协议约束下的 visual adapter plugin。

### 2.3 首批应显式支持两类后端

推荐至少支持以下两类后端：

1. `flat_display_backend`
2. `spatial_multiview_backend`

前者用于操作系统级单屏 / 多屏平面显示。

后者用于一个统一 3D 场景被多个预校准视角截取，并分别输出到多个物理显示屏的空间显示装置。

### 2.4 GLSL Hub 的角色是“场景算法源”，不是“输出后端”

GLSL Hub 应被定义为可扩展的 shader / scene 算法集合，提供例如：

- 3D 粒子系统
- 3D 线条系统
- 体积雾、速度场、噪声场等辅助算法

它负责“生成什么场景”，不负责“这个场景最终如何映射到几个屏幕、几个视角和什么窗口布局”。

进一步说，GLSL Hub、scene model hub、texture hub 都应属于外部资源供给层，而不是 Visual Runtime Core 本体。

## 3. 需求重述

将本需求用更专业、精确的语言表述如下：

```text
系统需要在 Visual Runtime Core 之上定义一个松耦合、可替换的 Visual Output Backend 抽象。
该抽象至少支持两类目标：

1. 基于操作系统多显示能力的单屏 / 多屏平面显示后端；
2. 基于统一 3D GLSL 场景的多视角空间显示后端。

对于多视角空间显示后端，系统以一个通用 3D 粒子 / 线条场景作为母场景，
通过预先标定的多摄像机或多视口配置，从不同观察角度截取若干画面，
分别输出到多个物理屏幕。观众通过多个屏幕的空间拼接，感知为同一 3D 场景的不同投影视图。
```

## 4. 功能目标

### 4.1 平面显示目标

系统应支持：

- 单屏窗口输出
- 多屏独立窗口输出
- 指定窗口位置、尺寸、全屏与边框模式
- 指定每个窗口绑定到哪个系统显示器
- 多屏共享同一 scene，但允许不同 view preset

### 4.2 空间显示目标

系统应支持：

- 在统一 3D 场景中定义多个观察视角
- 每个视角输出到一个独立屏幕窗口
- 每个窗口具有独立分辨率、宽高比与显示位置
- 视角集合可通过标定配置预先对齐
- 在运行期依据统一 Show State 同步刷新所有视角

### 4.3 工程目标

系统还必须保证：

- 后端替换不影响 DSL、IR、Scheduler 和 patch 语义
- 单个屏幕异常不会污染整个视觉内核状态
- 后端状态可追踪、可降级、可回退
- 同一 scene 算法可在不同后端上重用

## 5. 架构拆分

### 5.1 推荐结构

```text
Scheduler / Show State
  -> Visual Runtime Core
       -> Scene Controller
       -> Uniform Automation Engine
       -> Camera Rig Manager
       -> Viewport Composer
       -> Render Budget Controller
       -> Visual Output Backend
            -> Flat Display Backend
            -> Spatial Multi-View Backend
```

### 5.2 模块职责

#### Visual Runtime Core

负责：

- 解释 Visual IR
- 更新 scene、uniform、camera 和 render mode
- 生成一个或多个待输出 view
- 管理 degrade、rollback 和 budget mode

不负责：

- 直接绑定操作系统显示设备策略
- 把某个具体硬件协议写死在核心逻辑里

#### Visual Output Backend

负责：

- 发现或接收显示端点配置
- 建立窗口、交换链和呈现目标
- 把 view 集合映射到具体屏幕
- 回传 backend status、display ack、resource sample

## 6. 统一后端抽象

### 6.1 建议能力接口

后端抽象至少需要回答以下问题：

- 当前后端支持多少显示端点
- 每个端点支持什么分辨率和呈现模式
- 当前 scene 需要输出几个 view
- 每个 view 输出到哪个 display endpoint
- backend 如何回报故障、降级和恢复

### 6.2 建议接口语义

可以抽象为如下能力：

- `prepare_backend(topology, calibration_profile)`
- `apply_show_state(show_state)`
- `render_views(frame_context, view_set)`
- `collect_backend_status()`
- `apply_degrade_mode(mode)`
- `shutdown_backend()`

这里的关键不是函数名，而是语义边界：

- Runtime Core 生成 `view_set`
- Backend 负责把 `view_set` 送到实际屏幕

## 7. 后端一：平面显示后端

### 7.1 定位

`flat_display_backend` 用于标准桌面或装置环境中的单屏 / 多屏输出。

它依赖操作系统提供的多显示器能力，而不是自定义空间投影模型。

### 7.2 适用场景

- 单屏演示
- 双屏 / 三屏并列输出
- 控制台 + 主显示分离
- 固定墙面屏幕阵列的平面分发

### 7.3 必要能力

- 枚举系统 display id
- 为每个 display 创建窗口
- 指定窗口坐标、大小、全屏模式
- 支持每屏独立 `view preset`
- 支持一个 scene 输出多个平面视图

### 7.4 MVP 建议

MVP 阶段可先支持：

- 单机场景
- 手工指定显示器与窗口大小
- 每屏一个窗口
- 一个主 scene + 多个 camera preset

## 8. 后端二：多视角空间显示后端

### 8.1 定位

`spatial_multiview_backend` 用于一种特殊装置：

- 系统内部维护一个统一 3D 场景
- 多个物理屏幕分别显示该场景从不同观察角度得到的画面
- 观众通过多个屏幕的组合，感知为一个被拆分展示的空间场景

它不是简单多屏复制，也不是多个完全独立的 shader。

它的核心思想是：

```text
同一个 3D 母场景
  -> 多个预校准 camera/view frustum
  -> 多个显示屏输出
  -> 形成一个跨屏拼接的空间视觉装置
```

### 8.2 母场景模型

该后端建议把以下类型的 GLSL 内容作为一等场景算法：

- 3D 粒子群
- 3D 线条 / 轨迹系统
- 基于噪声场、速度场和 attractor 的空间结构

这些算法来自 GLSL Hub，但在运行时应被包装为统一的 scene kernel，而不是散落的单文件试玩 shader。

### 8.3 多视角输出原则

每个物理屏幕对应：

- 一个 `display endpoint`
- 一个 `camera preset` 或 `view frustum`
- 一个预定义输出窗口大小和位置

系统在每一帧需要完成：

1. 更新统一 3D scene state
2. 用多个 camera 对同一 scene 做采样
3. 生成多个 view target
4. 分别输出到对应屏幕

### 8.4 该后端解决的问题

它要表达的不是“多个屏幕各播各的内容”，而是：

- 多个屏幕共同组成一个空间观看系统
- 不同屏幕展示的是同一空间对象的不同投影视角
- 视觉一致性来自同一母场景，而不是屏间手工对时

### 8.5 与传统平面多屏的差别

| 维度 | 平面多屏后端 | 多视角空间后端 |
| --- | --- | --- |
| 场景关系 | 可相同也可不同 | 必须来自同一母场景 |
| 视角关系 | 可无关系 | 明确的 camera rig 关系 |
| 对齐方式 | 以窗口布局为主 | 以标定后的视角与屏幕映射为主 |
| 观众感知 | 多块平面画面 | 一个被多屏切分观察的空间场景 |

## 9. 关键对象模型

### 9.1 DisplayEndpoint

```json
{
  "display_id": "display-02",
  "os_handle": "monitor-uuid-02",
  "window": {
    "x": 1920,
    "y": 0,
    "width": 1280,
    "height": 720,
    "fullscreen": true
  },
  "role": "spatial_view_right"
}
```

### 9.2 CameraPreset

```json
{
  "camera_id": "cam-right-01",
  "position": [3.2, 1.4, 5.1],
  "target": [0.0, 0.0, 0.0],
  "up": [0.0, 1.0, 0.0],
  "fov_deg": 42.0,
  "projection": "perspective"
}
```

### 9.3 SpatialViewBinding

```json
{
  "binding_id": "view-bind-right",
  "display_id": "display-02",
  "camera_id": "cam-right-01",
  "viewport": {
    "width": 1280,
    "height": 720
  },
  "calibration_profile": "install-a-v1"
}
```

### 9.4 CalibrationProfile

建议至少包含：

- 物理屏幕角色与编号
- 屏幕分辨率和窗口定位
- camera pose
- fov / aspect / near / far
- 屏幕命名与装置拓扑版本

## 10. 标定与运维流程

### 10.1 为什么必须预标定

多视角空间后端不是“运行时猜测哪个视角投到哪个屏”。

它必须依赖一份预先确认的标定配置，原因是：

- 屏幕数量、尺寸和摆位是装置事实
- camera rig 关系应稳定复用
- 运行时不应把对齐工作放进热路径

### 10.2 推荐流程

1. 枚举系统显示器并生成 `display endpoint`。
2. 给每个屏幕分配逻辑角色，如 left / center / right / top。
3. 为每个逻辑角色绑定窗口尺寸和坐标。
4. 为每个角色绑定一个 camera preset。
5. 用校准测试 scene 预览对齐结果。
6. 固化为 `calibration profile`，供 live / offline run 复用。

### 10.3 运行期要求

- 若屏幕缺失，应明确降级到减少视角数而不是 silent fail
- 若某视口失败，应回传独立 backend status
- patch 不应直接破坏已确认的 display-to-camera binding

## 11. 与现有 Runtime 协议的关系

### 11.1 不新增新的顶层调度范式

该设计不改变当前系统的核心关系：

- Scheduler 仍然是视觉事件的统一发布点
- Visual Runtime 仍然是视觉事件的消费者
- Show State、timing、semantic、patch 语义保持不变

### 11.2 主要扩展点

真正新增的是 Visual Runtime 内部的能力边界：

- 在 scene / camera / uniform 之外，新增 output backend 抽象
- 在单 camera 输出之外，新增 multi-view scene projection 能力
- 在单窗口输出之外，新增 display topology 与 calibration profile

## 12. MVP 范围建议

### 12.1 建议纳入 MVP 的内容

- `flat_display_backend` 单屏 / 多屏平面输出
- `spatial_multiview_backend` 的最小骨架
- 基于 3D 粒子或线条场景的单母场景多视角输出
- 手工标定的 `calibration profile`
- 每屏一个窗口、每窗一个 camera preset

### 12.2 建议后置的内容

- 投影融合与 edge blending
- 非平面曲面映射
- 自动标定
- 多机分布式显示同步
- 复杂编辑器式装置配置界面

## 13. 一句话总结

```text
Vidodo 的视觉执行层不应只是一套 shader 技术栈，
而应是“统一 3D 场景内核 + 可替换输出后端”的结构。

其中最基本的后端是基于操作系统多显示能力的平面显示后端，
更特殊的后端是基于统一 3D GLSL 场景、通过多摄像机 / 多视口把不同视角分发到多个屏幕的空间显示装置后端。
```
