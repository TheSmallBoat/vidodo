# 调度运行时与补丁详细设计

日期：2026-04-01
状态：Draft
前置阅读：04-音频与视觉运行时事件协议与数据结构.md，08-实时补丁授权与回退策略.md，12-视听联动协议设计书.md，15-功能模块设计方案总览.md

## 1. 文档目的

本文细化 Scheduler、Audio Runtime Client、Visual Runtime 与 Patch Manager 四个模块的详细设计。

目标是明确系统如何在统一时间语义下执行、联动、打补丁、回退与降级。

## 2. 核心关系

```text
Revision / Timeline
  -> Scheduler
  -> Audio Runtime Client
  -> Visual Runtime
  -> Trace Writer

PatchProposal
  -> Patch Manager
  -> Scheduler / Runtime
  -> Rollback / Degrade / Commit

Lighting Intent
  -> Scheduler / Show State
  -> Lighting Control Core / Backend
  -> Trace / Rollback / Degrade
```

## 3. Scheduler 设计

### 3.1 职责

- 维护 musical clock
- 维护 lookahead queue
- 发布 timing / audio / visual / semantic 事件
- 维护最小 Show State

### 3.2 输入

- Unified Timeline
- active revision
- patch activation instruction
- rollback / degrade instruction

### 3.3 输出

- runtime event stream
- timing boundaries
- show state snapshots

### 3.4 核心状态

- current_bar
- current_beat
- current_phrase
- current_section
- lookahead_cursor
- pending_patch_activation

### 3.5 调度状态机

```text
Idle
  -> Primed
  -> Running
  -> Paused
  -> Stopped
```

### 3.6 实时约束

- 不做阻塞 I/O
- 不访问慢速数据库
- 不动态解析大工件
- 所有执行对象必须事先编译为 runtime-friendly payload

## 4. Audio Runtime Client 设计

### 4.1 职责

- 消费 audio event
- 翻译为外部音频后端命令
- 跟踪 action 到 command / node mapping
- 报告 ack、error、timeout、resource summary

### 4.2 输入

- audio event
- timing event
- transport event
- patch activate / rollback

### 4.3 输出

- audio backend command
- execution acknowledgement
- runtime health sample

### 4.4 内部子模块

- Command Translator
- Node Mapping Store
- Ack Collector
- Backend Health Monitor

### 4.5 最小动作集

- load_buffer
- free_buffer
- launch_asset
- stop_layer
- set_gain
- set_filter
- crossfade

### 4.6 状态归属

- action -> command mapping
- command -> backend node mapping
- rollback command set
- pending ack set

### 4.7 音频输出路径模型

Audio Runtime Client 不应假定系统只有一种发声方式。建议显式支持三类路径：

- `audio_asset_playback`
- `audio_synth_render`
- `audio_midi_to_instrument`

三者的运行语义如下：

| 路径 | 输入 | 后端动作 |
| --- | --- | --- |
| audio_asset_playback | asset launch / layer update | load buffer、play、mix、automate |
| audio_synth_render | note / control / synth params | create synth node、set param、free node |
| audio_midi_to_instrument | midi clip / midi note stream | route MIDI to instrument backend |

### 4.8 外部虚拟乐器接入模型

外部虚拟乐器建议通过 Instrument Adapter 接入，而不是直接暴露给 Scheduler 或 Core。

```text
Scheduler
  -> Audio Runtime Client
  -> Instrument Adapter
  -> External Instrument Backend
```

Instrument Adapter 的职责：

- 解析 midi binding
- 选择目标 instrument backend
- 管理 preset / patch / program change
- 上报 instrument ack / error / timeout

### 4.9 Instrument Binding 对象

建议引入绑定对象：

```json
{
  "binding_id": "inst-bind-018",
  "binding_type": "midi_to_instrument",
  "target_backend": "soundfont_sampler",
  "preset_ref": "preset://gm/piano-1",
  "midi_channel": 1,
  "latency_profile": "normal"
}
```

这样 Scheduler 只负责下发 MIDI / note 相关事件，不需要理解具体 SoundFont、插件或硬件细节。

### 4.10 SoundFont 的处理位置

在该模型中：

- MIDI clip 属于 Planning / IR 输入
- SoundFont 属于 instrument backend 资源
- Audio Runtime Client 负责把 MIDI 事件路由给对应 Instrument Adapter

因此，SoundFont 不应直接出现在 Scheduler 核心逻辑中，而应出现在：

- preset registry
- instrument binding config
- backend adapter

### 4.11 代码写完后仍需具备的运行流程

即使运行时代码已经写好，系统要真正发声，还必须具备以下流程：

1. 后端可用性注册。
sample backend、synth backend、instrument backend 必须上线并通过健康检查。

1. 资产或预设准备。
buffer 必须可加载，synth preset 必须可调用，instrument preset / SoundFont 必须可解析。

1. 绑定建立。
建立 `layer -> backend` 与 `midi channel -> instrument preset` 绑定。

1. 运行前检查。
确认 backend 在线、资源存在、fallback 存在。

1. 运行期观测。
收集 ack / timeout / missing preset / load fail，并写入 trace 与 diagnostics。

## 5. Visual Runtime 设计

### 5.1 职责

- 执行 scene enter / update
- 应用 uniform automation
- 管理 camera / render mode
- 消费 semantic 与 timing 联动信息
- 通过可替换 output backend 把 view 集合映射到实际显示装置

### 5.2 输入

- visual event
- timing event
- semantic event
- patch / rollback / degrade command

### 5.3 输出

- scene ack
- render status
- resource sample
- export frame / stream

### 5.4 内部子模块

- Scene Controller
- Uniform Automation Engine
- Camera State Controller
- Render Budget Controller
- Camera Rig Manager
- Viewport Composer
- Output Backend Adapter

### 5.5 最小状态

- active_scene
- active_scene_variant
- uniform_state
- camera_state
- safety_mode
- output_backend_state
- display_topology_ref
- calibration_profile_ref

### 5.6 输出后端抽象

Visual Runtime 不应把具体显示装置写死在核心渲染逻辑中。

建议在 Runtime Core 下增加 `Visual Output Backend` 抽象，至少支持：

- `flat_display_backend`：基于操作系统多显示能力的单屏 / 多屏平面输出
- `spatial_multiview_backend`：基于统一 3D 场景、多 camera / 多 viewport 的空间显示装置输出

Scheduler 不需要理解显示器数量、窗口坐标或 camera rig 细节，它只需要把 visual event、timing event、semantic event 和 patch 指令交给 Visual Runtime。

### 5.7 灯光控制与统一编排边界

灯光装置虽然不属于当前 MVP 的双 Runtime 最小闭环，但从系统边界上应被视为与视觉、音频同一编排控制域中的第三类执行后端。

推荐结构：

```text
Scheduler / Show State
  -> Lighting Control Core
       -> Lighting Output Backend
```

其边界应保持与视觉、音频一致：

- Scheduler 负责统一时间语义和状态窗口
- Lighting Control Core 负责把场景语义解析为 cue / fixture command set
- Lighting Output Backend 负责把 cue 映射到具体 universe、fixture address 或分布式灯具节点

这样可以保证视觉 view group、音频 route set 与灯光 cue set 在同一 patch / rollback / trace 机制下协同运行，而不引入独立的临时控制面。

Visual Runtime 再依据 active scene、camera rig、display topology 和 calibration profile 生成 view 集合，并交给 backend 呈现。

## 6. Patch Manager 设计

### 6.1 职责

- 接收 patch proposal
- 做窗口、授权与资源准入判断
- 生成 patch ticket
- 驱动 activate / monitor / commit / rollback / degrade

### 6.2 输入

- PatchProposal
- run status
- validator result
- asset readiness
- scheduler window state

### 6.3 输出

- PatchDecision
- patch ticket
- activation instruction
- rollback instruction

### 6.4 Patch 状态机

```text
Proposed
  -> Authorized
  -> Validated
  -> Staged
  -> Activated
  -> Monitored
  -> Committed
```

失败分支：

```text
Proposed -> Rejected
Staged -> Expired
Activated -> RolledBack
Activated -> Degraded
```

### 6.5 最小支持 patch 类型

MVP 阶段仅支持：

- local_content patch
- 参数收缩类 patch

不支持：

- 全局 tempo rewrite
- 大范围 structural patch

## 7. Show State 设计

### 7.1 作用

Show State 是调度层和双 Runtime 的共享上下文，而不是某一模块的私有状态。

### 7.2 最小字段

- revision
- mode
- time
- semantic
- transition
- patch
- active_audio_layers
- active_visual_scene

### 7.3 更新触发

- bar / phrase / section 边界
- patch 生效
- rollback / degrade
- scene 切换

## 8. 运行时失败模型

### 8.1 Scheduler Failure

- event queue overflow
- invalid revision activation
- missed boundary

处理：

- trace
- degrade if needed
- freeze further patch

### 8.2 Audio Runtime Failure

- backend timeout
- resource overload
- buffer missing

处理：

- error ack
- patch rollback or fallback command
- resource warning trace

### 8.3 Visual Runtime Failure

- scene load failed
- shader compile failed
- gpu budget exceeded

处理：

- switch safe scene
- mark degraded
- report issue to Patch Manager and Trace Writer

## 9. 回退与降级设计

### 9.1 回退来源

- patch proposal fallback_revision
- checkpoint snapshot
- previous published revision

### 9.2 回退流程

```text
Issue Detected
  -> Patch Manager decides rollback
  -> Scheduler stages revert window
  -> Audio / Visual receive rollback commands
  -> Trace Writer records rollback
```

### 9.3 降级流程

```text
Runtime Stress Detected
  -> Patch Manager enters degrade mode
  -> Visual Runtime drops complexity
  -> Audio Runtime reduces optional layers
  -> System continues with safe state
```

## 11. MVP 范围

MVP 阶段收敛为：

- 单 active revision 执行
- 最小 show state
- timing / audio / visual / patch 四类事件
- local_content patch
- deferred rollback
- safe degrade 模式

## 12. 验收标准

该模块链路达标，当且仅当：

1. Scheduler 能稳定驱动双 Runtime 消费同一 revision。
2. Audio 与 Visual 均可返回结构化 ack / error。
3. local_content patch 可在窗口内成功生效。
4. rollback 与 degrade 都有明确 trace 记录。

## 13. 结论

调度、运行时与补丁模块共同构成系统的执行主骨架。它们的设计重点不是“效果多强”，而是“边界清晰、时间一致、可补丁、可回退、可追踪”。
