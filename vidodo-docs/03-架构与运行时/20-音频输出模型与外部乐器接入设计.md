# 音频输出模型与外部乐器接入设计

日期：2026-04-01
状态：Draft
前置阅读：13-产品技术栈决策依据建议书.md，19-调度运行时与补丁详细设计.md

## 1. 文档目的

本文单独定义系统的音频输出模型，以及外部乐器、SoundFont、MIDI 音源后端的接入方式。

目标是回答以下问题：

1. 系统最终如何发出声音。
2. 没有内置大素材库时，音频输出如何成立。
3. 外部虚拟乐器、SoundFont、sampler 在系统中的位置是什么。
4. 代码完成后，要让系统真正工作，还需要哪些运行流程。

## 2. 顶层结论

### 2.1 音频输出不应被建模成单一路径

系统不应假定“所有声音都来自 sample 回放”，也不应假定“所有声音都来自 synth”。

更合理的抽象是三条并行存在的输出路径：

- `audio_asset_playback`
- `audio_synth_render`
- `audio_midi_to_instrument`

### 2.2 音频输出依赖后端能力，而不是依赖内置素材

系统是否能发声，不取决于是否内置大量音色，而取决于是否有：

- 可用素材
- 可用 synth backend
- 可用 instrument backend

这些素材和预设本身应优先被组织在外部资源 HUB 中，而不是长期内置在核心仓库里。

### 2.3 外部虚拟乐器应被建模为可替换 backend

外部虚拟乐器不是系统主脑，也不是 DSL 的一部分。

它应被看作：

- 一个可替换的 instrument backend
- 由 Audio Runtime Client 通过 Instrument Adapter 调用

在系统级抽象上，它进一步应被理解为统一协议约束下的一类 adapter plugin。

### 2.4 音频装置输出也应被建模为可替换 backend

除了“声音从哪里来”，系统还必须回答“声音最终输出到哪里”。

因此，Audio Runtime 在输出层还应支持可替换的 `Audio Output Backend` 抽象，包括：

- `system_audio_backend`
- `spatial_speaker_matrix_backend`

详见 28 号文档。

## 3. 音频输出三路径

### 3.1 audio_asset_playback

定义：

通过音频素材文件直接回放产生声音。

典型输入：

- stem
- loop
- one-shot
- rendered clip

典型运行链路：

```text
Plan / Performance IR
  -> Scheduler
  -> Audio Runtime Client
  -> Sample Playback Backend
  -> Audio Output
```

后端职责：

- load buffer
- launch playback
- stop layer
- gain / filter / crossfade automation
- report ack / load fail / timeout

适用场景：

- MVP 主路径
- 确定性高的回放型编排
- trace / replay / rollback 最容易先打通

### 3.2 audio_synth_render

定义：

通过合成器或参数驱动音频引擎实时生成声音。

典型输入：

- note event
- control event
- synth param set
- envelope / filter / modulation automation

典型运行链路：

```text
Plan / Performance IR
  -> Scheduler
  -> Audio Runtime Client
  -> Synth Backend
  -> Audio Output
```

典型后端：

- SuperCollider server
- 其他可控 synth engine

后端职责：

- create synth node
- set node param
- route bus
- free node
- report execution ack / error

适用场景：

- 没有外部样本时仍能发声
- 程序化声音生成
- 受控参数驱动的动态声音

### 3.3 audio_midi_to_instrument

定义：

通过 MIDI 或 note stream 驱动外部乐器 / sampler / 音源后端发声。

典型输入：

- midi clip
- midi note stream
- program / patch change
- control change

典型运行链路：

```text
Plan / MIDI IR
  -> Scheduler
  -> Audio Runtime Client
  -> Instrument Adapter
  -> Instrument Backend
  -> Audio Output
```

适用场景：

- 乐器化编排
- 外部虚拟乐器接入
- SoundFont / SFZ / 插件宿主 / 硬件音源驱动

## 4. 三路径比较

| 路径 | 确定性 | 工程复杂度 | MVP 优先级 | 典型风险 |
| --- | --- | --- | --- | --- |
| audio_asset_playback | 高 | 低到中 | 最高 | 素材准备与 buffer 管理 |
| audio_synth_render | 中到高 | 中 | 高 | synth 参数模型与 server 桥接 |
| audio_midi_to_instrument | 取决于音源后端 | 中到高 | 中低 | 音源绑定、preset 管理、兼容性 |

## 5. 外部虚拟乐器模型

### 5.1 基本抽象

外部虚拟乐器不直接暴露给 Core，而通过 Instrument Adapter 接入：

```text
Scheduler
  -> Audio Runtime Client
  -> Instrument Adapter
  -> External Instrument Backend
```

### 5.2 Instrument Adapter 的职责

- 解析 instrument binding
- 管理 preset / patch / channel mapping
- 把 MIDI 事件翻译为具体 backend 指令
- 收集 backend ack / timeout / missing preset / load fail

### 5.3 External Instrument Backend 的可能形态

- soundfont sampler
- sfz sampler
- plugin host 中的 virtual instrument
- 外部硬件音源
- 网络音源后端

## 6. SoundFont 的位置

### 6.1 SoundFont 不是系统主模型

SoundFont 是一种具体资源形式，不应被抬升为系统音频模型本身。

对系统来说：

- MIDI 是控制信号
- SoundFont 是某种可载入的乐器预设资源
- 真正发声的是载入该资源的 sampler / instrument engine

### 6.2 更合理的系统抽象

推荐引入以下对象：

- `instrument_backend`
- `instrument_preset_ref`
- `midi_binding`

例如：

```text
instrument_backend = soundfont_sampler
instrument_preset_ref = preset://gm/piano-1
midi_binding = channel-1 -> preset://gm/piano-1
```

### 6.3 为什么不直接把 SF2 写进核心逻辑

因为如果系统把 SF2 / SFZ / 某个插件格式写死到调度核心里，后续会出现：

- 后端难以替换
- patch / trace 无法保持统一抽象
- IR 被某种资源格式绑死

```json
{

  "binding_type": "audio_asset_playback",
  "target_backend": "sample_backend_a",
  "target_ref": "asset-loop-kick-a",

  "fallback_ref": "asset-fallback-kick"

```

### 7.2 InstrumentBinding

```json
{
  "binding_id": "inst-bind-018",
  "binding_type": "midi_to_instrument",
  "target_backend": "soundfont_sampler",
  "preset_ref": "preset://gm/piano-1",
  "midi_channel": 1,
  "latency_profile": "normal"

```

```json
{
  "backend_id": "soundfont_sampler",
  "backend_kind": "instrument",
  "status": "available",
  "supports": ["midi_note", "program_change", "preset_load"],
  "latency_class": "interactive"
}
```

## 8. Audio Runtime Client 的角色

### 8.1 Audio Runtime Client 不直接“做声音”

它的职责是：

- 消费 Performance IR / runtime event
- 识别该 action 应走哪条输出路径
- 把 action 翻译给正确 backend
- 管理 action / command / node / binding 映射
- 收集 backend 执行反馈

### 8.2 不同路径的翻译职责

- asset playback：action -> buffer / layer command
- synth render：action -> synth node / param command
- midi to instrument：midi event -> instrument adapter command

## 9. 运行前必须补齐的流程

### 9.1 资产准备流程

- 导入音频素材
- 生成分析结果
- 发布 AssetRecord
- 建立 fallback 资产

### 9.2 后端准备流程

- 启动 sample backend / synth backend / instrument backend
- 建立健康检查与状态注册
- 验证最小命令集可执行

### 9.3 绑定配置流程

- layer -> sample backend
- layer / note -> synth backend
- midi channel -> instrument preset

### 9.4 运行前检查流程

- revision 已发布
- backend 在线
- preset / buffer / synth 可加载
- fallback 存在
- 当前 mode 资源预算满足

### 9.5 运行期观测流程

- 收集 ack / timeout / load fail
- 收集 missing preset / binding mismatch
- 写入 trace、diagnostics、resource summary

## 10. MVP 建议

### 10.1 MVP 主路径

优先实现：

- `audio_asset_playback`
- `audio_synth_render`

后置实现：

- `audio_midi_to_instrument`

### 10.2 为什么 MIDI 乐器路径后置

不是因为它不重要，而是因为它天然引入：

- instrument backend 选择问题
- preset 管理问题
- 格式兼容问题
- 设备 / 插件 / sampler 差异问题

这些问题都不应阻塞 MVP 最小闭环。

### 10.3 MVP 对外部乐器的最小支持

MVP 若要支持外部乐器，建议只做到：

- 单一 instrument backend 类型
- 少量 preset binding
- 单通道 MIDI routing
- 明确的 trace 与错误上报

## 11. 失败模型

### 11.1 asset playback 失败

- buffer_missing
- asset_not_ready
- backend_timeout

### 11.2 synth render 失败

- synth_create_failed
- param_apply_failed
- backend_unavailable

### 11.3 midi to instrument 失败

- preset_missing
- backend_not_available
- midi_binding_invalid
- program_change_failed

## 12. 结论

音频输出模型的关键不是“系统是否内置素材”，而是系统是否清楚地区分三类发声路径，并把每条路径都建模为可绑定、可观测、可回退的后端能力。

关键判断如下：

1. 音频输出至少应区分 asset playback、synth render、midi to instrument 三条路径。
2. 外部虚拟乐器与 SoundFont 应通过 instrument backend + preset binding 接入，而不是写死在调度核心里。
3. 代码完成后，仍需要资产、后端、绑定、校验、观测五类运行流程一起打通，系统才真正能工作。
