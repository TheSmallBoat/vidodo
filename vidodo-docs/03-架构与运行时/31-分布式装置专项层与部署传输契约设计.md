# 分布式装置专项层与部署传输契约设计

日期：2026-04-02
状态：Draft
前置阅读：14-MVP技术架构图与Phase0实施清单.md，19-调度运行时与补丁详细设计.md，29-灯光输出后端抽象与离散式分布式灯光装置设计.md，30-统一适配插件协议与外部资源HUB设计.md

## 1. 文档目的

30 号文档已经给出了一条关键判断：多节点分布式装置所需的消息总线和节点编排，不应直接并入 Vidodo Core。

但要让这条判断在工程上可执行，还需要进一步回答：

1. 这个“分布式装置专项层”到底是什么。
2. 它与 Vidodo Core、Adapter Plugin、Concrete Device 的边界如何划分。
3. 多节点装置至少需要哪些正式对象来表达节点、部署和传输契约。

本文的目标就是把这三点对象化，为后续 schema 和专项仓库设计提供稳定边界。

## 2. 顶层设计结论

### 2.1 分布式装置专项层不是 Core 内部模块

它更准确的定位是：

```text
Vidodo Core 之外、Concrete Device 之上、面向多节点部署的专项编排层
```

它只在特定装置部署形态下存在，不是所有项目都必须携带的一层。

### 2.2 它解决的是部署问题，不是作品语义问题

这个专项层负责的不是：

- 定义作品结构
- 重写 DSL / IR
- 改写 patch 授权规则
- 重新定义 show state 语义

它负责的是：

- 把统一工件分发给多个节点
- 保持跨节点时序与状态一致
- 管理节点存活、健康与故障隔离
- 让多个 adapter plugin 以稳定方式协同执行

### 2.3 它应围绕三类正式对象展开

最小对象集合建议至少包括：

1. `distributed_node_descriptor`
2. `deployment_profile`
3. `transport_contract`

它们共同描述：

- 谁参与部署
- 这些节点如何被编组
- 节点之间通过什么总线与传输语义协作

## 3. 架构位置

建议采用以下分层：

```text
Vidodo Core
  -> Adapter Contract
  -> Distributed Device Orchestrator
       -> Node Registry
       -> Deployment Planner
       -> Transport Manager
       -> Health Aggregator
       -> Node Agent
       -> Concrete Adapter Plugin
       -> Concrete Device
```

其中：

- `Vidodo Core` 只输出统一工件和统一状态
- `Distributed Device Orchestrator` 只负责把这些工件稳定分发到多个节点
- `Node Agent` 负责单节点内的 plugin 生命周期和本地设备映射

## 4. 三类正式对象

### 4.1 DistributedNodeDescriptor

该对象用于描述一个实际参与分布式装置的节点。

它至少应回答：

- 节点是谁
- 节点承担哪类角色
- 节点上运行哪些 adapter plugin
- 节点承载哪些 topology shard 或 endpoint group
- 节点通过哪些 transport 接入编排层

建议字段：

- `node_id`
- `node_role`
- `host_ref`
- `plugin_refs`
- `assigned_topologies`
- `resource_hub_mounts`
- `transport_refs`
- `health_endpoint`
- `status`

### 4.2 DeploymentProfile

该对象用于描述一次具体装置部署。

它至少应回答：

- 当前是单节点还是多节点部署
- 哪些节点属于这次部署
- 哪些 transport contract 生效
- 谁是时钟主节点
- 是否需要预热资源和滚动发布
- 节点故障时的降级策略是什么

建议字段：

- `deployment_id`
- `mode`
- `node_refs`
- `transport_refs`
- `time_authority`
- `resource_prewarm_policy`
- `rollout_strategy`
- `failure_policy`
- `trace_policy`

### 4.3 TransportContract

该对象用于描述节点之间如何通信。

它不是单纯的网络地址表，而是一个传输语义契约。

它至少应回答：

- 这是控制总线、状态总线还是健康总线
- 使用什么协议
- 是否要求有序、可靠、广播或点对点
- 延迟预算是多少
- 节点掉线时允许什么退化行为

建议字段：

- `transport_id`
- `bus_kind`
- `protocol`
- `topology`
- `ordering`
- `delivery_guarantee`
- `latency_budget_ms`
- `reconnect_policy`
- `security_profile`

## 5. 与消息总线的关系

### 5.1 消息总线只是 TransportContract 的一种实现

在这个模型里，不应先问“要不要做消息总线”，而应先问：

```text
当前部署需要什么 transport contract？
```

只有当 `transport_contract` 需要：

- 跨节点广播 show state
- 汇总健康与 ack
- 保证控制消息的时序或可靠性

时，消息总线才成为合适实现。

### 5.2 这不是作品协议的一部分

无论 transport 最终选用：

- WebSocket
- NATS
- ZeroMQ
- OSC bridge
- gRPC stream
- Art-Net / sACN bridge 组合

都不应改变 Core 的：

- DSL
- IR
- Show State
- Patch 语义
- Trace 语义

## 6. 与专项仓库的关系

如果一个装置进入以下形态：

- 多机屏幕墙
- 多节点扬声器矩阵
- 多节点灯具网络
- 需要专门的 transport manager、agent、deploy tool

则更推荐把其实现放进独立仓库，例如：

- `vidodo-adapter-displaywall-*`
- `vidodo-adapter-speakergrid-*`
- `vidodo-adapter-lightgrid-*`

这些仓库共享：

- Adapter Contract
- Deployment Object Model
- Transport Contract Model

但不要求与 Vidodo Core 同步发布所有实现细节。

## 7. 对当前文档体系的影响

本文引入的判断是：

1. 分布式装置专项层是可选部署层，不是 Core 常驻模块。
2. `distributed_node_descriptor`、`deployment_profile`、`transport_contract` 应被提升为正式对象。
3. 这些对象更适合进入独立 `deployment/` schema 分类，而不是塞进 runtime 或 capability 里。

## 8. 结论

Vidodo 的核心边界到此应进一步明确：

1. Core 负责作品语义、调度语义和证据链语义。
2. Adapter Plugin 负责单节点协议映射。
3. Distributed Device Orchestrator 负责多节点部署协调。
4. 消息总线只是在 `transport_contract` 约束下的实现技术，而不是产品核心特性本身。

因此，分布式装置专项层应被正式对象化，但仍然保持在 Vidodo Core 之外的部署边界上。
