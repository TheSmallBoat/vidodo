# 音频输出后端抽象与空间扬声器矩阵装置设计

日期：2026-04-02
状态：Draft
前置阅读：04-音频与视觉运行时事件协议与数据结构.md，14-MVP技术架构图与Phase0实施清单.md，19-调度运行时与补丁详细设计.md，20-音频输出模型与外部乐器接入设计.md

## 1. 文档目的

20 号文档已经定义了音频输出三路径与外部乐器接入方式，但仍然偏向“声音如何生成或接入”，没有充分回答“声音最终如何在具体装置上被分发和呈现”。

本文补齐以下问题：

1. 为什么 Audio Runtime 不应绑定单一物理输出形态。
2. 为什么需要一个与 Runtime Core 松耦合、可替换的音频输出后端抽象。
3. 常规系统音频输出与空间扬声器矩阵装置应如何统一建模。
4. 一个由规划驱动、离散扬声器协同发声的舞台系统，如何区别于传统消费级环绕声或杜比式通道混合输出。

## 2. 顶层设计结论

### 2.1 Audio Runtime 不应直接等同于系统声卡输出

Audio Runtime Core 的职责应是：

- 消费 Performance IR、timing、semantic 与 patch 事件
- 维护 layer、bus、routing intent 与 render state
- 生成可被输出装置消费的统一音频结果或离散路由结果

它不应直接假定最终输出一定是：

- 单一立体声设备
- 操作系统默认音频输出
- 固定通道布局的传统多声道总线

### 2.2 应引入可替换的音频输出后端抽象

推荐把音频执行层拆成两部分：

```text
Audio Runtime Core
  -> Audio Output Backend
```

其中：

- `Audio Runtime Core` 负责 layer、voice、bus、spatial intent、gain / filter / timing 调度
- `Audio Output Backend` 负责把这些结果映射到具体音频设备或扬声器装置

在系统级边界上，这类后端应进一步被理解为统一协议约束下的 audio adapter plugin。

### 2.3 首批应显式支持两类后端

推荐至少支持以下两类后端：

1. `system_audio_backend`
2. `spatial_speaker_matrix_backend`

前者用于常规操作系统音频设备输出。

后者用于一个预先标定的扬声器矩阵装置：多个喇叭分布在空间中的明确位置，系统依据规划和统一调度，把不同音频对象或音频层路由到不同扬声器，形成离散、可编排的舞台化空间声场。

### 2.4 这不是杜比式消费级多声道混合输出

本文描述的特殊后端，不以固定 channel bed 和自动对象渲染为核心，也不是“把一段内容混成 5.1 / 7.1 / Atmos”后再交给消费级解码器。

它的核心思想是：

```text
规划侧显式定义空间声场意图
  -> Runtime 生成离散 speaker routing
  -> 多个实际扬声器按统一时间语义协同发声
  -> 构成一个可编排的舞台声音装置
```

同样，驱动这些声场的素材、合成 preset 与 instrument preset 应优先来自外部资源 HUB，而不是把可变资源写死在核心里。

## 3. 需求重述

将本需求用更专业、精确的语言表述如下：

```text
系统需要在 Audio Runtime Core 之上定义一个松耦合、可替换的 Audio Output Backend 抽象。
该抽象至少支持两类目标：

1. 基于操作系统音频设备的常规输出后端；
2. 基于预标定空间扬声器矩阵的离散声场输出后端。

对于空间扬声器矩阵后端，系统维护一组带有空间坐标、角色和能力描述的扬声器端点，
并依据统一的规划、时间语义和运行状态，把不同音频层、音频对象或事件流路由到不同扬声器。
它表达的是一个由规划驱动的离散扬声器编排系统，而不是传统家庭影院式的多声道混合输出。
```

## 4. 功能目标

### 4.1 常规输出目标

系统应支持：

- 绑定操作系统默认音频设备或指定设备
- 立体声或常规多通道设备输出
- 统一 master、bus 和 layer 输出管理
- 运行期设备状态检测与恢复

### 4.2 空间扬声器矩阵目标

系统应支持：

- 定义一组带空间位置的 speaker endpoint
- 每个扬声器绑定独立输出端口或设备通道
- 按规划把不同音频层、对象或事件分配给不同扬声器
- 在统一时间语义下同步触发多个扬声器输出
- 通过预先标定的空间拓扑配置稳定复用装置布局

### 4.3 工程目标

系统还必须保证：

- 输出后端替换不影响 DSL、IR、Scheduler 和 patch 语义
- 单个扬声器端点故障可被诊断、降级和回退
- 扬声器拓扑与空间路由结果可写入 trace
- 运行时不依赖人工临时调整设备映射

## 5. 架构拆分

### 5.1 推荐结构

```text
Scheduler / Show State
  -> Audio Runtime Core
       -> Layer Controller
       -> Voice / Bus Router
       -> Spatial Intent Resolver
       -> Speaker Routing Engine
       -> Audio Output Backend
            -> System Audio Backend
            -> Spatial Speaker Matrix Backend
```

### 5.2 模块职责

#### Audio Runtime Core

负责：

- 解释 Performance IR 与 runtime audio events
- 管理 layer、voice、bus 与 timing 对齐
- 生成离散 speaker routing 或常规 bus output
- 维护 degrade、rollback 与 fallback 行为

不负责：

- 直接写死某个声卡、通道号或硬件矩阵布局
- 把特定装置协议嵌入调度核心逻辑

#### Audio Output Backend

负责：

- 建立音频设备或扬声器端点映射
- 接收 runtime 输出目标并写入实际硬件端口
- 上报 backend status、speaker ack、resource sample
- 在端点缺失时执行降级策略

## 6. 统一后端抽象

### 6.1 建议能力接口

后端抽象至少需要回答以下问题：

- 当前后端暴露哪些 audio endpoint
- 当前运行需要常规 bus output 还是离散 speaker routing
- 每个 layer / object 当前应输出到哪个 endpoint
- backend 如何汇报设备异常、路由失败和恢复状态

### 6.2 建议接口语义

可以抽象为如下能力：

- `prepare_backend(audio_topology, calibration_profile)`
- `apply_show_state(show_state)`
- `render_audio(frame_context, route_set)`
- `collect_backend_status()`
- `apply_degrade_mode(mode)`
- `shutdown_backend()`

关键语义边界是：

- Runtime Core 生成 `route_set`
- Backend 负责把 `route_set` 映射到实际设备与扬声器

## 7. 后端一：常规系统音频后端

### 7.1 定位

`system_audio_backend` 用于标准操作系统音频输出场景。

它依赖操作系统提供的设备枚举、默认设备切换与基础通道输出能力。

### 7.2 适用场景

- 本地开发与调试
- 常规单机场景演示
- 立体声或固定多通道输出
- 离线导出前的快速监听

### 7.3 必要能力

- 枚举系统 audio device id
- 指定默认设备或目标设备
- 支持常规 bus / channel 输出
- 支持设备失联检测与重连

## 8. 后端二：空间扬声器矩阵后端

### 8.1 定位

`spatial_speaker_matrix_backend` 用于一种特定装置：

- 场地中存在一组预先布置、编号和标定的扬声器
- 每个扬声器位于空间中的明确位置
- 系统将不同音频层、对象或事件分配到不同扬声器
- 观众感知到的是一个由多点协同形成的舞台声场

### 8.2 核心思想

它不是“把声音混成固定多通道总线”，而是：

- 先定义空间拓扑和扬声器角色
- 再定义每段声音在空间中的分发意图
- 最后由 Runtime 在统一时间语义下生成离散 speaker routing

### 8.3 典型使用方式

- 某组粒子音色在左后方扬声器群中游走
- 某条节奏层固定在舞台前沿扬声器上
- 某个场景切换时，声音从外围矩阵向中心聚拢

这里的一致性来自统一规划和时序调度，而不是依赖消费级声像算法自动渲染。

### 8.4 与传统多声道输出的区别

| 维度 | 常规多通道输出 | 空间扬声器矩阵后端 |
| --- | --- | --- |
| 主要目标 | 设备兼容与通道回放 | 规划驱动的离散空间编排 |
| 声道语义 | 固定声道床 | 可定义的 speaker endpoint 集合 |
| 路由方式 | 混合后写入通道 | 运行时生成离散 speaker routing |
| 场景表达 | 通道分配为主 | 舞台拓扑与空间意图为主 |

## 9. 关键对象模型

### 9.1 SpeakerEndpoint

```json
{
  "speaker_id": "spk-left-rear-02",
  "device_ref": "audio-device-a",
  "channel_ref": 7,
  "role": "left_rear",
  "position": [-4.2, 2.1, -3.6],
  "gain_trim_db": -1.5,
  "status": "available"
}
```

### 9.2 AudioSpatialBinding

```json
{
  "binding_id": "spatial-bind-drone-01",
  "source_ref": "layer-drone-a",
  "routing_mode": "discrete",
  "speaker_group": [
    "spk-left-rear-02",
    "spk-right-rear-02"
  ],
  "gain_profile": "wide-rear"
}
```

### 9.3 SpeakerMatrixTopology

```json
{
  "topology_id": "stage-a-v1",
  "speaker_count": 12,
  "reference_space": "stage_local",
  "endpoint_refs": [
    "spk-left-front-01",
    "spk-right-front-01",
    "spk-left-rear-02"
  ]
}
```

### 9.4 CalibrationProfile

建议至少包含：

- 扬声器编号与物理位置
- 设备与通道映射
- 角色分组，如 front / rear / center / overhead
- 增益修正与延迟补偿
- 场地 / 装置版本号

## 10. 标定与运维流程

### 10.1 为什么必须预标定

空间扬声器矩阵不是运行时临时猜测输出设备。

它必须依赖一份预先确认的标定配置，原因是：

- 扬声器位置与编号是舞台事实
- 通道映射和延迟补偿需要稳定复用
- 运行时热路径不应承担设备拓扑推断

### 10.2 推荐流程

1. 枚举设备与硬件输出通道。
2. 为每个扬声器分配逻辑角色与空间坐标。
3. 记录 gain trim、delay offset 与 device / channel mapping。
4. 用校准音序列验证每个 endpoint。
5. 固化为 `speaker matrix calibration profile`。

### 10.3 运行期要求

- 若某个扬声器缺失，应回落到受控降级，而不是静默继续
- 若一组 speaker endpoint 不可用，应记录 trace 与 backend status
- patch 不应直接破坏已确认的 speaker-to-channel binding

## 11. 与现有音频模型的关系

### 11.1 不改变音频三路径抽象

该设计不替代 20 号文档中的：

- `audio_asset_playback`
- `audio_synth_render`
- `audio_midi_to_instrument`

三路径仍然描述“声音从哪里来”。

本文新增的是“声音最终输出到哪里、如何形成装置级呈现”。

### 11.2 主要扩展点

真正新增的是：

- `Audio Output Backend` 抽象
- `system_audio_backend` 与 `spatial_speaker_matrix_backend`
- `speaker endpoint`、`speaker matrix topology` 与 `route_set`

## 12. MVP 范围建议

### 12.1 建议纳入 MVP 的内容

- `system_audio_backend` 常规设备输出
- `spatial_speaker_matrix_backend` 的最小骨架
- 手工标定的扬声器矩阵配置
- 离散 speaker routing 的最小 trace 记录

### 12.2 建议后置的内容

- 自动声像或复杂空间渲染算法
- 多机分布式音频网络同步
- 自动标定
- 大规模编组编辑器界面

## 13. 一句话总结

```text
Vidodo 的音频执行层不应只是一条默认声卡输出链，
而应是“统一音频运行时内核 + 可替换输出后端”的结构。

其中最基本的后端是基于操作系统设备能力的常规音频输出后端，
更特殊的后端是基于预标定空间扬声器矩阵、通过统一规划与调度把不同声音离散分发到多个扬声器的舞台声音装置后端。
```
