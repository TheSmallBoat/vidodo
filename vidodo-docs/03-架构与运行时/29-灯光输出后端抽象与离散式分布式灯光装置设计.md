# 灯光输出后端抽象与离散式分布式灯光装置设计

日期：2026-04-02
状态：Draft
前置阅读：04-音频与视觉运行时事件协议与数据结构.md，12-视听联动协议设计书.md，19-调度运行时与补丁详细设计.md，27-视觉后端抽象与多视角空间显示装置设计.md，28-音频输出后端抽象与空间扬声器矩阵装置设计.md

## 1. 文档目的

27 号文档定义了多视角空间显示装置后端，28 号文档定义了空间扬声器矩阵后端，但目前仍缺少与这两者同一设计意图下的灯光装置抽象。

本文补齐以下问题：

1. 为什么灯光控制应属于同一编排控制范畴，而不是独立的临时控制面。
2. 为什么需要一个与调度和 Runtime Core 松耦合、可替换的灯光输出后端抽象。
3. 标准灯光控制输出与离散式分布式灯光装置应如何统一建模。
4. 同一个虚拟 3D 场景，如何同时驱动视觉屏幕、离散扬声器与离散灯光装置，保持统一时序与语义。

## 2. 顶层设计结论

### 2.1 灯光控制属于同一编排控制范畴

如果视觉、音频和灯光分别由独立控制面驱动，那么系统会失去以下能力：

- 同一 musical time 下的确定性触发
- 同一 semantic state 下的跨媒介一致性
- patch、回退、降级和 trace 的统一证据链

因此，灯光不应被视为演出现场外接的随机附属系统，而应被纳入同一个 `Scheduler -> Show State -> Execution Backend` 编排闭环中。

### 2.2 应引入可替换的灯光输出后端抽象

推荐引入如下结构：

```text
Scheduler / Show State
  -> Lighting Control Core
       -> Lighting Output Backend
```

其中：

- `Lighting Control Core` 负责解释统一时序、语义状态、场景状态和 patch 约束
- `Lighting Output Backend` 负责把这些结果映射到具体灯具协议、灯具拓扑和物理装置

### 2.3 首批应显式支持两类后端

推荐至少支持以下两类后端：

1. `fixture_bus_backend`
2. `spatial_lighting_matrix_backend`

前者用于常规 DMX / Art-Net / sACN 等总线型输出。

后者用于离散式分布式灯光装置：多个具备空间位置、朝向和能力差异的灯具被映射到同一个虚拟 3D 场景中的光场意图。

### 2.4 灯光不是视觉的附庸，也不是独立产品

灯光与视觉共享场景语义，但不应被降格为“从屏幕亮度推导出来的副产物”。

它是同一编排系统中的第三类执行后端：

- 音频负责离散声场输出
- 视觉负责多视角空间显示输出
- 灯光负责离散光场与灯具编排输出

三者共享统一的时间、场景和 patch 语义，但保持各自独立的设备拓扑与降级策略。

## 3. 需求重述

将本需求用更专业、精确的语言表述如下：

```text
系统需要在统一调度和 Show State 之下定义一个松耦合、可替换的 Lighting Output Backend 抽象。
该抽象至少支持两类目标：

1. 面向标准灯光控制协议与灯具总线的常规输出后端；
2. 面向离散式分布式灯具阵列的空间灯光输出后端。

对于空间灯光输出后端，系统维护一组带有空间坐标、朝向、能力集与分组信息的灯具端点，
并依据统一的虚拟 3D 场景、时序语义与运行状态，把不同的光场意图、颜色、亮度、运动与 cue
映射到不同灯具或灯具组，形成可编排的舞台光场装置。
```

## 4. 功能目标

### 4.1 常规灯光输出目标

系统应支持：

- 指定 universe、subnet、fixture address 或等价灯具地址
- 对接标准灯光输出协议或桥接设备
- 管理亮度、颜色、strobe、beam、pan / tilt 等基础参数
- 运行期检测端点可用性和回执状态

### 4.2 空间灯光矩阵目标

系统应支持：

- 定义一组带空间位置和朝向的 fixture endpoint
- 每个灯具声明能力集，例如 dimmer、rgb、white、pan、tilt、zoom、gobo
- 按统一场景语义把光场意图分发到不同灯具或灯具组
- 与视觉 view group、音频 route set 在同一时间窗口内同步切换
- 在运行期按拓扑稳定复用装置布局，而不是临时手工打灯

### 4.3 工程目标

系统还必须保证：

- 更换灯光后端不影响 DSL、IR、Scheduler 和 patch 语义
- 单个灯具或 universe 故障可被诊断、降级和回退
- 灯光拓扑、cue 生效和异常状态可写入 trace
- 灯光输出不依赖单独的人工现场跟随操作才能成立

## 5. 架构拆分

### 5.1 推荐结构

```text
Scheduler / Show State
  -> Lighting Control Core
       -> Cue Resolver
       -> Fixture Group Mapper
       -> Spatial Light Intent Resolver
       -> Degrade / Fallback Manager
       -> Lighting Output Backend
            -> Fixture Bus Backend
            -> Spatial Lighting Matrix Backend
```

### 5.2 模块职责

#### Lighting Control Core

负责：

- 消费统一 timing、semantic、transition 和 patch 状态
- 把场景意图解析为灯光 cue、颜色场、亮度场和运动目标
- 生成可下发的 fixture command set 或 fixture group update
- 管理安全模式、降级和回退窗口

不负责：

- 直接写死某个灯控协议实现细节
- 把某个控制台品牌或设备地址嵌入调度核心

#### Lighting Output Backend

负责：

- 发现或加载灯具拓扑与协议映射
- 把 fixture command set 映射到实际 universe、节点或设备端口
- 回传 backend status、fixture ack、timeout 和 degrade 状态

## 6. 与视觉 / 音频的统一设计意图

### 6.1 统一母场景

这里的核心设计意图不是“三套设备各自跑自己的节目”，而是：

```text
同一个虚拟 3D 场景 / 语义状态
  -> 视觉后端输出多视角屏幕画面
  -> 音频后端输出离散扬声器路由
  -> 灯光后端输出离散灯具光场
```

三者共享：

- 同一个 section / phrase / beat 边界
- 同一个 transition window
- 同一个 patch 授权与回退边界
- 同一个 trace / replay 证据链

### 6.2 各后端的独立性

尽管三者共享同一编排意图，但具体物理映射仍彼此独立：

- 视觉映射到 `display_topology + view_group`
- 音频映射到 `speaker_matrix_topology + route_set`
- 灯光映射到 `fixture_topology + cue_set`

因此，灯光应被纳入同一控制域，而不是并入视觉 backend 内部实现。

## 7. 后端一：常规灯具总线后端

### 7.1 定位

`fixture_bus_backend` 用于标准灯具控制环境。

它依赖成熟灯控协议和网关能力，用于常规舞台灯具、基础灯带和通用控制链路。

### 7.2 适用场景

- 本地预演与装置联调
- 中小型舞台或展陈装置
- 固定灯具地址和组态较稳定的环境

### 7.3 必要能力

- 枚举或加载 universe / fixture addressing
- 下发 brightness、color、movement、effect 参数
- 处理灯具缺席、冲突地址和响应超时

## 8. 后端二：离散式分布式灯光矩阵后端

### 8.1 定位

`spatial_lighting_matrix_backend` 用于一种特定装置：

- 场地中存在一组预先布置、编号和标定的灯具
- 每个灯具位于空间中的明确位置，且可能具有朝向和能力差异
- 系统把同一个虚拟场景中的光场变化分发到不同灯具端点
- 观众感知到的是一个离散但统一的空间光场

### 8.2 核心思想

它不是简单的 cue list 回放，而是：

- 先定义灯具拓扑与能力边界
- 再定义某一段场景的光场意图
- 最后由 Runtime 在统一时间语义下生成离散 fixture command set

### 8.3 典型使用方式

- 场景中的粒子流动方向被映射为灯具组的依次点亮与色温迁移
- 某段 build-up 的张力增长，被映射为外围灯具向中心灯具的收束
- 某次 patch 生效时，视觉 view group、音频 route set 和灯光 cue set 在同一 phrase 边界切换

## 10. 运行与安全约束

### 10.1 时序约束

- 灯光 cue 必须与统一 musical time 对齐
- 需要支持 phrase / section 边界生效与有限 lookahead
- 不允许在热路径上进行设备发现和大规模拓扑重计算

### 10.2 安全约束

- 超出装置亮度或运动边界的命令必须被拦截或裁剪
- patch 不得突破锁定区间与安全白名单
- 灯具丢失时必须可降级到保守输出组态

### 10.3 Trace 约束

至少记录：

- cue 生效时间
- fixture ack / timeout
- degrade 与 fallback 触发原因
- 与视觉 / 音频切换对齐的证据点

## 11. 结论

灯光控制应被纳入与视觉、音频相同的编排控制域，而不是作为一个独立、临时、手工补救式的外设系统。

对 Vidodo 而言，更合理的结构是：

1. 由统一 Scheduler 和 Show State 提供时间、语义和 patch 边界。
2. 由视觉、音频、灯光三个执行后端分别消费同一场景意图。
3. 由各自独立的 topology / route / cue 结构映射到具体物理装置。

这样，系统才能围绕“同一个虚拟 3D 场景”同时驱动离散式分布式的屏幕、扬声器和灯光装置，并保持可追踪、可回退、可替换的工程边界。
